# 方法轮前置研究阶段说明 v0.1

> 位置：`00_知识库设计规范/03_版本迭代/方法轮前置研究阶段说明_v0.1.md`  
> 分线：版本迭代 / AI Method Wheel 前置阶段  
> 状态：v0.1，阶段定义稿  
> 用途：说明当前“需求澄清讨论之后、正式 spec/plan/tasks 之前”的研究上下文固化阶段应该怎么做。

---

## 1. 本文目的

本文用于说明当前项目在 AI Method Wheel 中所处的位置，以及当前阶段应该做什么、不应该做什么。

当前项目不是代码实现阶段，也不是 Codex 执行阶段，而是处在：

```text
需求澄清讨论之后的理论收敛与项目上下文固化阶段
```

本阶段的核心任务是把前期讨论中形成的策略研究结论，整理为稳定、可审计、可复用的项目上下文，为后续进入 Good Question Brief、spec、plan、tasks、GitHub issue、Codex handoff、TDD / QA / PR 做准备。

---

## 2. 当前阶段定位

当前阶段属于：

```text
需求澄清讨论
→ 理论收敛
→ 项目上下文固化
→ 方法轮输入材料准备
```

还没有进入：

```text
spec 编写
plan 拆解
tasks 任务化
GitHub issue 创建
Codex 执行
TDD 开发
QA / CI 验证
PR 合并
```

因此，当前阶段的产出重点不是“实现”，而是“沉淀”。

---

## 3. 为什么现在不能直接进入 Codex 阶段

Codex 适合执行边界清晰、验收标准明确、输入材料稳定的任务。

但当前项目仍然存在大量策略定义层面的不确定性，例如：

```text
第一控盘区到底如何锁定
成本中枢到底用什么表达
深洗和死亡盘如何区分
重新接受和普通反抽如何区分
二次启动和诱多如何区分
哪些失败样本可以推翻模型
哪些样本不能再靠补规则解释
策略是否存在后视偏差
图形成立是否等于交易可执行
外部成熟理论只是参考，还是要成为核心规则
```

这些问题如果没有先收敛，就直接交给 Codex，容易产生以下问题：

```text
1. Codex 被迫替 Owner 做策略定义。
2. Codex 把模糊概念写成看似确定的规则。
3. 后续回测验证的是一个并未真正定义清楚的模型。
4. 项目从“研究策略”滑向“为已有观点找证据”。
5. 失败样本被不断解释掉，导致模型不可证伪。
6. 代码产出看似推进，实际放大前置定义错误。
```

因此，当前阶段必须先把需求讨论结论固化为项目上下文。

---

## 4. 当前阶段的核心任务

当前阶段要完成的不是实现，而是以下五类沉淀：

```text
1. 固化研究总纲
2. 固化核心术语
3. 建立反证框架
4. 设计样本标注模板
5. 建立 Owner 决策记录
```

---

## 5. 任务 1：固化研究总纲

目标：

```text
把当前策略研究的主要方向、核心问题、风险点和后续路径整理成总纲。
```

对应文件：

```text
00_知识库设计规范/00_总纲/需求澄清后研究总纲_v0.1.md
```

该文件应回答：

```text
项目研究的总目标是什么
当前最重要的研究主线有哪些
每条主线对应哪些目录
每条主线要解决哪些核心问题
当前阶段应该如何继续推进
哪些内容后续会进入 spec / plan / tasks
```

研究总纲是后续阶段的母文档之一。

---

## 6. 任务 2：固化核心术语

目标：

```text
把策略中的关键语义概念从“聊天语言”转化为“可讨论、可标注、可反证”的术语体系。
```

对应文件：

```text
03_语义概念/核心术语表_v0.1.md
```

至少应包含：

```text
第一控盘区
成本中枢
深洗
死亡盘
重新接受
普通反抽
二次启动
诱多
失败样本
可执行买点
不可执行样本
后视偏差
反证样本
消融实验
```

每个术语不应只写一句解释，而应包含：

```text
术语名称
当前工作定义
可观察特征
非该概念的情况
常见误判
待 Owner 决策点
后续是否可量化
```

---

## 7. 任务 3：建立反证框架

目标：

```text
在继续优化模型之前，先定义什么情况会推翻模型。
```

对应文件：

```text
09_规则与回测/反证实验设计_v0.1.md
```

该文件要回答：

```text
哪些样本能推翻模型
哪些失败不能继续靠补规则解释
哪些大涨被模型漏掉
哪些图形成立但交易不可执行
哪些条件需要做消融实验
如果去掉某个条件表现不变，是否说明该条件只是叙事
如果失败样本过多，模型如何降级或重构
```

如果没有反证框架，项目容易变成不可证伪的叙事模型。

## 9. 任务 5：建立 Owner 决策记录

目标：

```text
把需要人类最终判断的策略定义、边界、假设和取舍记录下来，防止后续反复横跳。
```

建议后续新增：

```text
11_问题清单/待决策问题_v0.1.md
12_决策记录/决策记录_v0.1.md
```

若暂不新增目录，也可先维护在：

```text
10_工程化交接/Owner_decision_brief.md
00_知识库设计规范/05_索引与变更记录/设计规范变更记录.md
```

当前必须由 Owner 决策的问题包括：

```text
第一控盘区的默认定义
成本中枢优先采用哪种表达
是否允许多个成本中枢
深洗最大容忍幅度
深洗最大容忍时间
重新接受的确认标准
二次启动的确认标准
诱多的失败判定
是否把 Wyckoff 作为核心理论
是否把 Liquidity Sweep 作为深洗解释
是否用 POC / Value Area 表达成本中枢
是否引入 AVWAP
AVWAP 锚点如何避免后视偏差
SMC 概念是辅助过滤还是核心规则
当前项目是研究模型，还是最终要做交易系统
先验证哪个市场
先验证哪个周期
回测结果达到什么标准才进入下一阶段
```

这些问题不能交给 Codex 自行决定。

## 10. 当前阶段和 AI Method Wheel 的关系

当前阶段不是脱离方法轮，而是为方法轮准备输入。

可以理解为：

```text
当前阶段 = 方法轮前置上下文构建
```

正式方法轮通常需要：

```text
Good Question Brief
→ spec
→ plan
→ tasks
→ GitHub issue
→ Codex handoff
→ TDD / QA / review
→ PR
→ Owner decision
```

但这些步骤需要稳定输入。

当前阶段要提供的输入包括：

```text
研究总纲
核心术语表
反证框架
样本标注模板
待决策问题
阶段性结论
Owner 已确认的定义
不可交给 Codex 自行发挥的边界
```

---

## 11. 哪些内容应该进入项目上下文

当前阶段应沉淀进项目上下文的内容包括：

```text
1. 项目目标
2. 当前研究主线
3. 核心术语和未决定义
4. 反证标准
5. Owner 决策
6. Codex 禁止自行发挥的边界
```

项目上下文应进入本地 Markdown 文件，而不是只保留在聊天中。

---

## 12. 当前阶段不应该做什么

当前阶段不应该：

```text
1. 直接写指标。
2. 直接回测。
3. 让 Codex 自己拆策略。
4. 继续无限扩展理论。
5. 只研究成功样本。
6. 把观察指标写成自动交易建议。
7. 把资料观点直接升级为策略结论。
```

当前阶段的关键是收敛，而不是扩张。

---

## 13. 当前阶段完成标准

当前阶段不是以“代码跑通”为完成标准，而是以“上下文是否足够稳定，可以进入方法轮”为完成标准。

完成标准：

```text
1. 研究总纲已完成。
2. 核心术语表已完成初版。
3. 反证实验设计已完成初版。
4. 待决策问题已列出。
5. 后续方法轮输入已准备。
```

---

## 14. 进入下一阶段的判断条件

只有当以下问题有初步答案时，才适合进入正式方法轮：

```text
1. 第一控盘区初版定义是什么？
2. 成本中枢初版表达方式是什么？
3. 深洗和死亡盘初版区分标准是什么？
4. 重新接受初版确认标准是什么？
5. 二次启动初版确认标准是什么？
6. 哪些样本能推翻模型？
7. 样本标注模板包括哪些字段？
8. 先验证哪个市场、周期和样本范围？
9. Codex 第一阶段只允许做什么？
10. 第一阶段验收标准是什么？
```

如果这些问题没有初步答案，则不应进入 Codex 实现阶段。

---

## 15. 后续 Codex 边界声明草案

后续如果进入 Codex 阶段，应在 handoff 中明确写入：

```text
Codex 只能执行本文档和相关 spec 中已经定义清楚的任务。

Codex 不得自行：
- 定义第一控盘区；
- 定义成本中枢；
- 修改深洗 / 死亡盘判断；
- 修改重新接受标准；
- 修改二次启动标准；
- 删除失败样本；
- 排除不利样本；
- 增加未经 Owner 确认的过滤条件；
- 将外部理论直接改写为核心规则；
- 扩大研究范围；
- 美化回测结果；
- 把不可执行买点视为有效信号。

如遇到策略定义不清、样本冲突、规则冲突、数据不足、后视偏差风险，应停止并返回 Owner 决策。
```

---

## 16. 本版结论

当前阶段的重点不是做更多理论扩展，也不是马上工程实现，而是把已经讨论出来的结论变成项目的稳定上下文。

只有完成这个阶段，后续方法轮才有可靠输入。

否则，spec、plan、tasks、Codex handoff、回测和自动化都会建立在不稳定定义上，容易产生看似推进、实际偏离的问题。
