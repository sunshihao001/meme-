# 版本迭代与更新流程规范 v0.1

> 位置：`00_知识库设计规范/03_版本迭代/版本迭代与更新流程规范_v0.1.md`  
> 目的：规范每次知识库更新如何进行，避免内容散落在聊天上下文中。  

---

## 1. 每次更新的入口问题

每次更新前先判断：

```text
这次更新属于哪条分线？
是新增定义，还是修正定义？
是否产生新关键词？
是否需要检索资料？
是否需要下载资料？
是否需要更新总纲？
是否需要更新版本日志？
```

---

## 2. 标准更新流程

```text
Step 1：确定分线
Step 2：确定主题
Step 3：创建/更新文章
Step 4：提炼关键词
Step 5：检索论坛/文章/代码库
Step 6：下载资料到对应目录
Step 7：建立补全来源清单
Step 8：写资料吸收笔记
Step 9：更新相关定义文档
Step 10：更新版本日志
```

---

## 3. 版本更新类型

| 类型 | 说明 | 是否更新日志 |
|---|---|---|
| 新增 | 新建概念/分线/文章 | 是 |
| 修正 | 改写旧定义 | 是 |
| 补充 | 增加资料来源或例子 | 是 |
| 降级 | 发现某定义不适合，降低权重 | 是 |
| 废弃 | 不再使用某概念 | 是，并放入归档 |
| 归档 | 旧版本保留但不再主用 | 是 |

---

## 4. 文章版本规则

```text
v0.1 初版
v0.2 补充资料后更新
v0.3 融合样本后更新
v0.4 融合指标需求后更新
v1.0 定义稳定版
```

不要直接覆盖旧版核心结论。如果定义变化较大，应新建：

```text
主题名_v0.2.md
```

并在旧版顶部标注：

```text
> 已被 v0.2 更新，保留用于历史追踪。
```

---

## 5. 来源资料处理规范

每次下载资料后必须建立：

```text
主题_补全来源清单_v0.1.md
```

每条来源记录：

```text
标题
URL
语言
本地路径
相关分线
相关概念
可吸收内容
是否改变当前定义
```

---

## 6. 资料吸收规范

下载资料不等于吸收。每个重要资料都要形成吸收笔记。

吸收笔记结构：

```text
1. 资料基本信息
2. 资料核心观点
3. 和本策略相关的概念
4. 可吸收内容
5. 对当前定义的影响
6. 新增关键词
7. 待验证问题
```

---

## 7. 版本日志规范

版本日志必须记录：

```text
日期
版本
更新主题
新增文件
修改文件
新增关键词
新增资料来源
改变的定义
下一步
```

---

## 8. 防止上下文丢失的要求

每次会话结束前必须确认：

```text
[ ] 重要结论已写入文件
[ ] 文件路径已明确
[ ] 来源资料已保存
[ ] 来源清单已更新
[ ] 版本日志已更新
[ ] 下一步已写入文档
```
