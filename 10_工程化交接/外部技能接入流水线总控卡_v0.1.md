# 外部技能接入流水线总控卡 v0.1

> 分线：10_工程化交接 / 外部技能接入 / 总控卡  
> 状态：v0.1 / A端可直接调用的一页控制卡  
> 目的：把“看到一个 GitHub 外部 skill 仓库后应该怎么判断、怎么接入、怎么验证”压缩成可执行总控卡，避免盲目安装、重复造 skill、或把文档变化误判成真实运行变化。

---

## 0. 使用场景

当出现以下输入时，先用本卡：

```text
1. 用户给出 GitHub skill / prompt / agent workflow 仓库。
2. 用户说内部技能不够，需要参考别人的技能。
3. 用户想把外部方法论加入 A/B/C/D/E/F 多端口体系。
4. 用户想判断某个外部 skill 是否该安装、桥接、合并或拒绝。
```

本卡不是安装指南，而是：

```text
A端外部技能接入门控卡。
```

---

## 1. 一句话总原则

```text
先补缺口，再看仓库；先做判断，再做接入；先小范围验证，再考虑固化。
```

禁止把流程简化成：

```text
看到好 skill → 复制安装 → 默认启用。
```

---

## 2. 七阶段流水线

```text
Stage 0  A端 Intake Gate          接入门
Stage 1  B端 Evidence Pack        证据包
Stage 2  A端 Adoption Gate        采纳判断门
Stage 3  C端 Packaging Design     内部包装设计
Stage 4  D端 Repo Landing         仓库落地
Stage 5  E端 Effect Verification  效果验证
Stage 6  F端 Owner Decision       Owner 决策
```

---

## 3. Stage 0：A端 Intake Gate

### 输入

```text
GitHub 仓库 URL / skill 包 / prompt 包 / slash-command 包 / agent workflow 包
```

### A端必须先问

```text
1. 这是什么类型的外部资产？
2. 它解决当前内部哪个缺口？
3. 它是否影响 A-F 端口职责边界？
4. 它是否值得让 B端做 Source Pack？
```

### 输出

```text
External Skill Intake Brief
```

### 允许结论

```text
ENTER_B_SOURCE_PACK
HOLD_FOR_MORE_CONTEXT
REJECT_AS_NO_CLEAR_GAP
```

---

## 4. Stage 1：B端 Evidence Pack

### B端收集

```text
1. 仓库名 / URL / commit SHA
2. 技能、命令、prompt、模板清单
3. 每个候选的 source path
4. 触发条件
5. 输入 / 输出结构
6. 明确禁区
7. 上游是否 deprecated / experimental / personal
```

### 输出

```text
External Skill Source Pack
```

### 证据要求

```text
没有 source path 和 commit 的候选，不进入采纳判断。
```

---

## 5. Stage 2：A端 Adoption Gate

对每个候选输出统一判断：

```text
候选名：
来源路径：
解决缺口：
接入结论：ADOPT / BRIDGE / MERGE / REJECT / PATTERN_ONLY
A-F端口：A / B / C / D / E / F / Cross-port
L1-L6层级：L1 / L2 / L3 / L4 / L5 / L6
别名策略：keep alias / rename / merge alias / no alias
验证样例：至少 1 个真实或半真实场景
状态：referenced_only / mapped / source_packed / designed / drafted / sample_validated / small_scope_enabled / project_skill_updated / needs_more_samples / pending_owner_decision / accepted
```

### 五类接入结论

```text
ADOPT        可直接成为内部稳定能力，但仍需验证与 Owner 决策。
BRIDGE       有价值，但只作为桥接层，不冒充内部主 skill。
MERGE        与现有能力重复，吸收为子模块、别名或模板。
REJECT       价值不清、冲突大、不可维护，不进入主栈。
PATTERN_ONLY 只吸收模式，不吸收原 skill 名称或文件结构。
```

---

## 6. Stage 3：C端 Packaging Design

C端不负责决定“该不该接”，只负责把已通过 A端判断的候选设计成内部包装。

### 包装原则

```text
1. 不盲目照搬外部 SKILL.md。
2. 优先合并到现有端口主 skill 或辅助能力组。
3. 输出统一模板。
4. 写清触发条件和跳过条件。
5. 保留来源与 alias，避免上下游断链。
```

### 输出

```text
Internal Wrapper Design
```

---

## 7. Stage 4：D端 Repo Landing

D端只做落地，不做理论发明。

### 落地范围

```text
1. 新增或更新项目级 skill / wrapper 草案
2. 更新索引
3. 更新变更记录
4. 运行验证
5. 提交 commit，必要时 push / PR
```

### 禁止

```text
1. 未经 F端授权，不写入全局 runtime skill。
2. 不全量安装外部仓库。
3. 不跳过索引和变更记录。
4. 不把 mapped 状态说成 accepted。
```

---

## 8. Stage 5：E端 Effect Verification

E端验证的重点不是“文档写得好不好”，而是行为是否真的改善。

### 验证问题

```text
1. 判断质量是否提升？
2. 是否减少技能重复和职责混乱？
3. 是否保护 A-F 端口边界？
4. 是否有真实样例，而不是漂亮话？
5. 是否区分了文档变化、项目级行为变化、全局安装？
```

### 输出

```text
Effect Verification Report
```

### 建议结论

```text
PASS
PARTIAL_ACCEPT_CONTINUE
REVISE_REQUIRED
REJECT
```

---

## 9. Stage 6：F端 Owner Decision

F端只做最终授权，不被外部 diagnosis / review skill 替代。

### 决策项

```text
ACCEPT：正式固化或明确小范围启用。
PARTIAL_ACCEPT_CONTINUE：继续跑真实样例。
REVISE：退回 C端调整包装或 A端重判。
REJECT：仅保留参考，不进入技能体系。
```

---

## 10. 状态词标准

任何外部 skill 候选必须使用精确状态词：

```text
referenced_only      只被提到
mapped               已映射到 A-F / L1-L6
source_packed        已有 B端证据包
designed             已有 C端包装设计
drafted              已写草案
sample_validated     已通过样例验证
small_scope_enabled  小范围启用
project_skill_updated 项目级 skill 已更新
needs_more_samples   需要更多样例
pending_owner_decision 等待 F端决策
accepted             已被 Owner 接受
```

必须区分：

```text
文档变化 ≠ 项目级 skill 行为变化 ≠ 全局 runtime 安装。
```

## 14. 最小可运行闭环

第一次接入外部仓库时，不要追求全量吸收，只跑最小闭环：

```text
1 个外部仓库
3-5 个候选 skill
1 张 Source Pack
1 张 Adoption Matrix
1 个包装草案
1 次样例验证
1 个 F端决策
```

通过后再扩大到第二批。

---

## 15. 禁止事项

```text
1. 禁止全量安装外部 skill 仓库。
2. 禁止让外部判断 skill 替代 F端 Owner 决策。
3. 禁止把一次性 prompt 升格为长期 skill。
4. 禁止为每个外部 skill 都创建独立内部 skill，优先合并/桥接。
5. 禁止只看 README，不看 SKILL.md / commands / prompts / templates。
6. 禁止 claim “已启用”，除非至少达到 small_scope_enabled 或 project_skill_updated。
```

---

## 16. 当前推荐执行顺序

```text
1. 继续把 dbskill 当作样板仓库。
2. 再选 1-2 个真实 GitHub skill 仓库横向测试。
3. 每个仓库只选 3-5 个候选跑最小闭环。
4. 稳定后再决定是否进入第二批状态链能力接入：dbs-decision / dbs-save / dbs-restore / dbs-report。
```

---

## 17. A端最新工作原则

这次 A 端要再加一条很重要的判断：

```text
不是所有 skill 都要接入。
```

更准确地说：

```text
1. 有的外部仓库是完整工程师 skill 包，只需要抽其中 1-2 个部件。
2. 有的 skill 与当前工作流冲突，接入只会污染体系。
3. A端要判断的是“这部分能力是否适合当前工作流”，不是“这个仓库值不值得收藏”。
4. 输出给用户时要尽量用自然语言讲清楚，但底层判断仍要保持专业门控。
5. 用户不是专业工程师或 AI 从业者，所以反馈要“看得懂”，而不是“堆术语”。
```

A端新的实际目标可以概括为：

```text
把外部 skill 拆成可理解、可判断、可局部吸收的能力部件；
只吸收真正补缺口的部分；
避免全量接入导致污染。
```

更深一层的 A端目标是：

```text
自然语言输入 → 意图挖掘 → 知识补全 → 专业化升级 → 用户可理解反馈 → 专家级执行路线
```

也就是：

```text
用户不需要先成为专业工程师或 AI 从业者；
用户可以先用普通自然语言表达想法；
A端负责通过 skill 组合把想法拆开、补全、升级、包装成更专业的方案；
但最后反馈必须仍然让用户看得懂、能判断、能拍板。
```

这是一种知识循环更替代理：

```text
个人初步想法
→ A端澄清和追问
→ B端补证据
→ C端提出更专业版本
→ D端落库
→ E端验证是否真的更好
→ F端决定是否吸收成新的认知/流程
```

### A端判断时的简化问法

```text
- 这个仓库整体适合我吗？
- 如果不适合，里面哪一小块值得取用？
- 取用后会不会污染我当前流程？
- 我能不能用简单自然语言说明它的作用？
- 它是否真的能让我像专业人士一样工作，而不是只是看起来更复杂？
```

## 18. 本卡结论

```text
外部技能接入不是“加技能”，而是“补缺口 + 过门控 + 做包装 + 跑验证 + Owner 决策”。
```

当前最优路线：

```text
小范围、多样例、强门控、慢固化。
```
