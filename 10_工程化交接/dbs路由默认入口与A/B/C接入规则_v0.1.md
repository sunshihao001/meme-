# dbs 路由默认入口与 A/B/C 接入规则 v0.1

> 分线：10_工程化交接 / dbskill 接入  
> 状态：v0.1 / 默认入口规则草案  
> 目标：把 dbskill 的 `/dbs` 路由接成工作流默认入口，让每次新输入先做能力识别，再决定进入 A/B/C/D/E/F 哪一端。  
> 原则：先路由、后展开；先判别、后执行；先最小可消费、后深度扩展。

---

## 1. 这个规则解决什么问题

用户现在面对的问题不是“没有技能”，而是：

```text
1. 技能太多，直接看会乱。
2. A 端一上来就全量思考，会慢。
3. 需要先有一个统一入口，把输入分给合适的能力包。
4. 需要让 dbskill 的路由能力先工作，再去解决更细的 A 端默认拷问规则。
```

所以这份规则的定位是：

```text
dbs 路由 = 工作流默认前置门。
A/B/C = 路由后的专业端口。
```

---

## 2. 默认入口原则

### 2.1 默认路径

```text
用户新输入
→ dbs 路由入口
→ 判断问题类型
→ 分配到 A / B / C / D / E / F
```

### 2.2 默认行为

```text
1. 先识别这是“问题定义 / 知识研究 / 证据收敛 / 理论生成 / 落库 / 验证 / 决策”中的哪一类。
2. 识别后，再决定具体调用哪些 dbs 能力包。
3. 不先把所有技能一次性展开。
4. 不先进入深度理论，除非路由结果明确需要。
```

### 2.3 默认输出要求

dbs 路由入口每次至少输出以下内容中的 2 项：

```text
- 问题类型判断
- 推荐端口
- 候选技能包
- 是否需要 deep_path
- 是否需要回退到 A 端
```

---

## 3. 路由层职责

### 3.1 路由层要做的事

```text
1. 先判断当前输入属于哪类问题。
2. 再决定是否进入 A 端需求澄清。
3. 再决定 B/C/D/E/F 是否需要。
4. 控制技能启用范围。
5. 把“想法”翻译成“可执行能力需求”。
```

### 3.2 路由层不要做的事

```text
1. 不替 A 端直接写最终需求答案。
2. 不替 B 端直接做 Source Pack。
3. 不替 C 端直接写长理论。
4. 不替 D/E/F 做最终执行和验证。
5. 不把 dbskill 变成“技能堆列表”。
```

---

## 4. A/B/C 接入规则

### 4.1 A 端接入条件

当输入属于以下情况，先进入 A 端：

```text
- 用户想法模糊
- 概念很多但边界不清
- 想法里混了多个问题
- 需要先判断“到底要研究什么”
- 需要把自然语言转成专业概念
```

A 端优先技能包：

```text
- dbs-good-question
- dbs-deconstruct
- dbs-goal
- dbs-decision
```

### 4.2 B 端接入条件

当输入属于以下情况，进入 B 端：

```text
- 需要收集外部资料
- 需要压缩本地文件/样本/知识库
- 需要整理证据、冲突、缺口、unknown
- 需要把材料做成 Source Pack
```

B 端优先技能包：

```text
- dbs-learning
- dbs-content-system
- dbs-deconstruct
- dbs-report
- dbs-save / dbs-restore（状态管理时）
```

### 4.3 C 端接入条件

当输入属于以下情况，进入 C 端：

```text
- A 端已经把问题说清楚
- B 端已经有足够证据
- 需要把 A+B 变成理论或方案
- 需要形成可交给 D/E 的中间产物
```

C 端优先技能包：

```text
- dbs-deconstruct
- dbs-learning
- dbs-content-system
- dbs-decision
```

## 5. 最小可运行路由集

为了先把系统跑起来，不要求一开始就把所有 dbskill 都自动化。

### 5.1 默认常驻路由包

```text
A 端：dbs-good-question / dbs-deconstruct / dbs-goal / dbs-decision
B 端：dbs-learning / dbs-content-system / dbs-deconstruct
C 端：dbs-deconstruct / dbs-learning / dbs-content-system / dbs-decision
D/E/F：dbs-save / dbs-restore / dbs-report / dbs-decision
```

### 5.2 条件启用包

```text
- dbs-slowisfast：当有过早收敛风险时启用
- dbs-diagnosis：当需要更强的诊断框架时启用
- dbs-hook / dbs-xhs-title：当明确进入内容生产时启用
- dbs-content-system：当要做内容资产工程时启用
- dbs-chatroom：当要做话题人物/讨论结构判断时启用
```

## 6. A 端与 dbs 路由的关系

A 端不是 dbskill 的替代品，而是 dbs 路由后的“第一专业判断门”。

```text
dbs 路由负责：
- 判断该用哪类能力
- 控制技能启用范围
- 先把输入分到正确端口

A 端负责：
- 在被路由到的问题上做需求澄清
- 把问题讲专业
- 把下一步路由清楚
```

如果路由结果显示问题还不清楚，则：

```text
路由 → A 端 → 澄清 → 再路由
```

这比一上来就把所有技能打开快得多。

---

## 7. 触发与跳过规则

### 7.1 必须先过路由的情况

```text
- 新输入
- 新想法
- 新任务
- 新问题
- 新知识点
```

### 7.2 可以跳过 A 的情况

```text
- 明确续接上次
- 已有任务包/issue
- 已明确是执行/恢复/验证
- 用户明确说“直接继续”
```

### 7.3 跳过 A 的风险

```text
- 容易把续接当新问题
- 容易把新问题当续接
- 容易错过真正的澄清门
```

所以默认策略仍然是：

```text
先路由，再决定是否进 A。
```

---

## 8. 运行约束

```text
1. 不全量接入所有 dbskill 功能。
2. 不把技能名当成工作流本身。
3. 不让 A 端被技能列表淹没。
4. 不在没有路由结果时直接进入深度理论。
5. 不让一个问题同时拉满 A/B/C/D/E/F。
```

---

## 9. A/B/C 收束口径

当输入已经进入 A/B/C 研究链时，统一采用以下收束口径：

```text
1. 目标、边界、方案、验证四项中至少 3 项达到可执行级别。
2. 连续 2 轮没有新增有效信息就停止继续追问。
3. 最多循环 2~3 轮后，强制复盘是否该转端。
4. 如果继续追问只会改细节、不改路线，就停止 A/B/C 研究循环。
```

停机后默认动作：

```text
- 进入 spec / issue / 执行版方案
- 或回到 A 端做最小补齐后重新路由
```

## 10. 验收标准

当这份规则生效后，应该能看到：

```text
1. 新输入先被分类。
2. A 端只处理真正需要澄清的问题。
3. B/C 不再无边界启动。
4. dbskill 的能力包是按需调用，而不是全量堆叠。
5. 输入路径更快，输出更稳。
```
