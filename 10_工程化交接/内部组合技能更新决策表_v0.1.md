# 内部组合技能更新决策表 v0.1

> 分线：10_工程化交接 / 技能栈更新  
> 状态：v0.1 / 逐层更新表  
> 目的：把 L1-L6 每一层都具体化为可操作的更新决策，避免只讲抽象分层不讲怎么改。

---

## 1. 读法

每一层都按照同一套问题来决策：

```text
1. 这一层负责什么？
2. 当前有哪些典型技能？
3. 该新增、拆分、合并、替换、桥接、弃用哪一种？
4. 更新后谁来消费？
5. 需要哪些边界/状态标记？
```

---

## 2. L1 端口主技能更新决策

### L1 负责什么

```text
定义 A/B/C/D/E/F 六端身份核。
```

### L1 当前典型项

```text
meme-demand-control-port
meme-source-pack-port
meme-theory-codex-port
meme-repo-landing-port
meme-verification-review-port
meme-owner-decision-port
```

### L1 更新决策

```text
- 原则：稳定，少动。
- 动作：replace / alias / deprecate 优先，慎用 remove。
- 适用场景：端口职责重新定义、主身份更名、端口合并或拆分。
```

### L1 规则

```text
- 一个端口只保留一个主 skill。
- 主 skill 不能混入辅助职责。
- 旧名若还在用，先保留 alias。
```

### L1 产物消费方

```text
A-F 端口总览、启动提示词、执行面板、路由器。
```

---

## 3. L2 通用辅助技能更新决策

### L2 负责什么

```text
提供跨端口复用的通用能力。
```

### L2 当前典型项

```text
- dbs-good-question
- dbs-deconstruct
- dbs-goal
- dbs-decision
- ai-method-wheel
- maintainer-orchestrator
- github-pr-workflow
- github-repo-management
- github-code-review
- systematic-debugging
- requesting-code-review
```

### L2 更新决策

```text
- 动作：split / merge / replace / alias 常用。
- 适用场景：某个辅助技能职责过大、重复、或命名不准。
```

### L2 规则

```text
- 辅助技能必须有明确主用途。
- 如果一个技能被多个端口重复依赖，应优先保证定义稳定。
- 辅助技能可以多端共用，但不要承担主端口身份。
```

### L2 产物消费方

```text
所有 A-F 端口。
```

---

## 4. L3 外部桥接技能更新决策

### L3 负责什么

```text
接入外部方法论、框架或 repo 的能力，但保持边界清楚。
```

### L3 当前典型项

```text
- spec-kit bridge
- superpowers bridge
- dbs 系列桥接
- 外部文章方法桥接
```

### L3 更新决策

```text
- 动作：bridge / pattern only / experimental 最常见。
- 适用场景：外部方法有价值，但还不想直接内化成主技能。
```

### L3 规则

```text
- 先桥接，后内化。
- 外部方法不直接冒充本地主 skill。
- 没有稳定重复价值的，只做 pattern only。
```

### L3 产物消费方

```text
A 端概念解析、B 端证据结构化、C 端理论生成、E 端审查模板。
```

---

## 5. L4 领域专用技能更新决策

### L4 负责什么

```text
承载当前业务域/研究域的具体知识与表达。
```

### L4 当前典型项

```text
- 结构术语解析
- 样本标注
- 反证样本收集
- MQL5 输出辅助
- 图表/指标解释
- 领域词典维护
```

### L4 更新决策

```text
- 动作：split / merge / replace / deprecate 常见。
- 适用场景：领域知识变化、术语升级、指标重定义、样本结构调整。
```

### L4 规则

```text
- 领域技能只服务当前项目。
- 如果领域变化明显，优先抽出模板或桥接层。
- 业务术语要和主干端口分开管理。
```

### L4 产物消费方

```text
B/C/D/E 端，以及知识库主线文档。
```

---

## 6. L5 模板/交付技能更新决策

### L5 负责什么

```text
统一输出格式、交付节奏、handoff 文档。
```

### L5 当前典型项

```text
- Demand Grilling Brief 模板
- Source Pack 模板
- Theory Package 模板
- Landing Handoff 模板
- Verification Report 模板
- Owner Decision Brief 模板
- 一页执行面板
- 新 bot 初始化 prompt 包
```

### L5 更新决策

```text
- 动作：replace / split / merge / alias / deprecate。
- 适用场景：模板字段变化、输出标准变化、交付链路变化。
```

### L5 规则

```text
- 模板要尽量稳定。
- 模板是交付标准，不是执行本身。
- 如果新模板只是旧模板的轻微改写，优先 alias 或 replace。
```

### L5 产物消费方

```text
所有端口的输出与落库文档。
```

---

## 7. L6 状态/边界标记更新决策

### L6 负责什么

```text
标记技能是否真的执行、是否仅引用、是否弃用、是否受限。
```

### L6 当前典型项

```text
listed_only
installed
loaded
activated
executed
output_consumed
referenced_only
conditional_not_executed
missing
forbidden
deprecated
alias
```

### L6 更新决策

```text
- 动作：add / alias / deprecate / remove。
- 适用场景：执行态变化、技能状态变更、旧名保留、新名切换。
```

### L6 规则

```text
- 任何技能都必须有状态。
- 不要把 referenced_only 当 executed。
- deprecated 技能建议保留 alias 一段时间。
```

### L6 产物消费方

```text
所有端口、验证层、prompt 包、索引记录。
```

---

## 8. 推荐更新顺序

```text
1. 先看是不是 L1 主技能问题。
2. 如果不是，再看 L2 通用辅助。
3. 再看 L3 外部桥接是否需要内化。
4. 再看 L4 领域专用是否该拆分或重命名。
5. 再看 L5 模板是否要统一。
6. 最后用 L6 标记状态和边界。
```

---

## 9. 一个简单的改法判断表

| 现象 | 建议动作 |
|---|---|
| 一个技能负责太多事 | split |
| 两个技能做同一件事 | merge |
| 名字变了但职责没变 | alias |
| 外部方法想接进来 | bridge |
| 功能稳定但名字不准 | replace |
| 长期不用且制造混乱 | deprecate / remove |

---

## 10. 结论

```text
L1-L6 都需要，但不是同等方式更新：

- L1 保稳定
- L2 保复用
- L3 保桥接
- L4 保领域
- L5 保交付
- L6 保边界

这样技能体系才能既能增长，又不会越长越乱。
```
