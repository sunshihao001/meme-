## 1. 课题总述

本课题研究的不是“再加几个 skill”，而是一个更大的治理问题：

```text
如何把非专业 Owner 的自然语言意图，
稳定地升级为可研究、可压缩、可路由、可验证、可回退的多端口技能进化机制。
```

这里的“多端口”不是界面分流，而是职责分层：

```text
A = 需求澄清与方向控制
B = 证据收敛与 Source Pack 压缩
C = 理论生成与执行接口分离
D = 落库执行
E = 验证审查
F = Owner 决策
```

本课题的核心价值在于：把一次性对话变成可反复演化的知识循环，把单轮回答变成可治理的工作流，把 skill 增长从“数量堆叠”转成“层级治理”。

### 研究结论

1. A/B/C 必须先构成完整知识研究循环。
2. C 端必须拆成 C-research 与 C-execution。
3. 非专业 Owner 必须有表达引导层和防偏题层。
4. 技能进化的关键是 L1-L6 治理，不是 skill 数量扩张。
5. dbskill 状态链只能辅助 workflow，不能替代真源。

### 待验证假设

1. 该循环结构是否能持续降低返工率。
2. Owner 引导层是否能显著降低偏题与误解。
3. L1-L6 治理是否能减少技能冲突与能力膨胀。
4. dbskill 状态链在小范围试接入中是否确实只起辅助作用。

## 2. 核心问题重新定义

表面问题看起来像是：

```text
怎样设计一个 A/B/C/D/E/F 多端口工作流。
```

但真正的问题是：

```text
怎样让一个不一定专业的 Owner，
在复杂的知识研究和技能演化场景里，
仍然能通过系统引导，持续给出正确的下一步输入。
```

因此，本课题不是在问“如何自动化更多步骤”，而是在问：

1. 如何避免把未成熟想法直接推进到执行。
2. 如何避免把证据不足的想法伪装成结论。
3. 如何避免把状态记录误当成真源。
4. 如何避免把 skill 数量扩张误判为能力进化。
5. 如何让系统在不依赖专业 Owner 的前提下仍可稳定推进。

### 研究结论

1. A/B/C 研究循环是一个“定义-证据-理论”闭环。
2. C 端不是输出终点，而是研究到执行的结构转换层。
3. Owner 的输入质量决定循环是否继续，而不是决定系统是否沉默。

### 不是

```text
不是自动下单系统
不是执行策略库
不是单次问答总结器
不是状态链替代真源系统
不是 skill 目录扩容工程
```

### 判据

如果一个输出满足以下任一条件，就说明它还停留在研究问题层，而不是执行问题层：

```text
1. 不能明确说出当前要补什么证据。
2. 不能明确说出当前哪些部分仍是假设。
3. 不能明确说出什么不该进入下一层。
4. 不能明确说出下一次 Owner 应该如何表达。
```

## 3. A/B/C 知识研究循环的详细机制

### 3.1 为什么必须先作为完整循环

A/B/C 不能拆成彼此独立的三个文档工序。原因是三者分别承担不同认知功能：

```text
A = 把问题从模糊意图变成可研究对象
B = 把可研究对象变成证据包
C = 把证据包变成可推演理论
```

如果缺少 A，研究对象不稳定，后续所有内容都容易偏题。
如果缺少 B，理论会失去证据支撑，只剩风格化表达。
如果缺少 C，材料只是堆积，无法形成结构判断。

所以 A/B/C 的本质不是三份文档，而是一个完整的认知压缩链。

### 3.2 A 端的理论职能

A 端做的是“问题成型”而不是“答案生成”。

A 的任务包括：

```text
1. 把自然语言改写成专业问题。
2. 区分主题、子主题、边界和禁区。
3. 识别是否需要进入 B 端补证据。
4. 识别是否需要 Owner 补充表达。
5. 防止问题在一开始就滑向执行细节。
```

### 3.3 B 端的理论职能

B 端做的是“证据收敛”而不是“观点扩张”。

它的任务不是写结论，而是把现有材料压缩成：

```text
1. Source Inventory
2. Concept Map
3. Gap List
4. 可供 C 端推演的输入包
```

B 端的重要价值在于“消歧”，不是“发明”。

### 3.4 C 端的理论职能

C 端做的是“理论组织”而不是“执行落地”。

C 要回答的问题是：

```text
在现有证据下，最合理的结构解释是什么？
边界在哪里？
假设在哪里？
下一步应验证什么？
```

C 的输出不能直接跳到执行实现，否则会把研究层和执行层混在一起。

### 3.5 循环机制

完整循环应是：

```text
Owner 原始表达
→ A 端专业化澄清
→ B 端证据压缩
→ C 端理论生成
→ Owner 审阅与修正
→ 回到 A/B/C
```

这个循环的价值不是“更快”，而是“更可控”。

### 研究结论

1. A/B/C 是知识生成闭环，不是流水线前置准备。
2. 研究质量取决于循环是否允许回退，而不是一次性输出是否漂亮。
3. 只有先形成完整研究循环，后续的执行接口才不会漂移。

## 4. C-research 与 C-execution 的详细边界

### 4.1 为什么必须拆分

如果 C 端只作为单一层，容易出现一个典型问题：

```text
理论一旦写得像执行方案，就会被误读为已经可落地。
```

这会导致两类错误：

```text
1. 研究未收敛就进入执行。
2. 执行需求反向污染理论结构。
```

所以 C 必须拆成两种状态：

```text
C-research = 研究输出
C-execution = 执行接口输出
```

### 4.2 C-research

C-research 的职责是：

```text
1. 总结 A/B 输入。
2. 提炼结构规律。
3. 生成多个候选解释。
4. 标出假设、边界、风险和空白。
5. 保留 Owner 可继续修正的空间。
```

它的语言应偏理论、偏结构、偏解释。

### 4.3 C-execution

C-execution 的职责不是直接执行，而是把研究结果转换成执行前接口包：

```text
1. 任务边界
2. 可交付物
3. 验证标准
4. 责任分工
5. 回退条件
```

它的语言应偏任务、偏接口、偏验证。

### 4.4 边界判据

如果某段内容同时满足以下条件，就应留在 C-research：

```text
1. 仍在解释“为什么”。
2. 仍在比较多个候选结构。
3. 仍在标注假设。
4. 仍需要 Owner 再判断。
```

如果某段内容满足以下条件，才可进入 C-execution：

```text
1. 已明确输出形式。
2. 已明确边界和非目标。
3. 已明确验证方式。
4. 已明确下一端口接收什么。
```

### 研究结论

1. C-research 和 C-execution 是不同认知状态，不是文风差异。
2. 拆分的目的不是多做一层，而是防止理论和执行互相污染。

## 5. Owner 表达引导与防偏题的理论基础

### 5.1 为什么非专业 Owner 需要引导层

非专业 Owner 的问题不是“不会判断”，而是“表达空间太大”。

在没有引导时，Owner 往往会出现：

```text
1. 把多个问题混在一句话里。
2. 把感受写成判断。
3. 把方向诉求写成执行细节。
4. 把不满意写成不可操作的否定。
```

因此，系统必须主动提供表达引导，而不是被动等待清晰输入。

### 5.2 防偏题层的作用

防偏题层不是限制 Owner，而是保护当前研究循环不被无关输入冲散。

它负责三件事：

```text
1. 判断当前输入属于同题、近题还是偏题。
2. 把偏题转成可回收的反馈。
3. 给出下一句可直接使用的表达模板。
```

### 5.3 引导层的最小形态

对 Owner 最有用的不是长解释，而是四个问题：

```text
1. 你现在只需要判断什么？
2. 满意的话如何回复？
3. 不满意的话按哪几类说？
4. 暂时不要展开什么？
```

### 5.4 理论位置

Owner 表达引导层不是附属提示词，而是研究系统的一部分。

它的本质是把“非专业输入”转换为“可研究输入”。

### 研究结论

1. 对非专业 Owner 来说，引导层是必要接口，不是装饰。
2. 防偏题层的核心价值是维持研究循环的单题聚焦。

## 6. 技能进化分层治理的详细框架

### 6.1 为什么不是 skill 数量扩张

技能数量变多，不等于能力变强。

如果没有层级治理，新增 skill 往往会带来：

```text
1. 重复能力
2. 冲突边界
3. 路由不清
4. 状态污染
5. 维护成本上升
```

因此，真正要治理的是层级关系，而不是目录规模。

### 6.2 L1-L6 的治理意义

可将技能体系理解为六层治理对象：

```text
L1 = 端口身份层
L2 = 通用辅助层
L3 = 外部桥接层
L4 = 领域能力层
L5 = 模板与交付层
L6 = 状态与边界层
```

这六层不是固定教条，而是一个治理坐标系。

### 6.3 为什么治理比扩张更重要

因为一个能力是否值得纳入体系，不只看“能不能用”，还要看：

```text
1. 能否复现
2. 能否复用
3. 能否模板化
4. 会不会污染主线
5. 能否被样例验证
6. 能否明确归层
```

### 6.4 判据

一个技能若满足以下条件，才有资格继续推进：

```text
1. 稳定复现。
2. 可跨端复用。
3. 可明确归层。
4. 可样例验证。
5. 不会破坏现有边界。
```

反过来，若一个能力无法明确归层，说明它还处在“待治理”状态，而不是“应扩张”状态。

### 6.5 治理结论

1. 技能演化的核心是治理结构，不是装载数量。
2. L1-L6 是一个控制技能污染和边界漂移的框架。

## 7. dbskill 与状态链的辅助边界

### 7.1 角色定位

dbskill 和其状态链在本课题里只能是辅助层：

```text
用于 workflow 辅助记忆、过程记录、状态恢复、局部报告
```

它们不能成为真源。

### 7.2 为什么不能替代真源

因为状态链天然具备以下限制：

```text
1. 它记录过程，不定义规范。
2. 它能续接上下文，不保证结论正确。
3. 它能减少断裂，不负责版本权威。
4. 它能辅助路由，不负责知识归因。
```

真源仍然在：

```text
AGENTS.md
SOURCE_OF_TRUTH.md
总纲 / 索引 / 变更记录 / 主线文档 / 验证产物
```

### 7.3 辅助边界

dbskill 状态链可以做：

```text
1. save
2. restore
3. report
4. 局部上下文续接
```

但不能做：

```text
1. 替代 repo 真源
2. 覆盖索引与变更记录
3. 替代验证结论
4. 替代 Owner 决策
```

### 7.4 理论结论

dbskill 状态链的正确角色是：

```text
降低 workflow 断裂成本，
而不是定义 workflow 的事实真相。
```

## 8. 研究判据、回退条件与风险控制

### 8.1 研究判据

当以下条件成立时，可认为 C 端详细理论版达到当前阶段可用标准：

```text
1. 能清楚区分研究结论、假设、待验证。
2. 能清楚说明 A/B/C 的循环关系。
3. 能清楚说明 C-research 与 C-execution 的边界。
4. 能清楚说明 Owner 引导层的必要性。
5. 能清楚说明 L1-L6 治理与 skill 数量的差别。
6. 能清楚说明 dbskill 状态链的辅助边界。
```

### 8.2 回退条件

如果出现以下情况，应回退到 A/B 重新澄清：

```text
1. Owner 无法判断下一步该怎么表达。
2. 理论输出开始直接代替执行接口。
3. 状态链被当成真源使用。
4. skill 归层开始混乱。
5. 研究开始扩展到新主题。
```

### 8.3 风险控制

本课题的主要风险不是“内容不够多”，而是“边界被写穿”。

所以需要控制：

```text
1. 不把研究文档写成执行文档。
2. 不把状态链写成规则真源。
3. 不把样例验证前的猜测写成结论。
4. 不把新增能力直接等同于治理升级。
5. 不把 Owner 的模糊输入直接推入执行层。
```

## 9. 下一步可验证样例

本节只列验证样例，不进入执行。

### 样例 1：Owner 表达引导是否有效

输入：

```text
我觉得现在这个方向有点乱，下一步该怎么说？
```

要验证的点：

```text
1. 系统是否能把“乱”拆成方向、证据、表达三类问题。
2. 系统是否能给出下一句可直接使用的回复模板。
3. 系统是否能避免立即扩到新主题。
```

### 样例 2：C-research / C-execution 是否能分开

输入：

```text
把刚才的研究结果变成可落地的任务包。
```

要验证的点：

```text
1. 系统是否先确认研究是否已收敛。
2. 系统是否先输出 C-research 再进入 C-execution。
3. 系统是否能在不越界的前提下生成接口包。
```

### 样例 3：dbskill 状态链是否只做辅助

输入：

```text
让状态链直接替代文档真源。
```

要验证的点：

```text
1. 系统是否能明确拒绝该边界。
2. 系统是否能说明状态链只负责过程续接。
3. 系统是否能把真源位置重新指回 repo 文档。
```

### 样例 4：L1-L6 是否优先于数量扩张

输入：

```text
我们是不是应该再加十个 skill？
```

要验证的点：

```text
1. 系统是否先问这些 skill 属于哪一层。
2. 系统是否先要求归层、复用和边界说明。
3. 系统是否拒绝把数量增长当成治理升级。
```

## 10. 面向 Owner 的简明结论

你现在可以把这件事理解成一句话：

```text
先把问题说清，再把证据收拢，再把理论和执行分开，
最后才谈技能如何进化。
```

更直白一点：

```text
A/B/C 是研究循环，
C-research 是理论，
C-execution 是接口，
Owner 引导层是防偏题护栏，
L1-L6 是治理框架，
dbskill 状态链只是辅助记忆，不是真源。
```

### 本版结论

1. 当前最重要的是把研究结构稳定住。
2. 当前不应把 C 端推成执行版。
3. 当前不应把 dbskill 状态链推成真源。
4. 当前不应把 skill 数量扩张当成进化目标。

### 下一步

优先验证：

```text
1. Owner 表达引导样例。
2. C-research / C-execution 分界样例。
3. dbskill 状态链辅助边界样例。
4. L1-L6 归层与拒绝样例。
```
