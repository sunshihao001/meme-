# dbskill 第二批接入执行卡片 v0.1

> 分线：10_工程化交接 / 外部技能接入  
> 来源：dbskill 外部仓库评估 + 接入映射表  
> 目标：把第二批最有价值的 4 个 skill（决策与状态链）写成可执行、可验证、可兼容的落地卡片。

---

## 卡 01：dbs-decision

### 目标

```text
把 dbs-decision 接入 F 端的 Owner Decision / 决策回填流程。
```

### 归属

```text
端口：F
层级：L2/L5/L6
动作：MERGE / BRIDGE
```

### 与现有体系关系

```text
它补强的是“决策事件建档、结果回填、阶段画像、规律沉淀”。
```

### 预期输出

```text
- 决策事件文件
- 事实 / 规律 / 定格 / 待解 四层落盘
- 状态画像 / 阶段快照
```

### 接入方式

```text
- 保留 alias
- 作为 F 端决策沉淀能力
- 与 Owner Decision Brief 联动
```

### 验证方式

```text
- 是否能把一个决策域整理成可回填知识工程
- 是否能把决策历史变成可复用资产
```

### 风险

```text
- 如果不控制边界，可能把所有事情都写成决策工程，导致过度文档化
```

---

## 卡 02：dbs-save

### 目标

```text
把 dbs-save 接入状态保存链，作为跨对话的状态快照能力。
```

### 归属

```text
端口：B/F
层级：L5/L6
动作：BRIDGE
```

### 与现有体系关系

```text
它用于保存当前判断、关键结论、否决方向、下一步。
```

### 预期输出

```text
- 可恢复的快照文件
- 当前状态摘要
- 决策结论存档
```

### 接入方式

```text
- 与现有任务表/状态矩阵联动
- 保留 alias
- 不取代仓库里的正式变更记录
```

### 验证方式

```text
- 是否能在下次对话中恢复上次上下文
- 是否能避免每次重复解释背景
```

### 风险

```text
- 如果存档位置不统一，容易和 repo 真源分离
```

---

## 卡 03：dbs-restore

### 目标

```text
把 dbs-restore 接入跨对话续诊能力，作为状态恢复入口。
```

### 归属

```text
端口：A/B/F
层级：L5/L6
动作：BRIDGE
```

### 与现有体系关系

```text
它补强的是“接着上次继续判断”，避免上下文断裂。
```

### 预期输出

```text
- 最近一份快照恢复
- 最近决策上下文恢复
- 最近问题/假设恢复
```

### 接入方式

```text
- 保留 alias
- 作为状态链入口
- 与 save/report 配合使用
```

### 验证方式

```text
- 是否能正确恢复上次判断状态
- 是否能快速回到未完成的问题节点
```

### 风险

```text
- 如果恢复的是旧错误判断，可能把错误状态一起续上，所以要配合时间和版本标记
```

---

## 卡 04：dbs-report

### 目标

```text
把 dbs-report 接入成决策/诊断/状态的汇总报告能力。
```

### 归属

```text
端口：E/F
层级：L5
动作：BRIDGE
```

### 与现有体系关系

```text
它可以作为 Review Report / Owner Brief 的辅助输出。
```

### 预期输出

```text
- 多次存档合并报告
- 时间索引
- 结论摘要
```

### 接入方式

```text
- 保留 alias
- 和 save / restore 配合
- 不替代正式的变更记录/索引
```

### 验证方式

```text
- 是否能把多个快照压缩成可读的报告
- 是否能清楚显示演化过程
```

### 风险

```text
- 如果报告和原始存档混在一起，后续会难以追溯
```

---

## 第二批执行顺序

```text
1. dbs-decision
2. dbs-save
3. dbs-restore
4. dbs-report
```

原因：

```text
这四个补“持续判断能力”和“跨对话状态能力”。
```
