## 2. 课题缘起

今天的想法并不是单独在讨论某一个 skill，而是在逐步形成一个更大的问题：

```text
一个非专业 Owner 用自然语言提出想法时，
AI 多端口系统如何把这些想法升级为专业研究课题、稳定知识框架、可执行方案，
并通过技能库、状态链、执行端口和验证端口持续进化？
```

此前容易出现的偏差是：

```text
1. 一有想法就急着 D端落库。
2. 一看到外部 skill 就想接入或安装。
3. A/B/C/D/E/F 被理解成单线流程。
4. C端被误认为只负责一次性写方案。
5. 技能更新被理解成“多加几个 skill”，而不是内部组合能力进化。
6. 状态链被理解成记忆工具，而不是 workflow loop 的可追溯控制层。
7. 非专业 Owner 自己输入时容易偏离主题，需要系统反过来引导 Owner 如何表达、如何补充、如何确认方向。
```

今天的核心修正是：

```text
A/B/C 先形成完整知识研究循环；
Owner 不满意可以随时调整；
满意后由 C 端输出执行版方案；
再进入 D/E/F 执行、验证、决策。
```

---

## 3. 核心研究问题

本课题要研究的主问题：

```text
如何构建一个由 A/B/C 知识研究循环驱动、由 C-D-E/F 执行验证循环承接的多端口技能进化系统？
```

拆成 7 个子问题：

```text
Q1. 用户自然语言想法如何被 A端升级成专业研究问题？
Q2. B端应该如何把资料、外部文章、GitHub skill、上下文压缩成 Source Pack？
Q3. C端如何在研究态生成理论框架，又如何在执行态生成可落库方案？
Q4. 什么情况下 Owner 不满意应回 A、回 B、回 C，而不是直接进入 D？
Q5. 内部组合技能应如何分层、更新、桥接、合并、弃用？
Q6. 外部 GitHub skill 应如何判断 ADOPT / BRIDGE / MERGE / REJECT / PATTERN_ONLY？
Q7. dbskill 第二批状态链如何辅助 save / restore / report，而不替代 git、E端验证和 F端决策？
Q8. 面向非专业 Owner，系统如何引导其正确表达下一步输入，防止自己越说越偏题？
```

## 4. 课题对象

本课题不是单纯研究 meme 策略，也不是单纯研究 AI 编程工具。

它研究的是：

```text
多端口 AI 工作流本身如何进化。
```

当前对象包括：

```text
1. A端：自然语言 → 专业问题 → 研究路由
2. B端：资料收集 → evidence/source pack → 上下文压缩
3. C端：理论生成 → 框架设计 → 执行版方案
4. D端：repo landing / 文件落库 / 索引 / commit
5. E端：验证 / CI / 反证 / review
6. F端：Owner 决策 / 授权 / 阶段门
7. 内部组合技能库：L1-L6 分层
8. 外部 skill 接入流水线
9. dbskill 第一批判断能力与第二批状态链
10. Owner 表达引导层：帮助非专业用户知道下一步该怎么说、怎么补充、怎么判断是否偏题
```

## 5. 研究假设

### H1：A/B/C 应先作为知识研究循环，而不是线性前置步骤

```text
如果 A/B/C 没有形成闭环，系统会过早执行错误方案。
```

验证方式：

```text
观察是否能在 D端落库前发现问题方向、资料缺口、理论不足。
```

---

### H2：C端必须拆成 C-research 与 C-execution

```text
C-research 可以发散、推演、比较。
C-execution 必须收敛、可落库、可验证。
```

验证方式：

```text
检查 C端输出是否能明确区分“研究版方案”和“执行版任务包”。
```

---

### H3：技能更新不是新增 skill，而是组合能力的结构进化

```text
真正需要管理的是 L1-L6 的组合关系，而不是 skill 数量。
```

验证方式：

```text
每次技能更新都必须说明：新增、合并、桥接、别名、弃用、暂缓或 pattern only。
```

---

### H4：外部 skill 只能经过 A端接入判断门

```text
外部 skill 仓库不能直接污染主线。
```

验证方式：

```text
每个候选外部 skill 必须经过 ADOPT / BRIDGE / MERGE / REJECT / PATTERN_ONLY 判断。
```

---

### H5：状态链只能辅助 workflow，不应替代真源

```text
dbs-save / dbs-restore / dbs-report 的价值是减少上下文断裂，
不是替代 repo、索引、变更记录、E端验证或 F端决策。
```

验证方式：

```text
所有状态链输出必须能回指 repo 内文件，而不是成为 repo 外独立真源。
```

---

### H6：系统必须引导非专业 Owner 的下一步表达

```text
Owner 不是专业研究员或工程师，直接自由输入时很容易偏离主题。
因此 A端不能只等待 Owner 给出完美问题，而要在每轮输出后给出“下一步你可以这样表达”的引导。
```

验证方式：

```text
每次 A/B/C 研究循环输出后，都应附带 Owner 表达引导：
1. 当前你只需要判断什么；
2. 如果满意，可以怎么说；
3. 如果不满意，可以按哪个方向说；
4. 哪些输入会偏题，暂时不要展开；
5. 推荐下一句 Owner 输入模板。
```

## 6. 研究框架

### 6.1 双循环框架

```text
第一循环：A/B/C 知识研究循环

A：定义问题、升级自然语言、控制方向
B：收集资料、压缩上下文、形成 Source Pack
C：生成理论、框架、候选方案
Owner：不满意可随时回 A/B/C 调整

第二循环：C-D-E/F 执行验证循环

C-execution：执行版方案 / spec / task package
D：落库 / 修改文件 / 索引 / commit / PR
E：验证 / CI / 反证 / review
F：Owner 决策 / 授权 / 阶段门
```

---

### 6.2 技能分层框架

```text
L1 端口主技能：A/B/C/D/E/F 的身份核
L2 通用辅助技能：dbs-good-question、ai-method-wheel 等
L3 外部桥接技能：dbskill、spec-kit、superpowers 等桥接层
L4 领域专用技能：meme/FCZ/MQL5/样本标注等
L5 模板交付技能：brief/source pack/report/owner decision 等模板
L6 状态边界标记：activated/executed/referenced_only/deprecated 等
```

---

### 6.3 外部技能接入框架

```text
ADOPT：直接接入
BRIDGE：桥接使用
MERGE：合并进现有能力
REJECT：拒绝接入
PATTERN_ONLY：只吸收模式
```

判断顺序：

```text
外部能力是什么？
补哪个内部缺口？
对应哪个端口？
属于 L1-L6 哪层？
是否污染主线？
是否能验证？
Owner 是否需要决策？
```

---

### 6.4 状态链框架

```text
dbs-decision：记录 F端决策事件
dbs-save：保存 repo 内 workflow snapshot
dbs-restore：从 repo 内 state/queue/snapshot 恢复
dbs-report：辅助 E/F 汇总报告
```

边界：

```text
不替代 git
不替代索引
不替代变更记录
不替代 E端验证
不替代 F端 Owner 决策
不变成全局 runtime 记忆
```

---

### 6.5 Owner 表达引导框架

因为 Owner 不是专业研究员、工程师或 AI workflow 架构师，所以系统不能假设 Owner 每次都能输入正确问题。

每轮 A/B/C 输出后，必须附带一个轻量表达引导：

```text
1. 你现在只需要判断：______
2. 如果你满意，可以回复：______
3. 如果你不满意，可以按这三类说：
   - 方向不对：______
   - 资料不够：______
   - 理论/方案不够好：______
4. 暂时不要展开：______
5. 推荐下一句输入：______
```

A端还要主动识别 Owner 输入是否偏题：

```text
1. 如果只是表达模糊但方向相关 → A端帮忙重写成专业问题。
2. 如果引入新主题但可能有关 → A端标记为“旁支候选”，问是否切换。
3. 如果明显偏离当前课题 → A端提醒先完成当前判断，再另开话题。
4. 如果 Owner 情绪化表达不满意 → A端转成可操作的回退类型：回 A / 回 B / 回 C。
```

## 7. 研究方法

本课题不应该只写理论，而应该用真实样例逐轮验证。

建议采用：

```text
样例驱动研究法
```

每轮样例流程：

```text
1. Owner 提出一个自然语言想法。
2. A端先判断该输入是否偏题、是否需要重写、是否要引导 Owner 换一种表达。
3. A端生成研究问题和路由判断。
4. B端生成 Source Pack 或上下文证据包。
5. C端生成研究版理论/方案。
6. 系统给 Owner 一个“下一步怎么表达”的轻量模板。
7. Owner 判断是否满意。
8. 不满意则按表达引导回 A/B/C。
9. 满意后 C 生成执行版方案。
10. D/E/F 再进入执行验证循环。
11. dbskill 状态链记录本轮状态变化。
12. E端验证是否减少上下文断裂、是否污染主线。
```

## 11. 成功标准

本课题成功，不是因为写了很多文档，而是因为系统能做到：

```text
1. 用户自然语言想法能被稳定升级成专业问题。
2. A/B/C 能多轮研究，不满意可调整。
3. C端能区分研究版与执行版。
4. D/E/F 不再过早执行未成熟想法。
5. 外部 skill 不再污染主线。
6. 内部组合技能更新有层级、有状态、有决策表。
7. dbskill 状态链能减少上下文断裂，但不替代真源。
8. 系统每轮都能告诉非专业 Owner 下一步应该怎么表达、怎么判断满意/不满意、哪些话题暂时不要展开。
9. 下一个 bot 或下一个对话能从 repo 文件恢复课题进度。
```

## 12. 当前结论

今天散发的想法可以统一收束为一个主课题：

```text
研究如何让 AI 多端口系统先通过 A/B/C 完成可调整的知识研究，
再由 C端把成熟方案转成执行包，
交给 D/E/F 做落库、验证和决策，
并让整个过程中使用到的技能库持续分层进化。
```

这个课题的本质不是“装更多 skill”，而是：

```text
建立一种 Owner 可控、知识可研究、方案可执行、结果可验证、技能可进化的 AI 工作流机制。
```
