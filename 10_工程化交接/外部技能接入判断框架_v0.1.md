# 外部技能接入判断框架 v0.1

> 分线：10_工程化交接 / 外部技能接入  
> 状态：v0.1 / 判断框架草案  
> 目的：当我要进入别人的 GitHub 仓库去看技能、准备添加外部技能时，先判断“该不该接、怎么接、接到哪一层、是否需要桥接/合并/拒绝”，避免盲目堆技能。

---

## 1. 核心变化

以前的问题是：

```text
内部技能不够 → 直接想加外部技能
```

现在需要变成：

```text
内部缺口识别 → 外部候选筛选 → 价值判断 → 接入方式判断 → 兼容性判断 → 落点判断 → 验证判断
```

---

## 2. 外部技能接入的四类结果

### A. Adopt（直接接入）

适用条件：

```text
- 该技能和当前体系高度匹配
- 职责清晰
- 输出稳定
- 可重复使用
- 直接能放到 L1-L5 的某一层
```

结果：

```text
- 直接纳入主栈
- 写入别名映射 / 状态矩阵 / 任务表
- 必要时加模板
```

---

### B. Bridge（桥接接入）

适用条件：

```text
- 技能有价值，但还不适合直接升格为主干
- 外部语义需要保留
- 需要本地包装成更稳定的内部能力
```

结果：

```text
- 先做桥接层
- 保留外部名或引用
- 不直接冒充主 skill
```

---

### C. Merge（并入现有能力）

适用条件：

```text
- 外部技能和现有技能重复
- 只是命名不同
- 可以作为别名或子模块并入现有能力组
```

结果：

```text
- 并入 L2/L3 的现有组合
- 保留 alias
- 记录来源
```

---

### D. Reject（拒绝接入）

适用条件：

```text
- 价值不清
- 只适合一次性场景
- 维护成本过高
- 和当前方法轮冲突
- 无法证明稳定复用
```

结果：

```text
- 不纳入主栈
- 仅保留研究记录
- 不写成技能资产
```

---

## 3. 判断顺序

建议按以下顺序判断：

```text
1. 这是什么？（归类）
2. 它解决什么缺口？（价值）
3. 它能放到哪一层？（落点）
4. 它和现有技能重复吗？（合并/别名）
5. 它需要桥接还是直接接入？（方式）
6. 它会不会破坏现有体系？（兼容性）
7. 接入后怎么验证？（验证）
```

---

## 4. 判断维度

### 4.1 价值维度

```text
- 是否解决一个明确的缺口
- 是否能减少重复劳动
- 是否能提升判断质量
- 是否能提升协作/验证/交付效率
```

### 4.2 稳定性维度

```text
- 是否有稳定的输入输出
- 是否可重复使用
- 是否不依赖一次性上下文
- 是否能被模板化
```

### 4.3 兼容性维度

```text
- 是否和 L1-L6 体系兼容
- 是否和现有别名/模板冲突
- 是否会造成职责重叠
- 是否需要保留旧名 alias
```

### 4.4 落点维度

```text
- L1 主技能
- L2 通用辅助
- L3 外部桥接
- L4 领域技能
- L5 模板交付
- L6 状态边界
```

### 4.5 维护成本维度

```text
- 是否值得长期维护
- 是否会让体系变乱
- 是否需要额外验证/索引/别名
```

---

## 5. 接入前必须回答的 7 个问题

```text
1. 这个外部技能的真实用途是什么？
2. 它对应我体系里的哪个缺口？
3. 它是主干、桥接、别名，还是仅参考？
4. 它和现有技能是否重复？
5. 它的输出能否标准化？
6. 它会不会改变我当前的端口职责？
7. 接入后如何验证它真的有用？
```

---

## 6. 判断结论模板

外部技能评估后，统一输出下面这类结论之一：

```text
ADOPT
BRIDGE
MERGE
REJECT
```

并附带说明：

```text
- 归属层级
- 兼容方式
- 是否保留 alias
- 是否需要模板
- 是否需要验证卡
```

---

## 7. 推荐的实践流程

### Step 1：先做候选池

```text
从外部仓库提取候选技能，不直接安装。
```

### Step 2：做评估表

```text
每个候选技能都填：
- 名称
- 来源
- 用途
- 缺口
- 价值
- 稳定性
- 兼容性
- 动作
```

### Step 3：做落点决策

```text
决定放在 L1-L6 哪层，还是拒绝。
```

### Step 4：做兼容处理

```text
决定 alias / deprecated / bridge / merge。
```

### Step 5：做验证

```text
验证它是否真的提升了判断和执行质量。
```

## 9. 和你当前体系的关系

你现在不是单纯“补技能”，而是要做：

```text
内部技能不足 → 外部技能评估 → 接入判断 → 体系落点 → 验证归档
```

这会直接接上你现在的：

```text
A-F 六端
L1-L6 分层
任务表 / 状态矩阵 / 别名映射
```

---

## 10. 结论

```text
外部技能不是“看到了就加”，而是先判定：
它是不是缺口、值不值得接、接到哪一层、怎么兼容、怎么验证。
```

这就是你要的判断推理能力的核心。
