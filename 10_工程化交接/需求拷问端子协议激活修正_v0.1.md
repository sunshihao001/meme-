## 1. 本轮正确表达

Owner 反馈：

```text
对，是存在，但是没有运用起来。
```

正确转写为：

```text
问题已经不是 installed / missing，而是 activation / execution / consumption。
`intent-brainstorm-grill` 已作为 `spec-first-ai-engineering` 的 `references/intent-brainstorm-grill.md` 子协议存在，但需求拷问端运行时没有稳定证明：它被激活、按 funnel 执行、产出澄清 artifact、并被后续 spec / issue / handoff / validator 消费。
```

一句话：

```text
存在 ≠ 运用；loaded_reference ≠ activated_subprotocol；activated_subprotocol ≠ executed_funnel；executed_funnel ≠ output_consumed。
```

## 2. 状态层级修正

以后需求拷问端报告必须区分 6 层：

```text
1. parent_skill_installed：父 skill 存在，例如 spec-first-ai-engineering。
2. subprotocol_present：子协议文件存在，例如 references/intent-brainstorm-grill.md。
3. subprotocol_loaded：本轮通过 skill_view(..., file_path=...) 实际加载。
4. subprotocol_activated：本轮明确选择该子协议作为运行门，而不是只读了一下。
5. subprotocol_executed：按协议完成必要 funnel，并产出 artifact。
6. subprotocol_output_consumed：该 artifact 被下游 spec / issue / handoff / validator / project patch 使用。
```

当前缺陷是：

```text
项目已经做到了 1-3；
但 4-6 没有形成稳定运行证据。
```

---

## 3. intent-brainstorm-grill 的激活条件

当本轮请求满足任一条件时，应激活 `intent-brainstorm-grill`：

```text
1. Owner 的问题仍然模糊，可能导致后续 Agent 做错方向。
2. Owner 正在质疑需求拷问端、方法轮、skill 是否真正运行。
3. 任务是战略、研究、产品定义、知识库结构化、长期 loop，而不是单次机械编辑。
4. 需要进入 spec / issue / Codex / PR / CI 前，但 goal / users / non-goals / assumptions / risks / acceptance criteria 仍不完整。
5. 需要从“用户表面说法”转成专业问题空间和可执行收敛切片。
```

不必激活的情况：

```text
1. 用户要求一个明确机械操作，且上下文、文件路径、验收标准都清楚。
2. 本轮只是验证、提交、查 CI。
3. Owner 已显式批准跳过澄清门。
```

## 4. 最小执行协议

激活后，不要求每轮写长文档，但至少要跑完以下最小 funnel：

```text
Goal：本轮到底要改变什么结果？
Users / Operators：谁使用、维护、审核、承担后果？
Boundaries / Non-goals：本轮不做什么？
Assumptions：哪些前提如果错了会导致输出失效？
Failure Modes：它怎样会误导、空转、越权、或制造假完成？
Options Compared：至少比较 2 个可行路径，含一个更轻/更安全路径。
Acceptance Criteria：什么证据证明本轮拷问已经完成？
```

如果 Owner 在线且缺关键输入：

```text
一次只问一个最高价值问题。
```

如果可以从项目文件判断：

```text
先给 recommended default，并标注 PROPOSED，等待 Owner 决策。
```

---

## 5. 最小 artifact 要求

每次激活必须至少留下一个 durable artifact：

```text
A. 更新 loop report 的 subprotocol_activation_evidence；或
B. 写入 specs/<feature>/clarification.md；或
C. 写入 10_工程化交接/<主题>_需求拷问子协议运行记录.md；或
D. 更新当前需求拷问端契约 / 变更记录。
```

如果没有 durable artifact，则不能报告为 executed，只能报告为：

```text
subprotocol_loaded_but_not_executed
```

## 6. Runtime report 新增字段

每轮 loop report 必须新增：

```text
subprotocol_activation_plan:
  parent_skill: spec-first-ai-engineering
  subprotocol: references/intent-brainstorm-grill.md
  trigger_reason: <why activated or why not>
  expected_artifact: <clarification/report/contract patch>
  downstream_consumer: <spec/issue/handoff/validator/project patch>

subprotocol_execution_evidence:
  loaded_reference: yes/no
  activated: yes/no
  funnel_layers_completed:
    - goal
    - users_operators
    - boundaries_non_goals
    - assumptions
    - failure_modes
    - options_compared
    - acceptance_criteria
  artifact_path: <path or not_applicable_reason>
  output_consumed_by: <downstream artifact/path or not_consumed_reason>
```

## 7. 对 Superpowers / grill-me / grill-with-docs 的准确用法

它们不是当前 profile 的本地独立 skill，因此不能写：

```text
Superpowers brainstorming executed
/grill-me executed
/grill-with-docs executed
```

正确写法：

```text
external_patterns_referenced:
  - Superpowers brainstorming: design-first / Socratic clarification / approval gate
  - Matt Pocock /grill-me: alignment-before-code questioning
  - Matt Pocock /grill-with-docs: docs-grounded grilling

absorbed_by_subprotocol:
  parent_skill: spec-first-ai-engineering
  subprotocol: references/intent-brainstorm-grill.md
```

如果未来要把它们变成可单独调用的本地能力，才创建 bridge skill；当前不必为了“存在感”创建新 skill。

## 8. Validator 修正方向

Validator 不只检查：

```text
skill_invocation_plan
skill_runtime_matrix
missing_skill_bridge
```

还要检查：

```text
subprotocol_activation_plan
subprotocol_execution_evidence
parent_skill
subprotocol
activated
funnel_layers_completed
artifact_path
output_consumed_by
```

这样才能防止：

```text
子协议存在，但每轮只是读了 / 提到了，没有真正作为需求拷问门运行。
```

## 9. 本轮结论

```text
1. intent-brainstorm-grill 不是空头说明；它作为 spec-first-ai-engineering 的子协议真实存在并可加载。
2. 当前真正的问题是：它没有被稳定激活为运行时 gate。
3. 修正方式不是再审计“有没有”，而是要求每轮报告 subprotocol_activation_plan 和 subprotocol_execution_evidence。
4. 当该子协议没有产出 durable artifact 且没有被下游消费时，只能写 loaded_reference，不得写 executed / used。
```
