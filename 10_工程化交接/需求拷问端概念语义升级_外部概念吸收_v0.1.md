> 分线：10_工程化交接 / 方法轮上游改进  
> 时间：2026-06-14 23:06:55 -0700  
> 状态：v0.1 / 用于修正当前拷问端“只会收敛最高价值问题”的偏差  
> GitHub Issue：#3 / https://github.com/sunshihao001/meme-/issues/3  
> 目的：把需求拷问端从“最高价值任务排序器”升级为“概念语义解析 + 问题空间建模 + 外部相似概念检索 + 证据收敛”的循环。

---

## 1. 用户批评

用户指出：

```text
现在这个拷问端有些问题，就是只会总结出最高价值的来做。
这并不是高明的做法。
需要使用其本身解析概念语义，然后从论坛搜索相关文章或者类似的概念，来升级修改其拷问端本身。
```

本批评成立。

当前项目中的 `demand_theory_evolution_loop.md` 和 `loop_agent_demand_grilling_contract.md` 确实过度强调：

```text
每轮只处理一个最高价值问题
钉住唯一最高价值问题
```

这个规则能防止发散失控，但副作用是：

```text
1. 太快收敛。
2. 容易把拷问端变成任务排序器。
3. 没有先解析概念语义和问题空间。
4. 没有系统性搜索外部相似概念。
5. 没有把外部语义回流到拷问端本身。
```

### 2.1 MQL5 中文 Articles 作为策略/MQL5实现问题的优先来源

当需求拷问端在“概念语义解析 → 关键词形成 → 外部搜索/资料吸收”阶段遇到以下主题时，必须把 `https://www.mql5.com/zh/articles` 纳入优先检索源：

```text
1. MQL5 指标 / EA / 回测 / 优化 / 过拟合 / 数据导出 / 可视化实现。
2. Market Profile、Volume Profile、POC、AVWAP、VWAP、成交量分布、价值区域。
3. 假突破、流动性抓取、深洗、死亡盘、二次启动、重新接受等策略语义的 MQL5 实现或案例。
4. 需要把策略概念落到 MQL5 观察器 MVP、指标需求、回测纪律或代码结构时。
```

最低检索契约：

```text
site:mql5.com/zh/articles <中文关键词>
site:mql5.com/zh/articles <中文关键词> MQL5
site:mql5.com/zh/articles <英文关键词> MQL5
```

证据边界：

```text
MQL5 中文 Articles 可以作为实现方法、指标计算、回测流程、平台机制、案例结构的优先证据；
但不能单独证明当前 FCZ 策略有效，策略有效性仍需样本、反证和回测验证。
```

如果本轮关键词形成后没有使用该来源，循环报告必须说明原因，例如：

```text
not_applicable：本轮是纯 GitHub/CI/方法轮运行态问题，不涉及 MQL5 交易实现或指标语义。
blocked：网络不可用或检索工具不可用。
already_covered：本轮只复核已下载资料，未新增外部搜索。
```

### 2.2 搜索前语义转译层：从 Owner 语义到可检索专业表达

本轮新增的问题不是“要不要搜索外部来源”，而是：

```text
即使知道要搜索外部来源，如果直接拿 Owner 的个人语义、中文口语、项目内部叫法去搜，也可能永远搜不到成熟概念。
```

因此，需求拷问端在外部搜索前必须增加一层：

```text
Search Intent Translation Layer
= 搜索意图转译层
= Owner 语义 → 专业概念族 → 多语言检索式 → 多视角查询 → 证据融合
```

这层解决的是信息检索里的几个成熟问题：

```text
1. vocabulary mismatch / vocabulary gap / term mismatch：用户词和文档词不一致。
2. query reformulation / query rewriting：把原始问题改写成更适合检索的查询。
3. query expansion：加入同义词、上位词、下位词、相关概念和领域术语。
4. query transformation：在 RAG / Agent 检索里，用 rewrite、multi-query、decomposition、step-back、HyDE 等方法改善召回。
5. cross-lingual / multilingual search：中文用户的问题不等于只能搜中文；需要翻译成英文、平台语言或领域通用语。
```

对需求拷问端的修正：

```text
不得把 Owner 原话直接等同于搜索词。
必须先判断：Owner 真正描述的是哪个问题世界、哪个信息检索失败模式、哪个专业概念族。
```

最低输出字段：

```text
owner_surface_terms：Owner 原话里的表层词。
interpreted_information_need：Agent 理解出的真实信息需求。
professional_concept_family：可能对应的专业概念族。
query_transformation_strategy：rewrite / expansion / decomposition / step-back / HyDE / multilingual。
search_query_variants：至少 5 条中英文或领域语言检索式。
query_drift_risk：哪些扩展词可能带偏。
retrieval_failure_mode：本轮是在修复词汇不匹配、过窄查询、过宽查询、语言不匹配，还是多问题混合。
```

本轮用户问题的实际转译示例：

| Owner 表达 | 专业表达 / 可搜概念 |
|---|---|
| “我说的只是我个人的语义认知” | user intent representation / information need articulation |
| “不能我说什么词就搜什么词” | vocabulary mismatch / term mismatch / query-document gap |
| “词语表达不对可以转化表达” | query rewriting / query reformulation |
| “根据概念问题本身知道我讲的是什么意思” | query understanding / semantic parsing / intent detection |
| “不只是中文，要灵活转化” | cross-lingual information retrieval / multilingual query expansion |
| “一个词搜不到，可能语言不对” | multilingual search strategy / translated query variants |
| “语义搜到了但没理解整体表达” | step-back prompting / problem decomposition / sensemaking |
| “论坛里有没有类似成品拼接补全” | pattern mining / reference architecture search / query transformation patterns |

本轮形成的外部检索式：

```text
semantic query expansion search terms formulation information retrieval user intent vocabulary mismatch
query reformulation vocabulary mismatch information retrieval semantic search multilingual query expansion
RAG query transformation query rewriting multi query retrieval user intent expansion
LLM query rewriting retrieval augmented generation multi query retriever HyDE step back prompting
vocabulary mismatch user information need query formulation information retrieval
```

本轮可吸收来源：

```text
LangChain Query Transformations：https://www.langchain.com/blog/query-transformations
Vocabulary mismatch：https://en.wikipedia.org/wiki/Vocabulary_mismatch
Query expansion：https://en.wikipedia.org/wiki/Query_expansion
Query Rewriting and Expansion in RAG：https://learnixo.io/blog/rag-query-rewriting
Which Query Transformation Techniques Actually Help RAG：https://alexchernysh.com/blog/query-transformation-for-rag
```

吸收边界：

```text
这些来源用于改进“搜索前如何表达问题”的方法层；
不能替代具体业务领域证据，例如 MQL5 策略实现仍要查 MQL5 Articles / Forum / CodeBase。
```

## 3. 可吸收概念

### 3.1 Problem Frames Approach

来源：

```text
https://en.wikipedia.org/wiki/Problem_Frames_Approach
```

可吸收点：

```text
需求分析不能只问“软件/Agent 要做什么”，而要先分析问题所在的真实世界。
Problem Frames 强调 user requirements 是关于 application domain 的关系，而不是关于软件界面本身。
```

对拷问端的修正：

```text
在选择最高价值问题之前，必须先画出：
1. problem world：真实世界中发生了什么。
2. solution world：Agent / 文档 / CI / GitHub 能改变什么。
3. shared phenomena：两者之间的接口是什么。
4. required effect：希望在问题世界产生什么变化。
```

不能再直接从聊天请求跳到任务排序。

---

### 3.2 Concept Mapping

来源：

```text
https://libraryguides.mcgill.ca/conceptmapping
```

可吸收点：

```text
Concept mapping 不只是列概念，而是明确概念之间的关系。
概念 + 连接词 = proposition。
好的概念图有层级、连接和 cross-links。
```

对拷问端的修正：

```text
拷问端应该先输出 concept map seed：
- 核心概念
- 相邻概念
- 概念之间的关系词
- 哪些关系是已确认，哪些是 PROPOSED / UNKNOWN
```

尤其适用于策略研究项目中的：

```text
第一控盘区
深洗
死亡盘
POC 回收
AVWAP 重新接受
二次启动
GMGN 数据源
AI 工作流运行态
```

---

### 3.3 Problem Framing

来源：

```text
https://edtechbooks.org/id/problem_framing
```

可吸收点：

```text
设计问题是 ill-structured problem，不存在唯一正确问题框架。
同一个 brief 可以被不同设计者框成不同问题。
Problem framing 是定义、限定、包含/排除、反复重构问题的过程。
```

对拷问端的修正：

```text
“唯一最高价值问题”只能出现在收敛阶段，不能出现在第一步。
前置阶段必须允许多个 problem frames 并列：
1. 这是概念语义问题吗？
2. 这是样本证据问题吗？
3. 这是工程运行问题吗？
4. 这是 owner 决策问题吗？
5. 这是外部知识不足问题吗？
```

---

### 3.4 Product Discovery

来源：

```text
https://herbig.co/product-discovery/
```

可吸收点：

```text
Product Discovery 是 evidence-informed process，用来降低“哪个问题值得解决、为谁解决、什么解法值得做”的不确定性。
它强调 problem space，不应该一开始就进入 solution / ideation。
```

对拷问端的修正：

```text
拷问端应先降低问题空间不确定性，而不是直接做执行切片。
每轮应记录：
1. 当前不确定性是什么。
2. 有哪些一手/二手证据。
3. 需要什么外部材料或论坛讨论。
4. 哪些结论只是 alibi-discovery，即为了证明已选方案而做的伪发现。
```

---

### 3.5 Issue Tree / Hypothesis Tree

来源：

```text
https://mconsultingprep.com/issue-tree
```

可吸收点：

```text
Issue tree 先把问题分解成分支，再基于证据测试假设。
不要跳层，不要没确认上层假设就直接处理下层任务。
```

对拷问端的修正：

```text
“最高价值问题”应该来自 issue tree 的某个被证据支持的分支，而不是 Agent 主观判断。
每轮收敛前必须至少有一个 mini issue tree：
root problem → branches → hypotheses → evidence needed → chosen branch。
```

### 3.6 Forum / Reddit 讨论：AI Requirements Management

来源示例：

```text
Reddit r/businessanalysis：Proof of concept AI requirements management tool
Reddit r/ProductManagement：Template/framework for problem framing
```

可吸收点：

```text
真实用户痛点不是“AI 能不能生成需求”，而是：
1. AI 生成内容不结构化。
2. Jira / ticket 关闭后，需求知识没有长期结构化管理。
3. 需求、验收标准、feature tree、release、bug、feedback 之间缺少映射。
```

对当前项目的提示：

```text
当前项目也有同类问题：
本地 issue、GitHub issue、任务包、spec、handoff、PR、CI、变更记录之间正在逐步连接，但还没有 machine-readable issue tree / concept map / run state。
```

## 4. 拷问端 v0.2 补丁：从“单点收敛”改为“双阶段循环”

### 旧逻辑

```text
读取上下文
→ 选择唯一最高价值问题
→ 写补丁 / issue / validator
→ 验证
```

问题：太快收敛。

### 新逻辑

```text
Phase A：语义发散 / 问题空间建模
1. 概念拆解：这个词/目标到底是什么意思。
2. Problem World 建模：真实世界发生了什么。
3. Concept Map Seed：核心概念与关系。
4. Issue Tree：至少列出 3 个可能问题框架。
5. 外部相似概念检索：论坛 / 文档 / 论文 / 产品实践。
6. 吸收判断：哪些外部概念能改进当前拷问端。

Phase B：证据收敛 / 执行切片
7. 只在完成 Phase A 后，选择一个当前切片。
8. 该切片不是“最高价值问题”本身，而是“当前证据下最该验证/写入/执行的分支”。
9. 写入理论、schema、issue、validator 或 owner decision。
10. 验证并记录哪些问题仍处于开放状态。
```

## 5. 新增输出类型

现有输出类型不足以表达“概念语义升级”和“外部吸收”。建议新增：

```text
TYPE_G_CONCEPT_SEMANTIC_MAP：概念语义图 / problem world map。
TYPE_H_EXTERNAL_CONCEPT_ABSORPTION：外部相似概念检索与吸收。
TYPE_I_LOOP_SELF_PATCH：对拷问端本身的规则补丁。
```

---

## 6. 新规则

```text
1. 如果用户批评拷问端本身，不得直接选择下一任务；必须先执行 TYPE_I_LOOP_SELF_PATCH。
2. 如果问题涉及概念不清，不得跳到最高价值问题；必须先执行 TYPE_G_CONCEPT_SEMANTIC_MAP。
3. 如果项目理论需要升级，不得只靠内部推理；必须至少做一次 TYPE_H_EXTERNAL_CONCEPT_ABSORPTION。
4. “最高价值问题”改名为“当前收敛切片”，避免把排序误认为理解。
5. 每次收敛前必须留下未选分支，防止假收敛。
```

---

## 7. 本版结论

当前拷问端最大问题不是“不够会选重点”，而是：

```text
它过早把理解问题变成了执行排序。
```

升级方向是：

```text
先建模问题世界，再建模概念关系，再检索外部相似概念，最后才选择当前收敛切片。
```

这会让拷问端从“任务推进器”升级为“理论进化器”。
