## 1. 用户批评

用户指出：

```text
现在这个拷问端有些问题，就是只会总结出最高价值的来做。
这并不是高明的做法。
需要使用其本身解析概念语义，然后从论坛搜索相关文章或者类似的概念，来升级修改其拷问端本身。
```

本批评成立。

当前项目中的 `demand_theory_evolution_loop.md` 和 `loop_agent_demand_grilling_contract.md` 确实过度强调：

```text
每轮只处理一个最高价值问题
钉住唯一最高价值问题
```

这个规则能防止发散失控，但副作用是：

```text
1. 太快收敛。
2. 容易把拷问端变成任务排序器。
3. 没有先解析概念语义和问题空间。
4. 没有系统性搜索外部相似概念。
5. 没有把外部语义回流到拷问端本身。
```

## 3. 可吸收概念

### 3.1 Problem Frames Approach

来源：

```text
https://en.wikipedia.org/wiki/Problem_Frames_Approach
```

可吸收点：

```text
需求分析不能只问“软件/Agent 要做什么”，而要先分析问题所在的真实世界。
Problem Frames 强调 user requirements 是关于 application domain 的关系，而不是关于软件界面本身。
```

对拷问端的修正：

```text
在选择最高价值问题之前，必须先画出：
1. problem world：真实世界中发生了什么。
2. solution world：Agent / 文档 / CI / GitHub 能改变什么。
3. shared phenomena：两者之间的接口是什么。
4. required effect：希望在问题世界产生什么变化。
```

不能再直接从聊天请求跳到任务排序。

---

### 3.2 Concept Mapping

来源：

```text
https://libraryguides.mcgill.ca/conceptmapping
```

可吸收点：

```text
Concept mapping 不只是列概念，而是明确概念之间的关系。
概念 + 连接词 = proposition。
好的概念图有层级、连接和 cross-links。
```

对拷问端的修正：

```text
拷问端应该先输出 concept map seed：
- 核心概念
- 相邻概念
- 概念之间的关系词
- 哪些关系是已确认，哪些是 PROPOSED / UNKNOWN
```

尤其适用于策略研究项目中的：

```text
第一控盘区
深洗
死亡盘
POC 回收
AVWAP 重新接受
二次启动
GMGN 数据源
AI 工作流运行态
```

---

### 3.3 Problem Framing

来源：

```text
https://edtechbooks.org/id/problem_framing
```

可吸收点：

```text
设计问题是 ill-structured problem，不存在唯一正确问题框架。
同一个 brief 可以被不同设计者框成不同问题。
Problem framing 是定义、限定、包含/排除、反复重构问题的过程。
```

对拷问端的修正：

```text
“唯一最高价值问题”只能出现在收敛阶段，不能出现在第一步。
前置阶段必须允许多个 problem frames 并列：
1. 这是概念语义问题吗？
2. 这是样本证据问题吗？
3. 这是工程运行问题吗？
4. 这是 owner 决策问题吗？
5. 这是外部知识不足问题吗？
```

---

### 3.4 Product Discovery

来源：

```text
https://herbig.co/product-discovery/
```

可吸收点：

```text
Product Discovery 是 evidence-informed process，用来降低“哪个问题值得解决、为谁解决、什么解法值得做”的不确定性。
它强调 problem space，不应该一开始就进入 solution / ideation。
```

对拷问端的修正：

```text
拷问端应先降低问题空间不确定性，而不是直接做执行切片。
每轮应记录：
1. 当前不确定性是什么。
2. 有哪些一手/二手证据。
3. 需要什么外部材料或论坛讨论。
4. 哪些结论只是 alibi-discovery，即为了证明已选方案而做的伪发现。
```

---

### 3.5 Issue Tree / Hypothesis Tree

来源：

```text
https://mconsultingprep.com/issue-tree
```

可吸收点：

```text
Issue tree 先把问题分解成分支，再基于证据测试假设。
不要跳层，不要没确认上层假设就直接处理下层任务。
```

对拷问端的修正：

```text
“最高价值问题”应该来自 issue tree 的某个被证据支持的分支，而不是 Agent 主观判断。
每轮收敛前必须至少有一个 mini issue tree：
root problem → branches → hypotheses → evidence needed → chosen branch。
```

---

### 3.6 Forum / Reddit 讨论：AI Requirements Management

来源示例：

```text
Reddit r/businessanalysis：Proof of concept AI requirements management tool
Reddit r/ProductManagement：Template/framework for problem framing
```

可吸收点：

```text
真实用户痛点不是“AI 能不能生成需求”，而是：
1. AI 生成内容不结构化。
2. Jira / ticket 关闭后，需求知识没有长期结构化管理。
3. 需求、验收标准、feature tree、release、bug、feedback 之间缺少映射。
```

对当前项目的提示：

```text
当前项目也有同类问题：
本地 issue、GitHub issue、任务包、spec、handoff、PR、CI、变更记录之间正在逐步连接，但还没有 machine-readable issue tree / concept map / run state。
```

## 4. 拷问端 v0.2 补丁：从“单点收敛”改为“双阶段循环”

### 旧逻辑

```text
读取上下文
→ 选择唯一最高价值问题
→ 写补丁 / issue / validator
→ 验证
```

问题：太快收敛。

### 新逻辑

```text
Phase A：语义发散 / 问题空间建模
1. 概念拆解：这个词/目标到底是什么意思。
2. Problem World 建模：真实世界发生了什么。
3. Concept Map Seed：核心概念与关系。
4. Issue Tree：至少列出 3 个可能问题框架。
5. 外部相似概念检索：论坛 / 文档 / 论文 / 产品实践。
6. 吸收判断：哪些外部概念能改进当前拷问端。

Phase B：证据收敛 / 执行切片
7. 只在完成 Phase A 后，选择一个当前切片。
8. 该切片不是“最高价值问题”本身，而是“当前证据下最该验证/写入/执行的分支”。
9. 写入理论、schema、issue、validator 或 owner decision。
10. 验证并记录哪些问题仍处于开放状态。
```

## 5. 新增输出类型

现有输出类型不足以表达“概念语义升级”和“外部吸收”。建议新增：

```text
TYPE_G_CONCEPT_SEMANTIC_MAP：概念语义图 / problem world map。
TYPE_H_EXTERNAL_CONCEPT_ABSORPTION：外部相似概念检索与吸收。
TYPE_I_LOOP_SELF_PATCH：对拷问端本身的规则补丁。
```

---

## 6. 新规则

```text
1. 如果用户批评拷问端本身，不得直接选择下一任务；必须先执行 TYPE_I_LOOP_SELF_PATCH。
2. 如果问题涉及概念不清，不得跳到最高价值问题；必须先执行 TYPE_G_CONCEPT_SEMANTIC_MAP。
3. 如果项目理论需要升级，不得只靠内部推理；必须至少做一次 TYPE_H_EXTERNAL_CONCEPT_ABSORPTION。
4. “最高价值问题”改名为“当前收敛切片”，避免把排序误认为理解。
5. 每次收敛前必须留下未选分支，防止假收敛。
```

---

## 7. 本版结论

当前拷问端最大问题不是“不够会选重点”，而是：

```text
它过早把理解问题变成了执行排序。
```

升级方向是：

```text
先建模问题世界，再建模概念关系，再检索外部相似概念，最后才选择当前收敛切片。
```

这会让拷问端从“任务推进器”升级为“理论进化器”。
