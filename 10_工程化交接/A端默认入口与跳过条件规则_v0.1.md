# A端默认入口与跳过条件规则 v0.1

> 分线：10_工程化交接 / A端入口门禁
> 状态：v0.1 / 默认入口规则草案
> 目标：解决“工作流有了，但用户输入没有默认进入 A 端”的问题。
> 原则：新输入默认先过 dbs 路由，再由 A 端做最小需求澄清；只有明确续接、执行、恢复、验证时才允许跳过 A。

---

## 1. 核心问题

当前工作流已经有：

```text
A/B/C 前置研究循环
A/B/C 正式合同提速版
dbs 路由默认入口
workflow state / queue
```

但实际运行中仍可能出现：

```text
用户输入
→ 助手直接回答
→ 没有先进入 A 端澄清/概念解析/路由判断
```

根因是：

```text
工作流规则存在，但默认入口门禁没有硬化。
```

---

## 2. 默认入口总规则

### 2.1 所有新输入默认路径

```text
用户新输入
→ dbs 路由入口
→ A 端最小澄清
→ 端口路由判断
→ B / C / D / E / F
```

### 2.2 不允许默认直接进入的端口

除非满足跳过条件，否则新输入不能直接进入：

```text
B：资料搜索 / Source Pack
C：长理论生成
D：落库执行
E：验证审查
F：Owner 决策记录
```

---

## 3. A 端最小动作

新输入进入 A 端后，不一定要长篇拷问，但至少要完成以下 4 个动作中的 3 个：

```text
1. 识别用户原话里的核心想法。
2. 判断问题类型：澄清 / 研究 / 搜索 / 理论 / 执行 / 验证 / 决策。
3. 标出当前缺口：概念不清、资料不够、理论不够、执行条件不足。
4. 给出下一步端口路由：A / B / C / D / E / F。
```

如果输入很清楚，A 端可以用 fast_path 输出一小段路由判断，不需要长文。

---

## 4. 默认进入 A 端的情况

以下情况必须先进入 A 端：

```text
1. 新想法。
2. 新问题。
3. 新的工作流改造建议。
4. 多个概念混在一起。
5. 用户表达“不确定”“不知道哪里有问题”。
6. 用户问“为什么没用起来”“哪里慢”“怎么结合”。
7. 外部技能、外部仓库、外部方法要接入当前工作流。
8. 任何可能改变 A/B/C/D/E/F 基线的输入。
```

---

## 5. 自动提问与自动判断规则

A 端进入后，默认不是一次性给结论，而是自动进入循环：

```text
自动提问 → 自动判断 → 自动推理 → 解决后继续自动提问
```

直到满足以下条件之一才停：

```text
1. 初步想法已经收束到可以执行的程度。
2. 继续追问只会改细节，不会改路线。
3. 你明确说“满意”或“先停在这里”。
```

### 5.1 A 端每轮默认动作

1. 先判断当前输入是否足够形成问题。
2. 不够就自动追问最关键的 1-3 个问题。
3. 如果已经够了，就自动调相关技能获取更多信息。
4. 根据新信息自动判断是否进入 B/C。
5. 如果还不够成熟，就继续下一轮自动提问。

### 5.2 A 端默认目标

```text
把初步想法推进到你满意为止。
```

---

## 6. 可以跳过 A 端的条件

只有以下情况可以跳过 A 端：

```text
1. 用户明确说：继续上次。
2. 用户明确说：直接执行。
3. 用户明确说：上传 / 提交 / 推送 / 保存到仓库。
4. 当前已有明确任务包、文件路径、验收标准。
5. 当前是在验证一个已经完成的输出。
6. 当前是在恢复已有状态或读取已有记录。
```

即便跳过 A，也要保留一句状态判断：

```text
本轮按续接/执行/验证处理，跳过 A 端深度澄清。
```

---

## 6. dbs 路由与 A 端的衔接

### 6.1 dbs 路由先做什么

```text
dbs 路由先判断：这次输入需要哪类能力？
```

候选能力包：

```text
- 问题定义包：dbs-good-question / dbs-deconstruct / dbs-goal
- 决策判断包：dbs-decision / dbs-slowisfast
- 知识研究包：dbs-learning / dbs-content-system
- 状态管理包：dbs-save / dbs-restore / dbs-report
```

### 6.2 A 端再做什么

```text
A 端把 dbs 路由结果转成端口动作：
- 继续澄清
- 交给 B 搜索/压缩
- 交给 C 理论生成
- 交给 D 落库
- 交给 E 验证
- 交给 F 决策
```

---

## 7. fast_path 输出格式

为了避免慢，A 端默认用 fast_path。

### 7.1 最小输出模板

```text
【dbs 路由判断】
问题类型：____
推荐能力包：____

【A 端最小澄清】
当前核心想法：____
当前缺口：____
下一步端口：____

【下一步】
直接执行：____
需要用户判断：____
```

### 7.2 何时进入 deep_path

只有满足任一条件才进入 deep_path：

```text
1. 用户明确要求深入。
2. fast_path 两轮后仍不清楚。
3. 涉及正式 baseline 修改。
4. 涉及外部技能大量接入。
5. 涉及 D/E/F 执行验证链路。
```

---

## 8. 防止再次失效的硬规则

```text
1. 新输入不得直接进入长理论。
2. 新输入不得直接进入 B 端搜索，除非 A 端已有搜索任务单。
3. 新输入不得直接调用多个 dbskill 技能，必须先由 dbs 路由选能力包。
4. A 端默认只做最小澄清，不得把每次输入都变成大课题。
5. 如果助手没有说明本轮为什么跳过 A，即视为入口规则未执行。
```

---

## 9. 验收标准

这份规则生效后，每次新输入应该能看到：

```text
1. 是否经过 dbs 路由。
2. 是否经过 A 端最小澄清。
3. 是否说明了跳过 A 的理由。
4. 是否明确下一步端口。
5. 是否避免不必要的长理论和无边界搜索。
```

---

## 10. 本版结论

```text
dbs 路由是默认前置门。
A 端是默认意图澄清门。
新输入默认不能绕过这两层。
跳过 A 必须有明确理由。
```
