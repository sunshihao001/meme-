## 2. C端研究视角下的课题定义

### 2.1 课题真正要研究什么

不是单篇文档，不是单个 skill，而是：

```text
多端口 AI 工作流的知识进化机制。
```

这里的“进化”包含三层：

```text
1. 问题进化：把粗糙想法升级成专业问题。
2. 方案进化：把研究结果升级成执行版方案。
3. 技能进化：把有效做法回收到分层技能库和状态链。
```

---

### 2.2 核心结构假设

本课题的理论骨架是双循环：

```text
第一循环：A/B/C 知识研究循环
第二循环：C-D-E/F 执行验证循环
```

A/B/C 不是前置准备，而是一个完整研究闭环：

- A：定义问题、升级自然语言、控制方向
- B：收集资料、压缩上下文、形成 Source Pack
- C：生成理论、候选结构、研究版方案

如果 Owner 不满意，应该返回：

- 回 A：方向不对
- 回 B：证据不够
- 回 C：理论浅 / 结构乱 / 方案不专业

只有 Owner 对 C 端研究版输出满意，才进入 C-execution。

## 3. C端的双状态模型

### 3.1 C-research

```text
C-research = 研究版理论生成
```

职责：

```text
1. 总结 A/B 的输入。
2. 提炼结构性规律。
3. 生成多个候选框架，而不是单一答案。
4. 标出假设、风险、边界、待验证点。
5. 保持可被 Owner 调整。
```

### 3.2 C-execution

```text
C-execution = 执行版方案生成
```

职责：

```text
1. 把研究结论收敛成可落库的任务包。
2. 明确 D/E/F 的分工与边界。
3. 明确验证标准和回退条件。
4. 为后续 repo landing / CI / owner gate 做准备。
```

### 3.3 C 端的桥接作用

C 不是单一节点，而是桥：

```text
A/B/C 知识研究循环的出口
C/D/E/F 执行验证循环的入口
```

所以 C 端要同时具备：

```text
研究能力 + 收敛能力 + 结构转换能力
```

## 4. Owner 表达引导层的理论位置

今天补充的关键修正，不是附属功能，而是 A端理论的一部分。

### 4.1 为什么必须有表达引导

因为 Owner 不是专业研究员或工程师，直接自由输入时很容易：

```text
1. 说偏题。
2. 说太散。
3. 想法混在一起。
4. 分不清研究、执行、决策。
5. 不知道下一步该怎么说。
```

所以系统不能只是“接收输入”，还必须：

```text
主动引导下一句表达。
```

### 4.2 表达引导层的职责

A端每次输出后，应该附带：

```text
1. 当前你只需要判断什么。
2. 如果满意，你可以怎么回复。
3. 如果不满意，你应该按哪几类说。
4. 哪些内容暂时不要展开。
5. 推荐下一句输入模板。
```

### 4.3 偏题处理理论

A端要把偏题分成四类：

```text
1. 模糊但相关 → 帮你重写。
2. 新主题但可能相关 → 标记为旁支候选。
3. 明显偏离当前课题 → 先拉回当前循环。
4. 不满意但表达不清 → 转成回 A / 回 B / 回 C 的反馈。
```

这个层的价值是：

```text
不是逼 Owner 变专业，
而是让系统承担“把非专业表达转换成可研究输入”的责任。
```

## 5. 技能进化机制的理论定义

本课题研究的技能进化，不是“多装几个 skill”，而是：

```text
把有效工作方式结构化成分层、状态、模板、桥接和边界。
```

### 5.1 L1-L6 分层

- **L1**：端口主技能，定义端口身份
- **L2**：通用辅助技能，跨端复用
- **L3**：外部桥接技能，吸收外部方法但不冒充主技能
- **L4**：领域专用技能，绑定当前业务域
- **L5**：模板/交付技能，统一输出格式
- **L6**：状态/边界标记，控制是否执行、引用、弃用、暂缓

### 5.2 技能进化的判据

一个能力是否该纳入技能库，不看“新不新”，而看：

```text
1. 是否稳定复现。
2. 是否可被其他端复用。
3. 是否能形成模板或边界。
4. 是否会污染主线。
5. 是否能通过样例验证。
```

### 5.3 外部 skill 的定位

外部 skill 不是默认要接入，而是先判断：

```text
ADOPT / BRIDGE / MERGE / REJECT / PATTERN_ONLY
```

这说明：

```text
技能进化的本质是治理，不是堆积。
```

## 6. dbskill 与状态链在课题中的角色

### 6.1 dbskill 的研究定位

dbskill 在这条课题里不是主角，而是：

```text
状态链辅助层 / 过程记录层 / 上下文恢复层
```

### 6.2 可接受的用途

```text
dbs-decision：记录 F端决策事件
dbs-save：保存 repo 内 workflow snapshot
dbs-restore：从 repo 内 state/queue/snapshot 恢复
dbs-report：辅助 E/F 汇总报告
```

### 6.3 不可越过的边界

```text
1. 不能替代 git。
2. 不能替代索引。
3. 不能替代变更记录。
4. 不能替代 E端验证。
5. 不能替代 F端 Owner 决策。
6. 不能成为 repo 外独立真源。
```

理论上，这表示状态链的角色是：

```text
减少断裂，但不定义真相。
```

## 7. 对 workflow 的研究性解释

当前 workflow 更像一个“研究-执行双相系统”，而不是普通流水线。

### 7.1 研究相

```text
A/B/C = 研究相
```

特点：

```text
- 可反复调整
- 可多方案比较
- 允许不满意回退
- 允许 Owner 修正方向
```

### 7.2 执行相

```text
C/D/E/F = 执行相
```

特点：

```text
- 需要收敛
- 需要落库
- 需要验证
- 需要决策
```

### 7.3 课题的关键理论点

```text
不要让执行逻辑抢在研究成熟之前启动。
```

这是避免“边想边落库”的核心方法论。
