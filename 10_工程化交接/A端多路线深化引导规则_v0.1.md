# A端多路线深化引导规则 v0.1

> 分线：10_工程化交接 / A端入口门禁
> 状态：v0.1 / A端 Owner 引导升级规则
> 目标：解决 A 端面对复杂想法时，只给“认可/不认可”或单一路径建议，未充分利用 dbs 路由与多技能组合的问题。
> 原则：复杂想法先给多条可深化路线，再由 Owner 选择方向；A 端控制组合，dbs 技能提供视角。

---

## 1. 解决的问题

当前已建立：

```text
dbs 路由默认入口
A端默认入口与跳过条件
A-F 强制 handoff
B/C/D/E 只接 handoff 纪律
```

但仍存在一个运行缺口：

```text
用户提出复杂想法后，A 端容易直接给一个下一步建议，或者只问 Owner 是否认可。
```

这会导致：

```text
1. dbs 路由没有真正发挥“多路线候选”能力。
2. Owner 只能认可/不认可，不能选择深化方向。
3. 复杂想法被过早收束成单一路径。
4. 多技能组合没有被灵活调用。
```

---

## 2. 总规则

当用户提出复杂想法、工作流改造、外部技能组合、多端口问题时，A 端必须先输出多路线深化引导，而不是直接进入 D 落库或 B/C 执行。

```text
复杂想法
→ dbs 路由识别候选能力
→ A 端列出 3-5 条深化路线
→ Owner 选择路线
→ A 端生成对应 handoff
→ B/C/D/E/F 执行
```

一句话：

```text
先给路线，再要选择；先让 Owner 选方向，再启动下游端口。
```

---

## 3. 触发条件

出现以下任意 1 条，A 端应启动多路线深化引导：

```text
1. 用户提出一个新工作流想法。
2. 用户说“要深入理解 / 升华 / 不要太死板 / 灵活组合”。
3. 用户提出多个 dbs 技能或多个端口可能同时有关。
4. 用户纠正“为什么直接给下一步建议”。
5. 用户想比较理论想法、实际输出和工作流执行差距。
6. 用户不确定问题应该走 A/B/C/D/E/F 哪一端。
```

---

## 4. 标准输出结构

A 端多路线深化输出必须包含：

```text
1. 当前想法核心
2. 可深化路线 3-5 条
3. 每条路线对应的 dbs 技能 / 技能组合
4. 每条路线适合解决什么问题
5. Owner 可以怎么回复
6. 暂时不要展开什么
7. 推荐默认路线（可选，但不能替 Owner 决策）
```

---

## 5. 常见路线模板

### 路线 A：对比验证路线

对应技能：

```text
dbs-benchmark
```

适合问题：

```text
理论目标和实际输出是否一致？
今天提出的想法是否真的落地？
规则写了，但执行时有没有用起来？
```

Owner 可回复：

```text
走对比验证路线，先看理论目标和实际输出差距。
```

---

### 路线 B：输出质量诊断路线

对应技能：

```text
dbs-content
```

适合问题：

```text
今天输出是否过多？
结构是否清楚？
是否能指导下一步？
Owner 是否看得懂？
```

Owner 可回复：

```text
走输出诊断路线，先检测今天产出的内容质量和可执行性。
```

---

### 路线 C：好问题重构路线

对应技能：

```text
dbs-good-question
```

适合问题：

```text
这个想法怎么表达，才能让 AI 生成更专业、更可验证的输出？
如何把模糊想法变成 Agent 可推理的问题说明书？
```

Owner 可回复：

```text
走好问题路线，把我的想法改写成 AI 可推理、可验证的问题说明书。
```

---

### 路线 D：组合诊断链路线

对应技能组合：

```text
dbs-benchmark → dbs-content → dbs-good-question
```

适合问题：

```text
我不是要单点诊断，而是要形成每天复盘用的组合链。
先对比理论与实际，再检测输出质量，最后改写成下一轮好问题。
```

Owner 可回复：

```text
走组合诊断链路线，把 benchmark → content → good-question 设计成每天复盘链。
```

---

### 路线 E：A端治理路线

对应技能组合：

```text
dbs-deconstruct → dbs-decision
```

适合问题：

```text
这个组合链到底应该放在 A 端、E 端之后，还是独立复盘链？
什么时候触发？什么时候停止？是否会拖慢工作流？
```

Owner 可回复：

```text
走 A端治理路线，先判断多技能组合应该放在工作流哪个位置、何时触发、何时停止。
```

---

## 6. 禁止事项

```text
1. 不允许复杂想法一上来直接 D 落库。
2. 不允许只给“认可/不认可”二选一。
3. 不允许把推荐路线当成 Owner 决策。
4. 不允许每次机械调用所有 dbs 技能。
5. 不允许路线选择后绕过 A 端 handoff。
```

## 8. 与 Handoff 的关系

Owner 选定路线后，A 端再生成 handoff。

```text
Owner 选路线
→ A 端生成 handoff
→ B/C/D/E/F 执行
```

如果 Owner 只是说“可以”，但前文存在多个路线，A 端必须确认“可以”指哪条路线，或按上下文默认路线说明理由。

---

## 9. 本版结论

```text
A 端不是只负责收束，还负责给复杂想法提供多条可深化路线。
dbs 路由不是只选一个技能，也可以生成单技能路线、组合技能路线和治理路线候选。
```
