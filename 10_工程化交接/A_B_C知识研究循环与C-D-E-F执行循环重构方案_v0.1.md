# A/B/C知识研究循环与C-D-E/F执行循环重构方案 v0.1

> 分线：10_工程化交接 / 多端口工作流重构  
> 输入修正：A+B+C 应先构成完整知识研究循环；不满意可随时调整；满意后再从 C 端开始进入后续执行端口。  
> 状态：v0.1 / 结构修正方案  
> 目的：纠正此前把 A/B/C/D/E/F 过早串成单一路径的偏差，把前半段定义为可迭代知识研究循环，把后半段定义为执行、落库、验证、决策循环。

---

## 1. 用户修正

用户指出：

```text
其实应该 a+b+c 做为一个完整的知识研究循环，
不满意可以随时调整，
然后再从 c 端开始到之后端口做执行一系列的问题。
```

这个修正成立。

之前的线性理解是：

```text
A 需求 → B 证据 → C 方案 → D 落库 → E 验证 → F 决策
```

但更准确的结构应该是双循环：

```text
第一段：A/B/C 知识研究循环
第二段：C/D/E/F 执行验证循环
```

---

## 2. 新总结构

```text
A 端：问题定义 / 目标控制 / 方向调整
  ↕
B 端：证据收集 / source pack / 资料压缩
  ↕
C 端：理论生成 / 框架设计 / 方案草案
  ↺
如果不满意：回 A 或 B 或 C 继续调整
如果满意：从 C 端输出执行版方案，进入 D/E/F
  ↓
D 端：落库 / 执行 / repo landing / commit
  ↓
E 端：验证 / 审查 / CI / 反证
  ↓
F 端：Owner 决策 / 授权 / 阶段门
  ↓
回 A：记录状态，决定下一轮研究或执行
```

---

## 3. 第一循环：A/B/C 知识研究循环

### 3.1 定位

A/B/C 不是一次性前置步骤，而是一个完整研究闭环。

```text
A = 研究问题与判断门
B = 证据与资料门
C = 理论与方案门
```

这三个端口共同回答：

```text
我们到底在研究什么？
资料是否够？
框架是否合理？
方案是否让 Owner 满意？
是否需要换方向、补资料、重写理论？
```

---

### 3.2 A端职责

A端在知识研究循环里负责：

```text
1. 把用户自然语言升级为专业问题。
2. 判断当前问题属于概念、证据、理论、执行还是决策。
3. 给 B 端下 source pack 任务。
4. 给 C 端下理论生成任务。
5. 收到 B/C 输出后判断是否满意、是否偏题、是否需要改方向。
6. 保留 Owner 不满意时随时调整路线的权利。
```

A端不是只做一次开头需求澄清，而是整个研究循环的控制面。

---

### 3.3 B端职责

B端在知识研究循环里负责：

```text
1. 根据 A 端问题收集资料。
2. 先返回 Search Strategy Brief，而不是直接乱搜。
3. 形成 Source Pack。
4. 标记证据强度、来源可信度、可吸收点、不可照搬点。
5. 不直接替 C 端生成完整理论。
```

B端输出不是最终答案，而是 C端可用的证据燃料。

---

### 3.4 C端职责

C端在知识研究循环里负责：

```text
1. 基于 A 的问题框架和 B 的证据包生成理论/方案。
2. 输出多个候选结构，而不是只给一个结论。
3. 明确假设、风险、可执行路径、待验证点。
4. 接受 A/Owner 的不满意反馈后重写、收敛或拆分。
5. 当方案成熟后，生成执行版 C 输出，交给 D/E/F。
```

C端既是研究循环的方案生成端，也是执行循环的入口端。

---

## 4. A/B/C 循环的调整机制

不满意时，不应该强行进入 D 端。

应按不满意类型回退：

```text
1. 问题不对 / 方向不对 → 回 A
2. 资料不够 / 证据弱 → 回 B
3. 理论浅 / 结构乱 / 方案不专业 → 回 C
4. 输出太大 / 不可执行 → C 拆分后再判断
5. Owner 觉得还不是想要的 → A 重新解释用户真实意图
```

### 4.1 A/B/C 收束停机规则

```text
当目标、边界、方案、验证四项中至少三项达到可执行级别，
且继续追问只会影响细节，不会改变路线时，停止 A/B/C 研究循环。
```

补充门槛：

```text
- 连续 2 轮没有新增有效信息时停止
- 最多循环 2~3 轮后强制复盘是否转端
- 停机后转入执行版方案、spec 或 issue，而不是继续在研究段空转
```

A/B/C 循环的 stop condition：

```text
Owner 对 C 端研究版输出基本满意，
并同意将其中一版转为执行版方案。
```

---

## 5. 第二循环：C/D/E/F 执行验证循环

进入执行段的前提不是“C 写完了”，而是：

```text
C 已经产出 Owner 可接受的执行版方案。
```

执行段流程：

```text
C 执行版方案 / spec / task package
  ↓
D 落库、实现、索引、变更记录、commit/PR
  ↓
E 验证、审查、CI、反证、风险检查
  ↓
F Owner 决策：接受、返修、扩大、停止
  ↓
必要时回 C 重写执行方案，或回 A/B/C 重启研究循环
```

---

## 6. C端的双重身份

C端需要拆成两个状态：

```text
C-research：研究版理论生成
C-execution：执行版方案生成
```

两者区别：

```text
C-research：可以探索、比较、推演、发散。
C-execution：必须收敛、可落库、可验证、可交给 D/E/F。
```

因此 C 端不是简单的一站，而是桥：

```text
A/B/C 知识研究循环的出口
C/D/E/F 执行验证循环的入口
```

## 7. 对 workflow_loop 的修正

原状态机应从单线性：

```text
INTAKE → SOURCE_PACK → DESIGN → LANDING → VERIFY → OWNER_GATE
```

修正为双循环：

```text
KNOWLEDGE_RESEARCH_LOOP:
  A_INTAKE_AND_FRAME
  B_SOURCE_PACK
  C_RESEARCH_DRAFT
  A_OWNER_FIT_CHECK
  REVISE_TO_A_OR_B_OR_C
  C_EXECUTION_PACKAGE_READY

EXECUTION_LOOP:
  C_EXECUTION_PACKAGE
  D_REPO_LANDING
  E_VERIFICATION
  F_OWNER_DECISION
  RETURN_TO_A_OR_C
```

## 9. 更新后的运行原则

以后遇到研究型、方法型、体系型问题时，默认先走：

```text
A/B/C 知识研究循环
```

只有当 C 端输出已经成熟，并且 Owner 基本满意时，才进入：

```text
C/D/E/F 执行验证循环
```

一句话：

```text
先把问题研究对，再把方案执行对。
```

---

## 10. 本版结论

本次修正将多端口工作流从：

```text
A → B → C → D → E → F 单线流
```

升级为：

```text
(A ↔ B ↔ C) 知识研究循环
→ C 执行版输出
→ (C → D → E → F) 执行验证循环
→ 回 A/C 继续迭代
```

这更符合用户真实目标：

```text
用户可以随时不满意、随时调整研究方向；
系统不急着执行，而是先把知识框架、证据和理论打磨好；
满意后再让 C 端生成执行包，交给 D/E/F 做落库、验证和决策。
```
