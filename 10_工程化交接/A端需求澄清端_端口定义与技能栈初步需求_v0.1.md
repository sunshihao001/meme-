## 1. 为什么先做这个

用户当前的核心判断是：

```text
现在主要要先把整体工作流各个端口分清楚职责，当前职责有些不明确、不标准；
并且要补全内部能力、接入外部技能作为处理工具；
只有需求澄清端和概念解析做好了，表达才会充实、专业，后续端口才能专业化。
```

本文件就是这个判断的第一份正式需求文档。

它要解决的不是“写更多方法名”，而是：

```text
1. 每个端口到底负责什么。
2. 哪些职责绝不能混。
3. 外部技能应该怎样作为处理工具接入。
4. A 端如何把用户自然语言转成专业可执行概念。
5. 先交付 A+B，再决定 C/D/E/F 的细化方式。
```

## 2. 当前问题定义

### 2.1 现象

当前多端口问题的主要症状：

```text
- A 端职责过多，容易同时做需求、概念、路由、甚至一部分理论。
- B/C/D/E 之间边界不够硬，导致重复劳动或互相偷工。
- 端口名称有了，但“核心合同”没锁死。
- 外部技能只是列出来，没有变成可调用的工具栈。
- 需求表达还停留在表层语义，专业概念解析不够深。
```

### 2.2 根因

根因不是模型不够强，而是：

```text
1. 端口是角色，但没有单一职责合同。
2. 技能是工具，但没有按任务类型组装。
3. 需求澄清不足，导致后续端口拿到的是模糊输入。
4. 只有“最高价值问题排序”，没有“概念语义解析 + 问题空间建模”。
```

### 2.3 当前最重要的矫正

```text
先把 A 端做成语义闸门，再把 B 端做成证据闸门。
```

也就是：

```text
A 负责“这件事到底是什么、为什么、边界在哪、下一步路由去哪”。
B 负责“这件事基于哪些证据、哪些资料、哪些上下文”。
```

只有 A/B 做干净，C/D/E/F 才能专业。

## 3. 术语定义

### 3.1 端口（Port）

端口不是一个单 skill，也不是一个大而全的助手。

端口是：

```text
一个非重叠的职责合同 + 一个主要产物 + 一个上游输入 + 一个下游交接。
```

### 3.2 技能（Skill）

技能是可调用能力模块，负责一个具体处理动作，例如：

```text
- 需求拷问
- 概念拆解
- 资料压缩
- 理论生成
- 代码/文件落库
- 验证审查
```

### 3.3 技能栈（Skill Stack）

技能栈不是“装得越多越好”，而是：

```text
主 skill + 辅助 skills + 外部技能桥接
```

其中：

- 主 skill 定身份。
- 辅助 skills 提供能力补位。
- 外部技能桥接负责把外部方法论变成可复用工具。

### 3.4 外部技能（External Skill）

这里的“外部技能”不是泛指整套框架，而是：

```text
单个可被引入、组合、调用、审计的能力模块。
```

例如：

```text
- 需求拷问里的概念拆解 skill
- 资料压缩里的 source pack skill
- 理论生成里的深度 Codex skill
- 验证审查里的 checker skill
```

### 3.5 概念解析（Concept Parsing）

概念解析不是简单改写句子，而是把用户表层表达转成专业概念族：

```text
用户话术 → 问题世界 → 专业概念 → 检索词 → 可执行切片 → 可验证产物
```

### 3.6 激活、执行、消费

一个技能或子协议不能只“存在”，还要区分：

```text
loaded / activated / executed / output_consumed
```

否则就会出现：

```text
“说用了”，但其实只是提到了。
```

## 4. 用户确认后的方向

基于用户最近的明确回答，本次需求澄清已经得到这些方向：

```text
1. A 端目前最容易过载，需要先减负和定界。
2. 目标是方便接入外部技能。
3. 外部技能指的是“单个技能模块”，不是整套框架一口吞。
4. 需要更专业、AI 更容易理解的表达。
5. 先做 A+B 两端。
6. 这是新话题，重点是端口定义和技能栈。
```

所以本文件的初步结论是：

```text
A 端先做概念语义门，B 端先做证据压缩门，
其余端口先按核心合同重画，再逐步接入。
```

## 5. 端口重分配原则

### 原则 1：单端单责

每个端口只保留一个核心职责，不允许把需求、证据、理论、落库、验证混在一起。

### 原则 2：A 端优先语义，不优先结论

A 端要先把问题解析清楚，再决定下一步去 B/C/D/E/F 哪个端口。

### 原则 3：B 端优先证据，不优先理论

B 端只做 evidence / source pack / coverage matrix，不做最终判断。

### 原则 4：外部技能作为工具，不作为身份

外部 skill 是 A/B/C/D/E 的工具箱，不是端口本身。

### 原则 5：先 A+B，再其他端口

只有 A/B 的职责稳定了，后续 C/D/E/F 的合同才值得进一步精修。

## 6. A 端核心合同（初稿）

### 6.1 A 端一句话定义

```text
A 端 = 需求澄清 / 概念解析 / 控制路由 / 决策前置门。
```

### 6.2 A 端要解决什么

A 端必须把以下问题回答清楚：

```text
1. 用户到底要解决什么。
2. 这个问题属于哪个问题世界。
3. 哪些概念需要先解析。
4. 该用哪些外部技能或内部技能。
5. 下一步应该路由到哪个端口。
6. 什么证据说明澄清已经足够。
```

### 6.3 A 端主要输入

```text
- 用户自然语言需求
- 现有项目上下文
- 已有真源文件
- 当前端口职责/技能栈定义
- 外部技能/方法论候选
```

### 6.4 A 端主要输出

```text
- Demand Grilling Brief
- Concept Parsing Brief
- Port Routing Brief
- Skill Invocation Plan
- Owner Decision Brief（必要时）
```

### 6.5 A 端禁止事项

```text
- 不直接写最终理论结论。
- 不替 B 端做证据压缩。
- 不替 C 端做深度方案生成。
- 不替 D 端做 repo 落库。
- 不替 E 端做独立验证。
```

### 6.6 A 端最小验收

```text
1. 术语被解析成专业概念族。
2. 问题世界 / solution world / shared phenomena 清楚。
3. skill stack 有主 skill、辅助 skill、外部技能桥接。
4. 路由方向明确：B / C / D / E / F。
5. 有可复用的 durable artifact。
```

## 7. B 端核心合同（初稿）

### 7.1 B 端一句话定义

```text
B 端 = 资料压缩 / Source Pack / 证据收敛门。
```

### 7.2 B 端要解决什么

```text
1. 把散乱资料变成可消费的证据包。
2. 明确哪些证据存在、哪些未知、哪些冲突。
3. 给 C 端或 A 端提供干净上下文。
4. 避免让 C 端直接读大堆原始材料。
```

### 7.3 B 端主要输入

```text
- 项目文件
- 原始资料
- 样本记录
- 旧知识库
- 外部文章/外部 skill source
```

### 7.4 B 端主要输出

```text
- Source Pack
- Coverage Matrix
- Evidence Index
- Gaps / Conflicts / Unknowns
```

### 7.5 B 端禁止事项

```text
- 不直接下最终理论判断。
- 不修改 repo 真源。
- 不替代 C 端做方案。
- 不把 unknown 填成结论。
```

### 7.6 B 端最小验收

```text
1. 证据被压缩成可读包。
2. gap/conflict/unknown 清晰列出。
3. 下游能据此继续工作。
```

## 8. 外部技能接入策略

### 8.1 不是“大框架全安装”

用户的真实想法不是把外部框架整个搬进来，而是：

```text
把外部技能作为处理工具，按任务场景组合到端口技能栈里。
```

### 8.2 外部技能的三种接入级别

#### Level 1：Pattern Only

```text
只吸收方法模式，不安装、不执行、不伪装成本地 skill。
```

适合：

```text
- 外部文章
- 外部框架理念
- 参考工作流
```

#### Level 2：Bridge Skill

```text
把外部方法包装成项目内桥接 skill，形成可复用的调用接口。
```

适合：

```text
- 需求拷问补深
- 概念解析补深
- 资料压缩
- 验证审查
```

#### Level 3：Core Stack Skill

```text
被验证长期有效、且会频繁使用的能力，才进入端口主 skill 或稳定辅助 skill。
```

### 8.3 外部技能的使用原则

```text
1. 单个技能优先。
2. 先作为工具，再考虑是否升级为本地 skill。
3. 必须可审计，可回退。
4. 不把外部框架直接当成当前端口的身份。
```

## 9. 需求澄清端的概念解析链

A 端以后必须遵循以下链路：

```text
用户原话
→ 表层词
→ 概念拆解
→ 专业概念族
→ 问题世界建模
→ 检索/证据需求
→ 外部技能候选
→ 端口路由
→ 输出 artifact
```

### 9.1 解析时必须问的核心问题

```text
1. 这句话里的核心名词是什么？
2. 这些名词在专业领域对应哪些概念族？
3. 这件事属于需求、证据、理论、执行还是验证？
4. 哪些地方是表层语言歧义？
5. 哪些地方需要外部技能补深？
6. 哪个端口最适合接手？
```

### 9.2 不允许的简化

```text
- 不能把“听起来最重要”当成“真正该做的”。
- 不能把表层词直接当成专业术语。
- 不能把用户的个人语义当成已标准化概念。
```

## 10. A+B 先交付什么

本次初步需求要求先产出两样东西：

### 10.1 A 端一页式拷问/控制 brief

包含：

```text
- 目标
- 概念解析
- 问题世界
- 端口路由
- 外部技能候选
- 风险/失败模式

```

### 10.2 B 端 Source Pack / 技能栈矩阵

包含：

```text
- 主 skill / 辅助 skill / 外部桥接 skill
- 每个 skill 的输入/输出/禁区
- 哪个环节消费哪个输出
- 哪些是 pattern only
```

## 11. 这份需求文档的验收标准

如果这份文档是对的，那么它应该满足：

```text
1. AI 一眼能看出 A/B 的边界。
2. 外部技能接入不是“堆列表”，而是“按职责编排”。
3. 概念解析比之前更深，不再只停留在最高价值问题。
4. 后续 C/D/E/F 可以在这个 A/B 基座上继续专业化。
5. 任何新增技能都能回答：为什么要加、加在哪、输出被谁消费。
```

## 12. 当前结论

```text
需求澄清端先做概念解析，是对的。

A 端不是问题排序器，而是概念语义闸门。
B 端不是理论生成器，而是证据压缩闸门。

外部技能应该作为单个可调用工具接入技能栈，而不是整体照搬外部框架。

先把 A+B 做专业，后面的 C/D/E/F 才有专业基础。
```
