# A端审视：是否拆分多端口 Bot v0.1

> 分线：10_工程化交接 / A端审视
> 问题：当前 AI 工作流是否应该拆成多个端口 bot（A/B/C/D/E/F 各自独立）？
> 结论先行：**暂不拆。先保留单 Hermes 控制面 + 逻辑端口分层。**

---

## 1. 把问题改写成 A 端可判断的问题

原问题：

```text
要不要把 A/B/C/D/E/F 拆成多个独立 bot？
```

A 端改写后：

```text
在当前工作流认知尚未完全稳定、端口提示词还在定义中的阶段，
是否应该优先用单 Hermes 控制面治理 A/B/C/D/E/F，
而不是马上把每个端口拆成独立 bot？
```

---

## 2. A 端结论

### 建议

**当前不要拆。**

原因不是“不能拆”，而是当前阶段更需要：

- 统一认知
- 统一提示词
- 统一 handoff
- 统一 skill 编组
- 统一验证标准
- 统一记忆和索引

如果现在拆成多个 bot，最容易出现：

```text
职责还没稳定 → bot 先分散 → 认知更乱 → 交接更复杂
```

---

## 3. A 端判断依据

### 3.1 当前认知还在成型期

目前刚刚完成的是：

- Hermes / Codex / Agent Reach / Skills / Repo / Owner 的边界定义
- A/B/C/D/E/F 的端口认知与提示词总纲
- 当前可调用 skill 组合清单
- 第一阶段 skill 编组
- A+B+C 知识循环治理

这说明：

```text
流程骨架刚立起来，还在稳态校准阶段。
```

如果此时拆 bot，容易把“正在定型的治理体系”拆散。

---

### 3.2 现在更需要“单一控制面”

当前最稳的是：

```text
单 Hermes 控制面
+ 多端口逻辑分层
+ 外部能力按需调用
```

这样好处是：

- 统一记忆
- 统一路由
- 统一索引
- 统一 prompt
- 统一 handoff
- 统一审查标准

---

### 3.3 多 bot 会带来额外协调成本

如果拆成多个 bot，立刻新增这些成本：

- 每个 bot 的职责边界还要重新解释
- 每个 bot 的 prompt 要单独维护
- 每个 bot 的输出格式要单独对齐
- 每个 bot 的记忆、索引、变更记录要分开
- Owner 需要在多个 bot 之间切换

这会让“治理成本”上升。

---

## 4. 拆分决策表

| 情况 | 现在是否适合拆 bot | 原因 |
|---|---|---|
| 端口职责还在定义 | 否 | 先把职责定稳，再考虑拆 |
| handoff 还在调整 | 否 | 交接协议不稳，拆了更乱 |
| skill 组合还在补齐 | 否 | 先把组合跑通，再拆分职责 |
| 权限边界很强（读/写/验/决策隔离） | 可以考虑 | 需要强隔离时，拆 bot 有意义 |
| 吞吐量很大、并行很多 | 可以考虑 | 单 bot 处理不过来时，拆分提升吞吐 |
| 各端口流程已稳定、输出标准固定 | 可以考虑 | 这时拆分才不容易乱 |

---

## 5. 当前更好的方案

### 方案 A：推荐

```text
Hermes 作为总控 bot
├─ A 逻辑端口
├─ B 逻辑端口
├─ C 逻辑端口
├─ D 逻辑端口
├─ E 逻辑端口
└─ F 逻辑端口
```

### 方案 B：暂缓

```text
A bot / B bot / C bot / D bot / E bot / F bot
```

当前不优先。

---

## 6. 什么时候才考虑拆

满足下面 3 条再拆：

1. **职责稳定**：每个端口的输入输出和禁区都固定了。
2. **交接稳定**：handoff 模板固定，传输边界清楚。
3. **权限稳定**：哪些 bot 能读、能写、能验、能决策，已经明确。

如果这三条还没稳定，先别拆。

---

## 7. A 端建议的下一步最小动作

不要先拆 bot，而是先做下面三件事：

### 7.1 把 A/B/C/D/E/F 的提示词继续压实

先把每个端口的：

- 输入
- 输出
- 禁区
- handoff
- 失败回退条件

全部定稳。

### 7.2 把当前可调用 skill 组合跑熟

先在单 Hermes 下跑通：

```text
A 端拷问 → B 端 Source Pack → C 端收束 → D 端落库 → E 端验证 → F 端决策
```

### 7.3 再决定是否需要 bot 级拆分

当你发现：

- 某个端口任务量明显过大
- 某个端口需要独立权限隔离
- 某个端口必须持续并行处理

再考虑拆 bot。

---

## 8. A 端最终判断句

```text
当前不建议拆分为多个端口 bot；
先用单 Hermes 控制面把 A/B/C/D/E/F 的认知、提示词、skill 组合和 handoff 跑稳。
```

---

## 9. 如果以后要拆，推荐拆法

### 优先拆成“职责 bot”，不是“端口 bot”

比如：

- 需求拷问 bot
- 外部资料 bot
- 理论收束 bot
- 执行落库 bot
- 验证审查 bot

这样比直接按 A/B/C/D/E/F 平铺更好治理。

---

## 10. 一句话收束

```text
现在先不拆 bot；先把单 Hermes 控制面下的端口治理跑稳，再决定是否做多 bot 分拆。
```
