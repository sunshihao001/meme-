# 内部组合技能更新分层方案 v0.1

> 分线：10_工程化交接 / 技能栈更新  
> 状态：v0.1 / 可落库草案  
> 目的：给 A/B/C/D/E/F 六端的内部组合技能建立一个可更新、可拆分、可桥接、可弃用的分层模型，避免“技能越加越乱”。

---

## 1. 更新目标

当前需要解决的问题是：

```text
1. 端口职责已分清，但内部组合技能还需要结构化。
2. 外部技能接入方式需要标准化。
3. 有些技能应当内化为核心，有些只做桥接或模式参考。
4. 更新时要能拆分、合并、替换、弃用，而不是只会新增。
```

因此，本方案把内部组合技能分成六层：

```text
L1 端口主技能
L2 通用辅助技能
L3 外部桥接技能
L4 领域专用技能
L5 模板/交付技能
L6 状态/边界标记
```

---

## 2. 六层定义

### L1 端口主技能

这是每个端口的身份核，决定它是谁。

```text
A/B/C/D/E/F 的主 skill 只保留一个核心身份，不混杂其他职责。
```

示例：

```text
A = meme-demand-control-port
B = meme-source-pack-port
C = meme-theory-codex-port
D = meme-repo-landing-port
E = meme-verification-review-port
F = meme-owner-decision-port
```

---

### L2 通用辅助技能

跨端口复用的底层能力。

```text
- dbs-good-question
- dbs-deconstruct
- dbs-goal
- dbs-decision
- ai-method-wheel
- maintainer-orchestrator
- github-pr-workflow
- github-repo-management
- github-code-review
- systematic-debugging
- requesting-code-review
```

这些技能适合作为“常用工具箱”，但不应替代端口主技能。

---

### L3 外部桥接技能

来自外部方法论、外部仓库或外部工作流，经过项目内包装后再使用。

典型形态：

```text
- spec-kit bridge
- superpowers bridge
- dbs 系列桥接
- 外部文章方法桥接
```

规则：

```text
- 可以桥接，不要直接伪装成主技能。
- 桥接前要先判断是否稳定、是否可复用、是否真正需要。
```

---

### L4 领域专用技能

绑定到具体业务域或研究域。

例如在当前项目里可能包括：

```text
- 结构术语解析
- 样本标注
- 反证样本收集
- MQL5 输出辅助
- 图表/指标解释
- 领域词典维护
```

规则：

```text
- 领域技能只对当前项目有效，不要冒充通用方法技能。
- 若领域变化大，优先抽象出模板或桥接层，而不是硬编码进主干。
```

---

### L5 模板/交付技能

用于统一输出格式和交付标准。

典型包括：

```text
- Demand Grilling Brief 模板
- Source Pack 模板
- Theory Package 模板
- Landing Handoff 模板
- Verification Report 模板
- Owner Decision Brief 模板
- 一页执行面板
- 新 bot 初始化 prompt 包
```

规则：

```text
- 模板应尽量稳定。
- 模板是交付标准，不是执行本身。
```

---

### L6 状态/边界标记

用于控制技能是否真的被执行、是否只是引用、是否处于边界状态。

推荐状态：

```text
listed_only
installed
loaded
activated
executed
output_consumed
referenced_only
conditional_not_executed
missing
forbidden
deprecated
alias
```

规则：

```text
- 每个技能都要有状态。
- 不要把 referenced_only 写成 executed。
- deprecated 可以保留别名，避免旧 prompt 直接失效。
```

---

## 3. 更新动作分类

每次更新一个技能时，只能落入下列动作之一：

```text
add      新增
split    拆分
merge    合并
replace  替换
alias    保留旧名映射
bridge   做桥接
deprecate 弃用但暂不删除
remove   删除
```

---

## 4. 如何判断该怎么改

### 4.1 该拆分的信号

```text
- 一个技能内部同时承担多个不同职责
- 你每次调用它只用其中一部分
- 它的输出格式不稳定
- 它经常让上下游混淆
```

### 4.2 该合并的信号

```text
- 两个技能本质上做同一件事
- 只是说法不同、实现重复
- 维护成本高于收益
```

### 4.3 该桥接的信号

```text
- 外部方法有价值，但还没稳定到主栈
- 需要保留原方法语义，但又要纳入项目流程
- 不适合立刻内化成主 skill
```

### 4.4 该弃用的信号

```text
- 职责已被更好的技能替代
- 技能长期不用
- 技能导致混乱大于收益
```

## 6. 建议的更新流程

### 6.1 先找重复出现的问题

```text
- 哪些环节总卡住？
- 哪些技能总是被临时加词？
- 哪些外部方法反复出现？
- 哪些输出格式最容易混乱？
```

### 6.2 再归类到层级

```text
- 端口职责 → L1
- 常用工具 → L2
- 外部方法 → L3
- 项目业务 → L4
- 输出模板 → L5
- 执行边界 → L6
```

### 6.3 再决定动作

```text
add / split / merge / replace / alias / bridge / deprecate / remove
```

### 6.4 最后做验证

```text
- 该技能是否仍有独立价值？
- 是否影响上游/下游理解？
- 是否需要保留别名？
- 是否可以模板化？
```

---

## 7. 推荐的最小更新原则

```text
1. 先更新主干，再更新辅助。
2. 先桥接，再内化。
3. 先模板化，再细化。
4. 先保留 alias，再删旧名。
5. 每次只改一类，不要全局乱动。
```

---

## 8. 结论

```text
内部组合技能的更新，最稳的方式不是不断加新技能名，而是先按层级分清楚：

- 谁是主干
- 谁是辅助
- 谁是桥接
- 谁是领域
- 谁是模板
- 谁是边界

这样后续更新才不会越改越乱。
```
