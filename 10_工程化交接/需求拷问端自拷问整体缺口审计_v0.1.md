## 2. problem_world

需求端现在不是“完全没有方法”，而是进入了更危险的中间态：

```text
1. 方法越来越多：V0.5、DBS overlay、intent-brainstorm-grill、semantic divergence、skill orchestration、runtime audit、validator、GitHub issue、Codex handoff。
2. 部分能力已经真实执行：Issue #2 / #3 / #4、loop report validator、P73 activation artifact、CI。
3. 但很多能力仍靠 Agent 手动记忆串联，尚未成为默认运行路径。
4. 因此主要风险从“缺方法”转为“方法-执行-验证-回填之间断链”。
```

核心冲突：

```text
Owner 要的是一个能持续拷问问题、吸收知识、生成可执行任务并自证执行状态的需求端；当前仓库已经有多个局部样例，但还没有一个稳定的端到端运行产品。
```

## 3. concept_map_seed

```text
Owner ask
→ semantic divergence
→ intent-brainstorm-grill clarification
→ durable artifact
→ skill runtime audit
→ issue / handoff
→ implementation / validator
→ CI / Git evidence
→ project index / changelog
→ continuation brief next entry
→ next owner decision
```

当前断点主要分布在：

```text
A. entry routing：什么时候进入哪个 loop 仍依赖 Agent 判断。
B. evidence generation：报告字段可被 validator 检查，但字段内容仍主要手写。
C. artifact consumption：artifact 已被索引/变更记录消费，但未都转为后续任务。
D. product boundary：需求端是项目内控制层，还是独立产品线，未决。
E. external knowledge absorption：能搜索/吸收外部概念，但尚未形成稳定检索队列和评价机制。
F. old knowledge recall：旧资料/旧会话/旧方法源的召回仍不够自动。
G. owner feedback loop：哪些问题必须问 owner、如何只问一个关键问题，仍靠纪律而非工具约束。
```

## 5. 需求端自身的 7 层 funnel

### 5.1 Goal

```text
不是继续增加方法名，而是让需求端稳定地产生：清晰问题 → 可审计 artifact → 可执行 issue/handoff → 验证结果 → 下一轮入口。
```

### 5.2 Users / Operators

```text
Owner：需要减少反复提醒 Agent “你只是写了，没有用起来”。
Agent：需要明确何时加载、执行、停止、回填。
Reviewer / CI：需要自动发现假完成。
Codex / worker：需要拿到边界清楚的 issue/handoff。
未来项目：可能复用需求端作为独立控制层。
```

### 5.3 Boundaries / Non-goals

```text
1. 需求端不是自动替 Owner 做产品方向决策。
2. 需求端不是把所有 DBS skill 机械执行一遍。
3. 需求端不是 trading executor。
4. 需求端不是无限生成方法论文档。
5. 需求端不应让业务仓库变成上游方法论仓库。
```

### 5.4 Assumptions

```text
1. 需求端最有价值的近期用途，是控制当前 MQL5 FCZ 项目的需求/研究/工程化节奏。
2. Owner 更关心真实调用、证据、反证、商业可用性，而不是漂亮的流程图。
3. 文档 + validator + GitHub issue + CI 是当前最可用的低成本产品化路径。
4. 独立产品线是否成立，仍需更多真实运行样本。
```

### 5.5 Failure Modes

```text
假完成：报告字段齐全，但没有真实 tool call / artifact / downstream consumer。
方法膨胀：每次批评都新增一层协议，实际执行更难。
过度收敛：只问“最高价值问题”，丢失语义发散和外部相似概念。
过度发散：一直做概念地图，不落 issue / validator /样本。
边界漂移：从需求端控制层滑向独立产品开发，但没有 owner 决策。
验证空洞：CI 只证明 Markdown/字段存在，不证明需求判断质量。
旧知识断链：新 Agent 读取入口文件，但不知道旧资料哪些能吸收、哪些只读。
商业误判：把一个对 owner 有用的内部工作流误报为可对外销售产品。
```

### 5.6 Options Compared

| 选项 | 好处 | 坏处 | 结论 |
|---|---|---|---|
| A. 继续每轮手写审计报告 | 灵活、成本低 | 依赖 Agent 纪律，容易复发假调用 | 不足以作为长期方案 |
| B. 立即拆成独立需求端产品仓库 / skill | 边界清晰，便于复用 | 过早产品化，可能脱离当前项目现实 | 暂不默认，需要 owner 决策 |
| C. 在当前项目内继续用 issue + handoff + validator 产品化关键断点 | 最小可验证、与现实任务相连 | 仍偏项目内控制层 | 当前推荐 |
| D. 建完整自动化平台 / cron / dashboard | 自动化程度高 | 过重，且 side effects 风险高 | 现在不做 |

### 5.7 Acceptance Criteria for “需求端更成熟一层”

```text
1. 至少 3 类能力达到 executed_once + workflow_integrated：GitHub issue、subprotocol activation、validator/CI。
2. 每个 loop report 都能说明 loaded / executed / output_consumed / forbidden，而不是只列方法。
3. 关键 artifact 都有 downstream consumer：issue、handoff、validator、index、changelog、brief。
4. 下一步任务不是“继续完善理论”，而是可验证的具体缺口。
5. Owner 决策点被压缩成少数高价值问题，不被 Agent 擅自决定。
```

## 6. 当前最重要的未解决问题

### P0：需求端产品边界未决

```text
问题：它到底是 MQL5 FCZ 项目内的控制层，还是一个可迁移的独立“需求认知编排器”？
影响：决定是否应该创建 standalone Hermes skill、独立 repo、独立 docs、独立 evals。
当前建议：先保持为项目控制层，累计 3-5 个 executed_once 样例后再决定是否拆出。
```

### P1：执行证据仍可被“字段存在”伪装

```text
问题：validator 检查报告有字段，但不能证明 skill_view/tool call 真的发生。
影响：未来仍可能出现“报告写得像执行过”。
当前建议：报告必须记录 tool evidence 摘要；必要时增加 run log / transcript reference，但不把聊天作为唯一真源。
```

### P2：activation template 尚未进入默认路径

```text
问题：P73 是一次成功样例，Issue #4 是手工创建的下一步；未来是否默认使用模板还未验证。
影响：下一轮 Agent 可能又忘记 subprotocol_activation_plan。
当前建议：执行 Issue #4，把模板接入 loop contract / handoff / validator tests。
```

### P3：需求端缺少“质量而非字段”的评价

```text
问题：现在能检查报告有没有字段，不能检查问题是否真的 hard-to-vary、是否暴露冲突、是否避免假设。
影响：会出现格式合格但认知质量低的报告。
当前建议：设计轻量 review checklist，不急于自动打分；先用于人工/code review。
```

### P4：旧资料与旧会话召回没有统一协议

```text
问题：SOURCE_OF_TRUTH 和索引解决项目文件入口，session_search / 外部方法源 / 旧资料吸收仍靠 Agent 判断。
影响：复杂问题可能重复发明，或漏掉过去已解决的边界。
当前建议：新增“旧知识召回协议”小切片：什么时候查 session_search，什么时候查项目索引，什么时候外部搜索。
```

### P5：外部概念吸收缺少 backlog / score

```text
问题：P66/P69 已证明可以吸收外部概念，但没有持续队列与吸收质量标准。
影响：外部搜索会变成偶发动作。
当前建议：给 concept search absorption loop 增加候选队列：source、concept、why relevant、absorbed_by、rejected_reason。
```

### P6：maker/checker 回路尚未真正跑多轮

```text
问题：Codex handoff 模板有了，但 maker 执行、checker review、CI、owner decision 的完整闭环样本不足。
影响：AI Method Wheel 仍偏“准备好了”，不等于“多轮稳定运行”。
当前建议：优先执行 Issue #2 或 #4 的一个小闭环，保留 RED/GREEN/CI/PR 证据。
```

### P7：商业可用性还没有被单独审计

```text
问题：需求端现在对 owner 有用，但是否对外部用户有产品价值、交付形态、失败成本、付费场景，未审计。
影响：如果过早称为产品，会把内部工作流误判为商品。
当前建议：等 3-5 个真实项目样例后，再做商业 readiness brief。
```

## 7. 当前推荐收敛切片

```text
优先执行 Issue #4：make demand subprotocol activation evidence reusable。
```

原因：

```text
1. 它直接承接 P73，不另起炉灶。
2. 它把“成功样例”升级为“未来默认模板”。
3. 它能减少下一轮再次靠 Agent 记忆手写 activation evidence 的风险。
4. 它不需要 owner 做产品线拆分决策。
```

暂不选：

```text
1. 立即新建 standalone Hermes skill：需要 owner 决策，且可能过早产品化。
2. 做商业 readiness：样例数量还不够。
3. 做 cron/dashboard：side effects 和复杂度过高。
4. 继续抽象理论：当前更缺默认执行路径。
```
