# dbskill-A端判断增强包设计 v0.1

> 分线：10_工程化交接 / 外部技能接入 / C端设计  
> 输入：dbskill第一批SourcePack_v0.1.md  
> 输出定位：C端把 dbskill 第一批四个外部技能包装为 A端可调用的判断增强模块。  
> 状态：v0.1 / 设计草案，等待 D端落库与 E端验证。

---

## 1. 设计结论

```text
不把四个 dbskill 作为四个独立主 skill 安装。
```

而是包装成一个内部增强包：

```text
dbskill-A端判断增强包
```

增强包只做一件事：

```text
辅助 A端把模糊输入转成可路由、可验证、可执行的 Demand Grilling Brief。
```

---

## 2. 四个能力的合并顺序

```text
1. dbs-good-question：先判断问题清晰度和 Agent 可解性。
2. dbs-deconstruct：再拆模糊概念和语言陷阱。
3. dbs-goal：再把愿望语法改成可检查目标。
4. dbs-diagnosis：最后判断商业问题是否成立、是否需要消解。
```

不是每次都四个全跑。A 端根据输入触发：

```text
模糊问题 → good-question
术语/概念混乱 → deconstruct
目标空泛 → goal
商业问题/业务判断 → diagnosis
```

---

## 3. A端调用规则

### 3.1 触发 dbs-good-question

触发条件：

```text
- 用户输入是模糊问题
- 用户问“能不能自动化”
- 用户想让 Agent 做事但目标/边界不清
- 输入缺少对象、目标、冲突、约束、反馈
```

输出进入 Demand Grilling Brief：

```text
问题定义
当前清晰度
Agent 可解性
最大缺口
最小补充问题
```

---

### 3.2 触发 dbs-deconstruct

触发条件：

```text
- 输入里有模糊概念
- 同一个词可能有多种含义
- 用户把 Problem 伪装成 Question
- 语言里有“赛道、定位、价值、系统性、长期”等未定义词
```

输出进入 Demand Grilling Brief：

```text
概念拆解
术语定义
伪概念判断
Question / Problem 区分
假设列表
```

---

### 3.3 触发 dbs-goal

触发条件：

```text
- 用户说“我想做…”
- 目标像口号、愿望、状态，而不是交付物
- 不知道什么时候算完成
- 不知道下一步是什么
```

输出进入 Demand Grilling Brief：

```text
原始目标
空转词
可检查目标
验收标准
下一步动作
```

---

### 3.4 触发 dbs-diagnosis

触发条件：

```text
- 用户提出商业模式、增长、变现、项目判断问题
- 问题可能本身不成立
- 存在语言陷阱、假设错误、逻辑错误、事实前提不明、信息不足
```

输出进入 Demand Grilling Brief：

```text
问题类型
是否需要消解
隐含假设
事实核验项
是否进入 C/F 决策
```

## 5. 权限边界

```text
1. dbskill-A端判断增强包不是 A端主 skill。
2. 它不能替代 meme-demand-control-port。
3. dbs-diagnosis 不能替代 F 端 Owner Decision。
4. 外部商业哲学只作为桥接观点，不作为项目真源结论。
5. 所有接入结果必须落到 repo 文件，可验证后再固化。
```

## 7. E端验证建议

E端用真实样例验证：

```text
“我现在内部技能不够完善，想从 GitHub 外部技能仓库添加技能，
但不知道哪些该接入、怎么判断、怎么放进我的 A/B/C/D/E/F 多端口循环里。”
```

检查标准：

```text
1. 是否识别为外部技能接入问题。
2. 是否拆清“内部技能 / 外部技能 / 接入 / 桥接 / 合并 / 拒绝”。
3. 是否把目标转为可检查交付物。
4. 是否给出 A/B/C/D/E/F 路由。
5. 是否给出停止条件和 F端决策点。
```
