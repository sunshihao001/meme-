# AI 技术对标与自我定位地图 v0.1

> 位置：`10_工程化交接/AI技术对标与自我定位地图_v0.1.md`  
> 分线：AI 技术对标 / 需求控制层 / 产品化自我定位  
> 状态：v0.1，第一版定位地图  
> 上游：`10_工程化交接/AI技术玩不明白感_需求澄清端诊断_v0.1.md`  
> 用途：把“别人 AI 技术玩得明白”拆成可对标维度，判断当前已经掌握什么、缺什么、下一步应补哪一类能力。

---

## 1. 结论先行

```text
当前不是“不会 AI 技术”，而是 AI 能力散在多个层级：

1. prompt / 对话能力：已有，但不是主要瓶颈。
2. agent / skill 编排能力：已有局部样例，但默认路径不稳定。
3. knowledge base / 项目记忆能力：已有大量文件，但主线权重刚开始治理。
4. coding workflow / 工程闭环能力：已有 issue / handoff / CI 样例，但还不是稳定产品流程。
5. automation / 自动化能力：较弱，更多还是人工驱动循环。
6. productization / 产品化展示能力：当前最弱。
```

最短判断：

```text
差距不是“技术名词少”，而是“可展示、可复用、可验证的 AI 工作流产品形态”还没成型。
```

---

## 2. 对标维度

### 2.1 Prompt / 提问与对话能力

当前状态：

```text
中高
```

证据：

```text
1. 已经能通过需求澄清端把模糊问题改写成 Agent 可执行问题。
2. 已经形成 Good Question / intent-brainstorm-grill / DBS overlay 等方法入口。
3. 已多次识别“问题不是表面词，而是语义转译、边界、主线、证据链”。
```

短板：

```text
prompt 能力没有稳定沉淀成短入口。
很多时候看起来像一堆方法名，而不是一个简单操作界面。
```

对标别人：

```text
别人看起来 prompt 玩得明白，往往是因为他们把入口包装成简单命令或固定流程；
不是因为每次都解释一大套方法论。
```

判断：

```text
不是 P0 短板。
```

---

### 2.2 Agent / Skill 编排能力

当前状态：

```text
中
```

证据：

```text
1. 已经能加载 dbs-good-question、dbs-decision、spec-first-ai-engineering 等 skill。
2. 已经识别 listed_only / loaded_reference / executed_once / workflow_integrated 的区别。
3. 已经建立需求拷问端子协议激活修正、runtime evidence、loop report validator 等文件。
```

短板：

```text
1. 是否调用某个 skill，仍主要依赖 Agent 本轮纪律。
2. 很多能力已经被记录，但还没有变成默认自动路由。
3. validator 能检查字段存在，但不能完全证明认知质量。
```

对标别人：

```text
别人看起来 agent 玩得明白，通常有更稳定的入口：
一个命令、一个 workflow、一个 dashboard、一个 agent role、一个固定 runbook。
```

判断：

```text
P1 短板：需要从“能调用”变成“默认路由”。
```

---

### 2.3 Knowledge Base / 项目记忆能力

当前状态：

```text
中高，但杂乱感强
```

证据：

```text
1. 项目已有总纲、策略定义、资料吸收、样本标注、规则回测、工程交接。
2. 已新增资料吸收主线参考总表，开始区分主线核心、主线辅助、参考资料。
3. 已把量化体系分层判断回填到总纲。
```

短板：

```text
1. 文件数量多，入口层级仍偏重。
2. 10_工程化交接 中混有项目交接、需求控制层、AI 方法论、产品化诊断。
3. 旧资料、旧会话、散乱知识点的召回协议还不够短。
```

对标别人：

```text
别人看起来知识库玩得明白，往往是因为他们有一个更短的主页：
现在是什么 / 下一步是什么 / 哪些是真源 / 哪些只是归档。
```

判断：

```text
P1 短板：需要一个更短的项目操作面板，而不是继续增加长文档。
```

---

### 2.4 Coding Workflow / 工程闭环能力

当前状态：

```text
中
```

证据：

```text
1. 已有 spec / plan / tasks / checklist。
2. 已有 GitHub issue draft / Codex handoff / PR / CI 思路。
3. 已有 validators：sample CSV、state paths、loop reports、index references、sample records、validate_all。
4. MQL5 observer MVP 曾完成源码、MetaEditor 编译、安装证据。
```

短板：

```text
1. coding workflow 还不是主线；有时会被理论扩展淹没。
2. Codex / maker / checker / PR review 的多轮稳定闭环样本还不足。
3. 工程任务和研究任务之间的 gate 仍需要更清晰。
```

对标别人：

```text
别人看起来 AI coding 玩得明白，通常是因为他们能展示：
Issue → Agent PR → Test → Review → Merge 的连续证据。
```

判断：

```text
P2 短板：不是没有工程能力，而是缺少可复用展示链路。
```

---

### 2.5 Automation / 自动化能力

当前状态：

```text
低到中
```

证据：

```text
1. 当前主要是 Agent 在会话里执行循环。
2. 有 validator 和 CI，但还少自动队列、自动路由、自动状态面板。
3. cron / dashboard / bot command / scheduled report 等产品化自动化还未成为主线。
```

短板：

```text
自动化还没有变成“你输入一个目标，系统自动跑一套固定闭环并输出状态面板”。
```

对标别人：

```text
别人看起来自动化玩得明白，往往是因为他们有可见自动执行物：
定时任务、机器人、仪表盘、自动报告、自动 PR、自动 QA。
```

判断：

```text
P2 短板：需要先做一个小自动化 demo，而不是直接做大平台。
```

---

### 2.6 Productization / 产品化展示能力

当前状态：

```text
低
```

证据：

```text
1. 已有很多能力，但还没有一句话产品名。
2. 当前成果更多是文件、流程、知识库，而不是可展示界面或稳定入口。
3. Owner 的焦虑主要来自“我做了很多，但不像别人能拿出一个清楚东西”。
```

短板：

```text
1. 没有产品主页。
2. 没有 demo 脚本。
3. 没有 before / after 展示。
4. 没有一键运行或短命令入口。
5. 没有明确 target user：只给自己用，还是给别人用。
```

对标别人：

```text
别人看起来 AI 技术强，很多时候是产品化展示强：
他们把一套内部流程包装成一个看得懂、跑得动、能演示的东西。
```

判断：

```text
P0 短板。
```

## 4. “别人 AI 技术玩得明白”可能具体强在哪

### 4.1 他们可能不是技术更深，而是入口更短

```text
别人：一个命令 / 一个页面 / 一个 demo。
当前：很多文件 / 很多方法 / 很多解释。
```

### 4.2 他们可能不是懂更多模型，而是更会闭环

```text
别人：输入 → 输出 → 验证 → 展示。
当前：输入 → 分析 → 文件 → 再分析 → 再文件。
```

### 4.3 他们可能不是研究更深，而是更会包装成产品

```text
别人：AI Agent for X。
当前：项目 + 方法轮 + 需求端 + DBS + MQL5 + GMGN + Codex 混合体。
```

### 4.4 他们可能不是文档更多，而是主线更短

```text
别人：三句话说明白。
当前：需要读多个总纲和交接文件才明白。
```

## 5. 当前最该补的不是哪类

### 5.1 不是先补模型知识

```text
不要默认去学更多模型名、插件名、prompt 技巧。
```

### 5.2 不是先补更复杂的方法论

```text
不要继续叠加新的框架名称。
```

### 5.3 不是先补大自动化平台

```text
当前还没到做完整 dashboard / SaaS / multi-agent platform 的阶段。
```

## 6. 当前最该补的三件事

### P0：一句话产品形态

需要回答：

```text
这套 AI 工作流如果对外展示，它叫什么？
解决谁的什么问题？
输入是什么？
输出是什么？
和普通 ChatGPT 聊天有什么不同？
```

候选表达：

```text
AI Demand Control Plane：把模糊目标变成可审计、可执行、可验证的项目推进系统。
```

### P1：一张操作面板

需要一个很短的入口文件，回答：

```text
现在项目是什么？
当前在哪一层？
下一步只做什么？
哪些文件是真源？
哪些只是参考？
哪些能力已执行？
哪些只是列出？
```

推荐文件：

```text
10_工程化交接/项目AI操作面板_v0.1.md
```

### P2：一个可展示 demo

不要先做大平台，先做一个 demo：

```text
输入：一句模糊需求
处理：需求澄清端拷问 + 资料召回 + 分层判断
输出：Good Question Brief + 下一步 issue/handoff + validator checklist
验证：文件落地 + 索引 + 变更记录 + validate_all
```

推荐 demo：

```text
“把一个混乱研究项目整理成可执行 AI 工作流”的 before / after demo。
```

## 8. 版本结论

```text
当前最短判断：

你已经有不少 AI 能力，但这些能力主要以文件和流程存在；
别人看起来更明白，往往是因为他们有更短的入口、更稳定的闭环、更清楚的产品展示。

所以当前 P0 不是学更多 AI 技术，
而是把已有能力压缩成一个可操作、可展示、可验证的 AI 工作流产品形态。
```
