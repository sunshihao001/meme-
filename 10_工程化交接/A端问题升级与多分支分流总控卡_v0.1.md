# A 端问题升级与多分支分流总控卡 v0.1

> 分线：10_工程化交接 / A端总控
> 目的：把用户自然语言想法升级为专业研究问题，并在 A 端自动分流到 B/C/D 相关轨道。

---

## 1. 核心判断

用户当前不是在问单一问题，而是在验证：

- A 端能否自动把自然语言升级为专业研究问题
- B 端能否压缩资料、外部文章、GitHub skill、上下文为 Source Pack
- C 端能否在研究态/执行态切换
- 组合技能能否自动判断 ADOPT / BRIDGE / MERGE / REJECT / PATTERN_ONLY
- dbskill 是否应辅助 save / restore / report，而不替代 git、E 端验证、F 端决策

因此，A 端的正确动作不是只问一个字母，而是：

```text
输入 → 专业化问题定义 → 分支识别 → 自动调用相关 skill → 继续循环
```

---

## 2. A 端升级问题的工作方式

### 2.1 先把自然语言提升成研究问题

A 端应先做：

1. 识别原始想法
2. 抽出目标
3. 抽出边界
4. 抽出缺口
5. 抽出验收标准
6. 识别是否需要 B / C / 组合技能 / 外部 skill 判断

### 2.2 研究问题的专业化标准

一个问题只有在以下条件下，才算进入 A/B/C 研究链：

- 能说清楚它要解决什么
- 能说清楚不解决什么
- 能说清楚缺少什么证据
- 能说清楚“满意/停机”的条件

---

## 3. 多分支主次关系

当前主次顺序建议如下：

### 主线

- Q1：A 端如何升级自然语言为专业研究问题

### 次线

- Q2：B 端如何压缩资料为 Source Pack
- Q3：C 端如何做研究态 / 执行态双态输出
- Q6：外部 GitHub skill 如何判断 ADOPT / BRIDGE / MERGE / REJECT / PATTERN_ONLY

### 后续支撑线

- Q4：何时回 A / B / C
- Q5：内部组合技能如何分层更新
- Q7：dbskill 如何辅助 save / restore / report

---

## 4. A 端自动调用 skill 的触发原则

A 端不要只靠自然语言。遇到下面情况应自动调用 skill：

- 问题太散 → `brainstorming` / `dbs-good-question` / `grill-me`
- 需要拆概念 → `dbs-deconstruct` / `dbs-goal` / `dbs-decision`
- 需要看外部材料 → `Agent Reach` / `dbs-learning` / `dbs-content-system`
- 需要判断 skill 的采用方式 → `dbskill` 评估链 / ADOPT-bridge-merge-reject 规则

---

## 5. 当前统一停机口径

```text
1. 目标、边界、方案、验证四项中至少 3 项达到可执行级别。
2. 连续 2 轮没有新增有效信息。
3. 最多循环 2~3 轮后，强制复盘是否该转端。
4. 如果继续追问只会改细节、不改路线，就停止研究循环。
```

## 6. 推荐的下一步分流

如果你要继续挖掘，建议按下面顺序：

1. 先定 Q1：A 端如何升级自然语言为研究问题
2. 再定 Q2：B 端 Source Pack 压缩规则
3. 再定 Q6：外部 GitHub skill 的采用判定
4. 最后再补 Q3 / Q4 / Q5 / Q7

```text
A 端的第一职责不是回答，而是把自然语言自动升级成专业研究问题，并按当前缺口自动调用 skill 继续循环。
```
