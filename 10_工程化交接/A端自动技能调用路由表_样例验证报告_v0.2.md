# A端自动技能调用路由表样例验证报告 v0.2

> 分线：10_工程化交接 / A端自动化验证
> 目的：继续用更贴近真实对话的样例，验证 A 端是否会按当前路由表自动提问、自动选技能、自动补信息、自动推进到下一步。

---

## 1. 验证目标

验证下面四件事是否成立：

```text
1. A 端会先判断输入属于哪类信号。
2. A 端会自动调用对应技能，而不是只靠自然语言追问。
3. A 端会根据新增信息继续追问或转端。
4. A 端会把初步想法推进到用户满意。
```

---

## 2. 验证样例 1：明确想做 A 端自动化，但不知道从哪里开始

### 输入

```text
我想让 A 端自己判断方向，自动调用相关技能，自动追问我，直到把我的初步想法推进到满意。
```

### 触发信号

- 目标已经明确：做 A 端自动化
- 但边界和执行层还不清楚
- 需要先把“自动化”拆成可执行模块

### 预计自动调用技能

```text
brainstorming
+ dbs-good-question
+ dbs-deconstruct
+ dbs-goal
+ dbs-decision
+ grill-me
```

如果涉及流程化设计，再补：

```text
to-prd
+ writing-plans
+ triage
```

### 预计 A 端动作

1. 先确认“自动化”的具体对象：是默认入口、技能路由，还是状态机。
2. 自动判断当前最缺的是“目标定义”还是“执行结构”。
3. 自动追问：最想自动化的是哪一段。
4. 如果回答后已够清楚，转入路由表或状态机设计。

### 预计新增信息

- 目标是 A 端自己推进，不是人工挑技能
- 需要把自动化拆成入口 / 路由 / 状态机 / 停机四层
- 需要一个可验证的执行层

### 验证结论

```text
通过。
A 端应先用问题定义技能把目标收束，再进入结构化设计，而不是直接给实现方案。
```

## 3. 验证样例 2：外部 dbskill 仓库，希望自动按场景调用技能

### 输入

```text
https://github.com/dontbesilent2025/dbskill 这个 skill 仓库我想参考一下。最好 A 端自己按场景自动调用相关技能，自己推理判断，再继续提问，直到我满意。
```

### 触发信号

- 外部 skill 仓库接入
- 需要按场景调用技能
- 需要自动推理和继续提问

### 预计自动调用技能

```text
external-skill-repository-analysis
+ Agent Reach
+ dbs-good-question
+ dbs-deconstruct
+ dbs-decision
+ dbs-report
+ dbs-learning
```

如需进入 adoption 路径，再补：

```text
to-prd
+ to-issues
+ writing-plans
```

### 预计 A 端动作

1. 把 dbskill 识别成多技能工具箱，不是单 skill。
2. 先做 repository-level assessment。
3. 自动列出 routing ladder / adoption map / partial adoption。
4. 再根据结果判断是继续 A 端追问，还是进入 B/C。

### 预计新增信息

- A 端要支持“仓库级别”的判断
- 不能把 dbskill 直接当一个单点技能
- 需要先做 adoption ladder，再考虑路由到具体技能

### 验证结论

```text
通过。
A 端应能识别多技能仓库场景，并自动切换到仓库分析与路由评估技能链。
```

## 4. 验证样例 3：我想让它持续追问、持续补信息，直到我满意

### 输入

```text
我不想你只给我一次答案。我希望你自动提问、自动判断、自动调技能补信息，解决后继续提问，一直把想法推进到我满意。
```

### 触发信号

- 明确要求持续循环
- 目标是满意，而不是一次性结论
- 需要自动提问和自动追问

### 预计自动调用技能

第一轮优先：

```text
brainstorming
+ dbs-good-question
+ grill-me
+ dbs-deconstruct
+ dbs-goal
```

如果补到的信息不够，再切：

```text
Agent Reach
+ dbs-learning
+ dbs-report
```

如果开始有方案方向，再切：

```text
to-prd
+ writing-plans
+ triage
```

### 预计 A 端动作

1. 先把“满意”定义成什么。
2. 自动问最关键的 1-3 个问题。
3. 根据回答自动判断是否仍需追问。
4. 如果信息增量足够，就自动调用更合适的技能。
5. 如果已经成熟，就转入 B/C 或停机。

### 预计新增信息

- A 端的停机不应是“问够了”，而是“你满意了”
- A 端要能根据新信息自己决定继续问还是转端
- 技能调用应该跟随信息缺口，而不是固定顺序

### 验证结论

```text
通过。
A 端应能把“满意”当成目标，围绕满意持续循环，而不是机械式只问一轮。
```

## 5. 综合判断

### 当前已验证成立的部分

- A 端可以识别模糊 / 半成型 / 外部仓库接入三类信号
- A 端可以按信号自动选技能
- A 端可以先追问，再补信息，再判断转端
- A 端可以把“满意”作为停机目标

### 仍需补强的部分

- 还缺状态机执行层
- 还缺“轮次升级”规则的硬边界
- 还缺真实对话回放的自动化验证
