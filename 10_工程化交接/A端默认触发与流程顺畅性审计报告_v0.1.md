# A端默认触发与流程顺畅性审计报告 v0.1

> 分线：10_工程化交接 / A端流程 / dbskill 接入运行审计  
> 输入问题：现在 A端流程执行是不是调用不起来的很多？这种架构方式是否能服务“自然语言到专家级想法升级”的目标？是否需要调整优化？  
> 状态：v0.1 / 已落实默认触发规则 + 架构审计结论  
> 目的：解释为什么之前没有自动跑、当前架构是否顺畅、需要如何优化 A端运行方式。

---

## 1. 一句话结论

```text
当前架构方向可以服务目标，但原来的触发方式不够顺畅。
```

具体说：

```text
A端项目 skill 已经有能力；
但当前聊天不会天然每次自动调用项目 skill；
所以需要在 A端 skill 里明确写入“默认自动触发规则”和“轻量版输出路径”。
```

本次已经落实：

```text
当用户输入想法、方案、工作流、skill 接入、是否贴合、是否需要调整、更专业方案等问题时，A端默认先运行 Natural Language to Expert Upgrade Brief。
```

---

## 2. 为什么刚刚没有自动跑

原因不是 dbskill 完全没接入，而是状态层级不同。

当前之前的状态是：

```text
project_skill_updated
small_scope_enabled
needs_more_samples
```

这代表：

```text
项目级 A端 skill 已经更新；
但不是 Hermes 全局 runtime 自动触发；
也不是当前 Telegram 每句话都强制先跑 A端模板。
```

所以会出现：

```text
规则在文件里，但聊天回复没有自动套用。
```

这就是：

```text
文档/项目 skill 接入 ≠ 当前对话自动运行
```

---

## 3. 当前 A端流程里调用不顺的地方

当前“不顺畅”主要有四类：

```text
1. 触发条件不够显式：什么时候默认跑 A端升级模板没有写死。
2. 输出模板太重：完整版 Demand Grilling Brief 字段太多，不适合每条自然语言都展开。
3. 状态词容易误解：project_skill_updated 容易被误以为 runtime_auto_triggered。
4. dbskill 是融合进 A端行为，不是独立按钮；如果 A端不主动套模板，就看起来像没调用。
```

因此问题不是：

```text
架构完全错了。
```

而是：

```text
架构需要一个更轻、更明确的默认运行层。
```

---

## 4. 本次已落实的调整

已在 A端主 skill 中新增：

```text
### 2.1 默认自动触发规则
```

触发条件包括：

```text
1. 用户表达一个想法、判断、方向、方案、工作流、skill 接入或优化问题。
2. 用户问“是否贴合 / 是否需要调整 / 更专业方案是什么 / 怎么做更好”。
3. 用户用自然语言描述自己不专业、想让 AI 帮忙升级认知或方案。
4. 用户在讨论 A端、dbskill、外部 skill 接入、方法轮、循环代理、端口职责。
```

默认输出改成轻量版：

```text
1. 我理解你真正想说的是
2. 当前想法的限制
3. 专业化升级版本
4. 建议工作流
5. 下一步最小动作
```

只有进入落库、外部仓库接入、跨端口执行或 Owner 决策时，才展开完整版。

## 5. 当前架构是否能服务上面的目标

结论：

```text
能服务，但要按“两层输出”运行。
```

### 第一层：日常轻量升级

适合大多数自然语言输入。

```text
用户随口说想法
→ A端轻量升级
→ 输出专业但易懂的解释和下一步
```

特点：

```text
快
短
能看懂
不堆模板
不每次落库
```

### 第二层：正式工程化流程

适合真正要落库、接 skill、跑外部仓库、进入多端口循环时。

```text
用户确定要推进
→ A端完整版 Brief
→ B/C/D/E/F 分派
→ 文件落库
→ 验证
→ Owner 决策
```

特点：

```text
慢
完整
可验证
可追溯
适合长期资产
```

---

## 6. 是否需要进一步优化

需要，但不是换架构，而是补一个“运行层”。

建议下一步优化为三层：

```text
L0 日常自动轻量触发
L1 A端正式需求澄清 / 自然语言升级
L2 A-F 多端口工程化循环
```

### L0：日常自动轻量触发

```text
当用户表达想法时，自动输出 5 项轻量升级。
```

### L1：正式 A端澄清

```text
当用户说“继续推进 / 落库 / 接入 / 做成方案”时，输出完整 Natural Language to Expert Upgrade Brief。
```

### L2：多端口循环

```text
当需要证据、理论、文件、验证、决策时，进入 B/C/D/E/F。
```

```text
是否需要继续样例验证：需要
```

## 8. 专业化建议

现在不要急着做全局 runtime skill。

更专业的做法是：

```text
1. 先在 A端项目 skill 中固定默认触发规则。
2. 后续 2-3 次真实输入都按轻量版自动跑。
3. E端验证输出是否更清楚、更专业、更可执行。
4. 如果稳定，再考虑做 Hermes profile runtime skill 或新 bot system prompt 固化。
```

原因：

```text
全局自动触发太早，容易把所有聊天都流程化，反而变重。
```

## 9. 本次结论

```text
当前架构方向正确，但原来缺少 L0 轻量自动触发层。
```

本次已经补上：

```text
A端默认触发规则 + 轻量版 Natural Language to Expert Upgrade 输出路径。
```

后续验证重点：

```text
用户不用显式说“跑A端”，只要输入复杂想法，本 A端就应该主动先升级自然语言表达，再给下一步。
```
