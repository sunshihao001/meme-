# A端强制触发词与 skill 自动调用入口卡 v0.1

> 分线：10_工程化交接 / A端入口门禁
> 目的：把 A 端从“能理解”变成“能被强制拉起”，让用户用一段固定口令把 A 端的自动拷问、自动判断、自动调用 skill、自动继续循环真正触发起来。

---

## 1. 结论

A 端现在不是缺能力，而是缺一个**强制触发门**。

这份卡的作用就是：

```text
用户一句话 → A 端自动进入拷问模式 → 自动调用相关 skill → 自动判断是否继续 → 满足条件后再转端
```

---

## 2. 适用场景

当你遇到以下情况时，直接用这张卡：

- 随口提了一个想法，但还没整理清楚
- 想先挖需求，不要直接方案
- 想让 A 端自动判断该问什么
- 想让 A 端自动调用相关 skill，而不是只聊天
- 想让 A/B/C 逻辑分端，但不拆成多个物理 bot

---

## 3. 强制触发词

### 3.1 标准版

你以后可以直接复制这一段：

```text
请按 A 端强制触发模式处理。

现在先不要直接给方案，也不要先进入 B/C/D/E/F。
请先自动拷问我的目标、边界、缺口、验收标准。
如果当前信息不足，请继续追问。
如果需要补信息，请自动调用相关 skill，而不是只靠自然语言问答。

请遵守 A/B/C 收束规则：
目标、边界、方案、验证四项中至少 3 项达到可执行级别；
连续 2 轮没有新增有效信息就停；
最多循环 2~3 轮后强制复盘是否该转端；
如果继续追问只会改细节、不改路线，就停止循环。
```

### 3.2 更强版

如果你希望更明显地进入 deep_path，可以用：

```text
请按 A 端深度拷问模式处理，进入 deep_path。
不要快结论，不要先给方案，不要先切端。
先把我的想法逼成可研究、可判断、可路由的问题说明书。
必要时自动调用相关 skill 来补齐信息，再继续追问。
```

---

## 4. A 端触发后要做什么

A 端被拉起后，默认按这个顺序跑：

```text
1. 识别当前输入属于什么问题
2. 判断缺什么：目标 / 边界 / 缺口 / 证据 / 方案
3. 自动调用最相关的 skill
4. 吸收新增信息
5. 重新判断是否继续问
6. 判断是否转 B/C
```

## 6. A 端输出格式

A 端每轮最好固定成这种格式：

```text
【当前理解】
我理解你现在的想法是：...

【已补齐】
- 目标：...
- 边界：...
- 缺口：...

【自动调用的 skill】
- ...

【新增信息】
- ...

【当前判断】
现在更像：澄清 / 研究 / 搜索 / 理论 / 执行 / 验证 / 决策

【下一步】
我建议继续问：...
```

---

## 7. 停机条件

A 端不是无限循环，以下任一条件满足就停：

```text
1. 目标、边界、方案、验证四项中至少 3 项达到可执行级别。
2. 连续 2 轮没有新增有效信息。
3. 最多循环 2~3 轮后，强制复盘是否该转端。
4. 如果继续追问只会改细节、不改路线，就停止循环。
5. 你明确说“满意”或“先停”。
```

停机后默认动作：

```text
- 转入 B/C 继续收束
- 或转入 spec / issue / 执行版方案
- 或回到 A 端做最小补齐后重新路由
```

---

## 8. 与“逻辑分端”的关系

这份卡不要求你真的拆成多个物理 bot。

它只要求：

```text
同一个 Hermes 实例内，按逻辑分端运行。
```

也就是：

- A 负责拷问和路由
- B 负责证据压缩
- C 负责理论/方案收束
- 但它们都可以在同一个入口里按阶段切换
