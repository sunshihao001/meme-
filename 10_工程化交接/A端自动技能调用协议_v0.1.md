# A端自动技能调用协议 v0.1

> 分线：10_工程化交接 / A端自动化
> 目的：让 A 端在自动提问、自动判断、自动推理的循环中，按需自动调用相关技能获取更多信息，直到初步想法收束到用户满意。

---

## 1. 总原则

A 端不是只做一次提问，也不是固定一个技能后就结束，而是：

```text
自动提问 → 自动判断 → 自动调用相关技能 → 自动推理 → 继续提问 → 直到满意
```

A 端的目标不是“问完所有问题”，而是：

```text
把初步想法推进到用户满意的地步。
```

---

## 2. 什么时候自动调用相关技能

A 端在以下情况应自动调用相关技能，而不是只靠自然语言追问：

### 2.1 需要把模糊想法问清楚

优先调用：

```text
brainstorming
+ dbs-good-question
+ grill-with-docs
+ grill-me
+ dbs-deconstruct
+ dbs-goal
+ dbs-decision
```

触发条件：

- 用户想法很散
- 目标和边界还没定
- 问题还不够像一个可研究任务

### 2.2 需要补更多背景或证据

优先调用：

```text
Agent Reach
+ dbs-learning
+ dbs-content-system
+ dbs-report
+ dbs-save
+ dbs-restore
```

触发条件：

- 需要更多资料
- 需要外部来源
- 需要把信息压成 Source Pack

### 2.3 需要把想法推进到可执行方案

优先调用：

```text
to-prd
+ writing-plans
+ to-issues
+ triage
+ improve-codebase-architecture
+ executing-plans
```

触发条件：

- 想法已经比较清楚
- 需要收束成方案
- 需要判断是否可以进入后续端口

---

## 3. A 端自动调用的判断顺序

每轮 A 端默认这样判断：

1. 当前输入是不是一个“还不够清楚的问题”
2. 如果不够，先自动追问 1-3 个关键问题
3. 如果已基本清楚，自动调用最相关的技能补信息
4. 根据补回来的信息自动判断是否进入 B/C
5. 如果还不成熟，继续下一轮自动提问

---

## 4. A 端自动调用时的输出结构

每轮建议固定输出：

```text
【当前理解】
我理解你现在的想法是：...

【自动调用的技能】
- ...

【新增信息】
- ...

【当前判断】
- 现在更像：澄清 / 研究 / 搜索 / 理论 / 执行 / 验证 / 决策

【下一轮】
- 我还要继续问：...
```

---

## 5. 停机条件

A 端不是无限循环，以下任一条件满足就停：

```text
1. 初步想法已经收束到你满意的地步。
2. 继续追问只会改细节，不会改路线。
3. 你明确说“满意”或“先停”。
```

---

## 6. 禁止事项

A 端不能：

- 不调用技能就直接下结论
- 把一个问题一次性问成一大堆
- 不判断信息增量就机械循环
- 还没收束就强行进入 B/C/D/E/F

---

## 7. 一句话版本

```text
A 端 = 自动提问 + 自动判断 + 自动调用技能 + 自动推理 + 持续循环，直到初步想法让用户满意。
```
