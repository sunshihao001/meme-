## 4. 循环代理分派

### A端：总控 / 判断门

任务：

```text
1. 确认第一批接入范围
2. 规定 A 端输出格式：Demand Grilling Brief
3. 判定何时调用 dbs-good-question / deconstruct / goal / diagnosis
4. 维护权限边界：dbskill 是辅助，不是主控
```

输出：

```text
A端接入控制简报
```

---

### B端：Source Pack / 原文证据包

任务：

```text
1. 从 dbskill 仓库提取第一批 skill 原文
2. 提取触发条件、步骤、输出结构、禁区
3. 形成 compact source pack
```

输出：

```text
dbskill第一批SourcePack_v0.1.md
```

---

### C端：内部包装方案 / 理论改造

任务：

```text
1. 设计 dbskill-A端判断增强包
2. 把四个外部 skill 合并成 A 端可调用模块
3. 输出调用顺序与合并模板
```

输出：

```text
dbskill-A端判断增强包设计_v0.1.md
```

---

### D端：Repo Landing / 安装落库

任务：

```text
1. 按 C 端方案生成内部包装文件
2. 更新索引与变更记录
3. 必要时准备 Hermes skill 草案，但不越权直接固化为主 skill
```

输出：

```text
落库文件 + diff + commit
```

---

### E端：效果验证 / 独立检查

任务：

```text
1. 用真实模糊需求跑样例
2. 检查输出是否确实提升判断质量
3. 检查是否破坏 A-F 边界
4. 检查是否只有漂亮话而无可执行路由
```

输出：

```text
dbskill第一批接入验证报告_v0.1.md
```

---

### F端：Owner 决策

任务：

```text
1. 看 E 端验证报告
2. 决定是否固化为常用能力
3. 决定是否进入第二批状态链接入
```

输出：

```text
ACCEPT / PARTIAL_ACCEPT / REVISE / REJECT
```

## 6. 验证样例

建议用这个真实样例验证：

```text
“我现在内部技能不够完善，想从 GitHub 外部技能仓库添加技能，
但不知道哪些该接入、怎么判断、怎么放进我的 A/B/C/D/E/F 多端口循环里。”
```

验证目标：

```text
1. 是否能识别这是外部技能接入问题
2. 是否能拆概念：内部技能、外部技能、接入、桥接、合并、拒绝
3. 是否能把目标变成可检查交付物
4. 是否能判断是否进入 B/C/D/E/F
5. 是否能给出停止条件和 Owner 决策点
```

---

## 7. 停止条件

本轮循环停止于以下任一条件：

```text
1. E 端验证通过，F 端接受固化
2. E 端发现边界破坏，退回 A/C 修订
3. Source Pack 证据不足，退回 B 补证据
4. 内部包装过度复杂，退回 A 端缩小范围
5. Owner 明确暂停接入
```

### 7.1 A/B/C 收束停机线

A 端在执行循环里仍要遵守统一的收束标准，避免前置澄清无限空转：

```text
当目标、边界、方案、验证四项中至少三项达到可执行级别，
且继续追问只会影响细节，不会改变路线时，停止 A/B/C 收束循环。
```

补充门槛：

```text
- 连续 2 轮没有新增有效信息时停止
- 最多循环 2~3 轮后强制复盘是否转端
- 停机后默认进入 spec / issue / 执行版方案，而不是继续空转
```
