# 今日理论认知收束：不要自建系统，而要桥接成熟工具 v0.1

> 分线：10_工程化交接 / 工作流认知收束
> 状态：v0.1 / 今日输入理论收束
> 目的：把今天关于 dbskill、A端入口、handoff、Trellis、Fractals、多技能组合与“不要造轮子”的关键认知写入仓库，作为后续工作流升级的判断基线。
> 核心结论：Owner 不是专业工程团队或 AI 专业人士，不应从零自建一套复杂 AI 编排系统；应以现有成熟工具和外部仓库为参照，按 A-F 工作流桥接、裁剪、验证后吸收。

---

## 1. 今日最高优先认知

今天最重要的修正是：

```text
不要把当前项目推进成“自己从零造一个 AI 工作流系统”。
```

更正确的方向是：

```text
以 Owner 的真实问题为中心，
用 A 端理解、收束、路由，
用 dbs / Trellis / Fractals 等外部成熟能力做桥接参考，
用 B/C/D/E/F 做资料、理论、落库、验证和决策，
避免为了工作流而工作流。
```

Owner 的定位不是：

```text
专业 AI 工程团队
专业系统架构团队
专业 agent orchestration 研究团队
从零写完整工作流平台的人
```

Owner 当前更需要的是：

```text
1. 能把自然语言想法变专业。
2. 能快速判断用什么现成工具/方法。
3. 能把外部工具桥接进自己的 A-F 工作流。
4. 能让 AI 帮忙执行，但保留 Owner 决策门。
5. 能减少空转，提高每天真实产出效率。
```

---

## 2. 今天的关键问题回顾

今天暴露了一个核心矛盾：

```text
工作流规则越写越多，但如果入口、路由、handoff 和外部工具桥接没有真正用起来，仍然会慢、散、像在自建系统。
```

此前的问题包括：

```text
1. A 端规则存在，但新输入没有稳定默认进入 A 端。
2. dbskill 已有大量技能，但没有先用 dbs 路由统一分发。
3. B/C/D/E/F 可以直接接用户输入，导致绕过 A 端。
4. 复杂想法被直接给下一步建议，而不是先给多条深化路线。
5. 外部编排工具选型容易变成“我要自己做一套”，而不是“先吸收成熟模式”。
```

---

## 3. dbskill 的今日定位

dbskill 不应被理解为“一堆技能全塞进 A 端”。

今天形成的新定位是：

```text
dbs = 默认前置路由层。
具体 dbs 技能 = A/B/C/D/E/F 按需调用的能力包。
```

### 3.1 dbs 路由的作用

```text
用户输入
→ dbs 路由识别问题类型
→ A 端做最小澄清和路由判断
→ B/C/D/E/F 按 handoff 执行
```

### 3.2 常用能力包

```text
A 端问题定义：dbs-good-question / dbs-deconstruct / dbs-goal / dbs-decision
B 端资料研究：dbs-learning / dbs-content-system / dbs-deconstruct / dbs-report
C 端理论深化：dbs-deconstruct / dbs-learning / dbs-decision
D/E/F 状态与决策：dbs-save / dbs-restore / dbs-report / dbs-decision
```

### 3.3 不能做的事

```text
1. 不全量调用所有 dbskill。
2. 不让技能列表替代工作流判断。
3. 不让 dbs-save / restore / report 替代 E 端验证或 F 端决策。
4. 不让 A 端被技能名淹没。
```

## 4. A 端今日升级

今天 A 端被强化为两个角色：

```text
1. 默认入口门。
2. 多路线深化引导门。
```

### 4.1 默认入口门

新输入默认路径：

```text
用户新输入
→ dbs 路由
→ A 端最小澄清
→ A 端 handoff
→ B/C/D/E/F
```

跳过 A 的情况必须明确，例如：

```text
继续上次
直接执行
恢复
验证
上传 / 提交 / 推送
```

即使跳过，也要说明理由。

### 4.2 多路线深化引导门

当 Owner 提出复杂想法时，A 端不能只给：

```text
认可 / 不认可
```

而应给出多条可深化路线，例如：

```text
A. 对比验证路线：dbs-benchmark
B. 输出质量诊断路线：dbs-content
C. 好问题重构路线：dbs-good-question
D. 组合诊断链路线：benchmark → content → good-question
E. A端治理路线：dbs-deconstruct → dbs-decision
```

核心原则：

```text
复杂想法先给路线，再让 Owner 选方向。
```

## 5. Handoff 纪律的今日定位

今天明确了：

```text
没有 handoff，就没有下游执行。
```

B/C/D/E 不应直接接收用户原始输入。

```text
B 只接 A 的搜索任务单 / 知识缺口。
C 只接 A 问题框架 + B Source Pack。
D 只接 C-execution + 授权 + allowed paths。
E 只接真实文件 / diff / 验证命令 / 验收标准。
```

这条规则的目的不是增加复杂度，而是防止：

```text
1. 端口乱跳。
2. 绕过 A 端。
3. 理论未经资料支撑就生成。
4. 未经授权就落库。
5. 没有验证就宣布完成。
```

## 6. 多技能组合链的今日定位

Owner 提出了一个重要思路：

```text
不要机械地一个问题只用一个技能。
复杂问题应该允许多个 dbs 技能组合使用。
```

今天形成的组合链示例是：

```text
dbs-benchmark → dbs-content → dbs-good-question
```

用途：

```text
1. benchmark：对比理论想法与实际输出。
2. content：诊断输出是否集中、清晰、可执行。
3. good-question：把发现的问题重写成 Agent 可推理、可验证的问题说明书。
```

但多技能组合必须受 A 端控制：

```text
先路由，后组合；
先控制范围，再灵活调用；
技能组合服务问题，不让问题服务技能。
```

## 7. Trellis 与 Fractals 的今日选型认知

今天对两个外部 AI 工作流编排仓库形成了初步判断。

### 7.1 Trellis

Trellis 更像：

```text
repo-backed AI coding engineering framework
```

它关注：

```text
spec
task
workspace memory
Plan / Implement / Verify / Finish
repo 内持久化上下文
多 agent 平台统一
```

它更适合当前主线，因为当前痛点是：

```text
工作流怎么真正沉淀、记忆、持续运行。
```

推荐定位：

```text
主线参考 / BRIDGE / PATTERN_ONLY
```

不建议：

```text
整套覆盖当前项目。
```

---

### 7.2 Fractals

Fractals 更像：

```text
recursive task tree + multi-agent worktree execution orchestrator
```

它关注：

```text
recursive decomposition
atomic/composite task classification
leaf task execution
git worktree isolation
agent swarm execution
```

它更适合作为：

```text
未来 C-execution / D-worker 执行层参考
```

风险：

```text
experimental
提交少
并行执行复杂
权限模式风险高
容易绕过 A/F gate
```

推荐定位：

```text
执行层参考 / PATTERN_ONLY
```

---

### 7.3 当前默认选择

```text
Trellis 做主线骨架参考。
Fractals 做执行层模式参考。
```

这不是二选一安装，而是：

```text
用 Trellis 吸收 repo-backed spec/task/memory。
用 Fractals 吸收 task tree / worktree leaf execution 思想。
```

## 8. 不造轮子的工作方式

后续遇到类似问题时，默认顺序应是：

```text
1. A 端先理解 Owner 的真实想法。
2. dbs 路由判断应该用哪些现成能力包。
3. B 端查外部工具/仓库是否已有成熟模式。
4. C 端只做桥接理论，不从零发明系统。
5. D 端只落库经过 A/B/C/E/F 认可的最小规则。
6. E 端验证是否真的减少复杂度、提升效率。
7. F 端决定是否吸收为基线。
```

严禁默认进入：

```text
我们自己做一套完整系统。
```

优先进入：

```text
成熟工具怎么桥接到当前工作流。
```

## 9. 今天形成的工作流认知基线

```text
1. dbs 是路由入口，不是技能堆。
2. A 是默认入口，不是普通总结器。
3. A 端面对复杂想法要给多路线深化，而不是只问认可。
4. B/C/D/E/F 必须只接 handoff。
5. 多技能组合可以用，但必须由 A 端控制范围。
6. Trellis 优先作为主线工程化参考。
7. Fractals 暂作未来执行层参考。
8. Owner 不是专业团队，不要默认自建完整 AI 编排系统。
9. 所有升级都应减少 Owner 负担，而不是增加概念负担。
10. 每次工作流改造都要问：它是否让明天更快、更稳、更少空转？
```

## 11. 本版结论

```text
当前工作流下一阶段的重点，不是继续自建系统，
而是让 A 端成为真正入口，
让 dbs 成为真正路由，
让外部成熟工具成为桥接参考，
让 B/C/D/E/F 通过 handoff 专职运转。
```

一句话：

```text
少造轮子，多桥接；少堆规则，多验证；少替 Owner 决定，多给可选深化路线。
```
