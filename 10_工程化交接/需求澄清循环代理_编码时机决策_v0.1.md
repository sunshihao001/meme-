## 1. Owner 当前问题

Owner 指出：

```text
实际代码的生成不是说好给 Codex 来做吗？
现在具体全部的项目实际理论体系设计还没有完善到可以做代码执行吧？
现在需要先思考的是：先把全部体系都设计好做代码执行，还是边做边改？
```

这不是一个普通实现问题，而是：

```text
需求澄清端对“何时允许 Codex 进入执行”的阶段门判断。
```

---

## 2. 语义发散：三个问题框架

### 框架 A：完整体系先行

问题写法：

```text
是否必须先把最终 GMGN 自建交易系统的全部理论、字段、策略、风控、执行路径设计完，再允许 Codex 写任何代码？
```

优点：

```text
减少返工。
防止半成品脚本误导成交易系统。
理论一致性更强。
```

缺点：

```text
容易陷入永远设计不完。
真实数据接口、样本质量、字段可得性不经小实验很难确认。
需求澄清端可能变成纯文档工程。
```

适用场景：

```text
实盘交易、资金执行、下单风控、自动买卖、对外产品承诺。
```

---

### 框架 B：边做边改

问题写法：

```text
是否可以一边设计理论，一边让 Codex 逐步写脚本，通过代码运行反推理论缺口？
```

优点：

```text
可以快速暴露 GMGN 数据缺口。
可以用真实接口/样本验证字段可得性。
能避免理论空转。
```

缺点：

```text
如果没有强边界，Codex 可能提前写出伪交易系统。
如果 spec 不清楚，代码会固化错误抽象。
Owner 会不断纠偏方向。
```

适用场景：

```text
只读数据抓取、字段探针、validator、fixture、样本生成器、schema 检查。
```

---

### 框架 C：分层闸门 / 研究型可执行切片

问题写法：

```text
不是“全部设计完才写代码”，也不是“直接边写交易系统边改”，而是建立阶段闸门：理论核心先收敛到能定义一个只读可验证切片，再交 Codex 写探针/validator；每个切片反哺理论，但禁止越级到交易执行。
```

优点：

```text
保留理论控制。
允许真实数据反馈。
防止代码越界成实盘系统。
符合 Codex 作为执行者、需求澄清端作为控制者的分工。
```

缺点：

```text
需要维护明确的 stage gate。
每轮都要记录：理论假设、代码边界、验证结果、下一轮理论修正。
```

适用场景：

```text
当前项目阶段。
```

## 3. 当前阶段判断

当前不能进入：

```text
完整自建交易系统代码执行
实盘交易脚本
swap / 下单 / 自动执行
资金风控模块
收益优化模块
```

原因：

```text
1. 根系需求刚刚修正：MQL5 降级，GMGN + 自建脚本成为主线。
2. GMGN → FCZ 状态机字段映射只有 v0.1，还没有真实样本/fixture 验证。
3. POC/VAH/VAL 在 GMGN kline 上的可用性仍是 implementable_with_approximation，不是 validated。
4. USD volume 和 token amount 哪个作为 AVWAP 权重仍未定。
5. 风控、执行、滑点、手续费、交易路由仍未形成需求澄清文件。
```

当前可以进入的只有：

```text
只读研究型可执行切片。
```

也就是：

```text
Codex 可以写“探针 / mapper / validator / fixture 测试”，不能写“交易执行系统”。
```

---

## 4. 推荐决策

推荐采用：

```text
框架 C：分层闸门 / 研究型可执行切片。
```

一句话：

```text
不是等全部体系设计完才写代码，也不是边写交易系统边改；而是先把每一层理论收敛到一个只读、可验证、不可交易的最小切片，再交给 Codex 执行，执行结果回流理论。
```

## 6. Codex 分工边界

Codex 是：

```text
maker / executor
```

需求澄清端是：

```text
control-plane / checker / stage gate owner
```

当前 Codex 不能直接接收：

```text
“做一个 GMGN 自建交易系统”
```

当前 Codex 最多可以接收：

```text
“实现一个只读 GMGN fixture validator，验证字段可得性和字段命名，不调用真实交易、不输出买卖建议。”
```

或：

```text
“实现一个 GMGN kline fixture → FCZ 字段草稿 mapper，只处理本地 fixture，不连接钱包、不交易。”
```

## 7. 当前 Owner Decision

决策题：

```text
当前项目是否采用“分层闸门 / 研究型可执行切片”作为后续推进方式？
```

推荐默认：

```text
是。
```

如果选择“先全部体系设计完”：

```text
短期不会写任何代码；继续补齐完整交易系统理论、风控、执行、样本、成本模型。
风险：理论长期空转，真实 GMGN 数据约束迟迟不暴露。
```

如果选择“边做边改”：

```text
可以快速暴露数据问题。
风险：如果没有闸门，容易提前写成错误交易系统。
```

推荐折中：

```text
允许 Codex 进入 Gate 2：只读数据探针 / fixture validator。
禁止进入交易执行代码。
```

## 9. 本版结论

```text
当前理论体系还没有成熟到可以让 Codex 写“实际交易系统”。

但也不需要等“全部体系”一次性设计完。

正确路径是：需求澄清端先建立分层闸门；Codex 只执行最小、只读、可验证、不可交易的研究型切片；每个切片输出真实验证结果，再回流理论研究。
```
