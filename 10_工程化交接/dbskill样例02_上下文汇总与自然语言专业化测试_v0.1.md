## 2. 已形成的关键共识

### 2.1 不是全量接入外部 skill

共识：

```text
不是所有 skill 都需要接入。
```

判断原则：

```text
1. 有些外部仓库是完整工程师 skill 包，只抽其中 1-2 个部件即可。
2. 全量接入会污染当前工作流。
3. A端要判断是否适合当前用户、当前项目、当前工作流。
4. 输出必须自然语言可理解，但底层判断保持专业化。
```

---

### 2.2 A端不是普通需求澄清，而是自然语言升级门

A端新定位：

```text
自然语言 → 深度理解 → 需求澄清 → 专业化重写 → 可执行方案 → 循环学习升级
```

含义：

```text
用户不需要先成为专业工程师或 AI 从业者；
用户可以先讲自然语言；
A端负责挖掘真实意图、补全专业视角、打开更好的可能性；
最后仍然用用户能看懂的话反馈。
```

---

### 2.3 dbskill 第一批的新定位

dbskill 第一批不再只是“判断增强包”，而是：

```text
A端自然语言专业化升级辅助包
```

四件套分工：

```text
dbs-good-question → 问题成型器
dbs-deconstruct → 概念拆解器
dbs-goal → 目标升级器
dbs-diagnosis → 问题反审器
```

---

### 2.4 为什么之前感觉“没跑起来”

原因：

```text
项目级 skill 已更新 ≠ 当前聊天全局 runtime 自动触发。
```

解决方案：

```text
在 A端主 skill 中补入默认自动触发规则。
```

默认触发对象：

```text
想法 / 判断 / 方向 / 方案 / 工作流 / skill 接入 / 是否贴合 / 是否调整 / 更专业方案
```

---

### 2.5 当前推荐架构

三层运行：

```text
L0 日常轻量自动触发
L1 A端正式需求澄清 / 自然语言升级
L2 A-F 多端口工程化循环
```

其中 L0 默认输出：

```text
1. 我理解你真正想说的是
2. 当前想法的限制
3. 专业化升级版本
4. 建议工作流
5. 下一步最小动作
```
