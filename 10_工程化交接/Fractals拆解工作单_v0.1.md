## 4. Fractals 的执行任务

### 4.1 先做 A 端问题再确认

Fractals 先检查 A 端底稿是否足够专业化：

```text
- 目标是否清楚
- 问题世界是否清楚
- 边界 / 非目标是否清楚
- 缺口是否清楚
- 是否已经能拆树
```

### 4.2 再做递归拆解

将当前阶段拆成至少以下五类端口骨架：

```text
A = 治理入口 / 需求控制 / 概念边界
B = 证据压缩 / Source Pack / 样本补强
C = 理论 / 状态机 / mapper / 风险结构
Bridge = 规则映射 / 任务路由 / handoff
E = 验证 / 质量门 / 索引 / 变更记录 / 假完成检测
```

### 4.3 再做优先级排序

输出一个“当前量化体系系统结构完善优先级”清单，排序依据按以下原则：

```text
1. 依赖先行：后层不能跳过前层。
2. 风险优先：最容易漂移的入口 / validator / 样本优先。
3. 事实优先：样本与 provenance 优先于理论扩张。
4. 结构优先：FCZ 状态机 / mapper 优先于回测和执行。
5. 安全优先：observe-only / research-only 固化在前，执行层放后。
```

## 5. 允许的输出格式

Fractals 最终应输出以下内容：

```text
1. A/B/C/Bridge/E 树状拆解结果
2. 每个节点的简短职责说明
3. 每个节点的优先级（P0 / P1 / P2 / P3）
4. 每个节点的前置依赖
5. 每个节点的停机条件
6. 当前阶段最值得优先补的 5 个子任务
7. 当前阶段不该做的 5 个子任务
```

### 推荐附加输出

```text
- 一个给 Owner 看的一页总结
- 一个给后续 Codex 用的 bounded work order 草案
- 一个“为什么不应该直接进入交易执行”的边界说明
```

## 7. 验证标准

Fractals 的输出只有在满足以下条件时才算可用：

```text
1. 树状拆解能清楚分出 A/B/C/Bridge/E。
2. 每个节点都能对应到当前阶段的真实工作。
3. 优先级排序能解释“先做什么、后做什么”。
4. 能明确指出当前阶段不该做什么。
5. 输出能直接交给后续 B/C/D/E/F 或 Codex 继续执行。
```

---

## 8. 停机门

满足以下任一条件即可停机并交回：

```text
1. 目标 / 边界 / 方案 / 验证 4 项中至少 3 项达到可执行级别。
2. 连续 2 轮没有新增有效信息。
3. 继续追问只会改细节，不会改路线。
4. 已经能生成 Fractals 可消费的 bounded contract。
```
