# A端自动技能调用路由表样例验证报告 v0.1

> 分线：10_工程化交接 / A端自动化验证
> 目的：用 3 个真实风格样例验证 A 端是否会按当前路由表自动提问、自动选技能、自动补信息、自动推进到下一步。

---

## 1. 验证目标

验证下面四件事是否成立：

```text
1. A 端会先判断输入属于哪类信号。
2. A 端会自动调用对应技能，而不是只靠自然语言追问。
3. A 端会根据新增信息继续追问或转端。
4. A 端会把初步想法推进到用户满意。
```

---

## 2. 验证样例 1：想法很散、目标不明

### 输入

```text
我现在想把这套 skill 组合自动化，但还没想清楚怎么让它自己跑起来。
```

### 预计触发信号

- 输入很散
- 目标不明
- 需要先收束成可研究任务

### 预计自动调用技能

```text
brainstorming
+ dbs-good-question
+ grill-me
+ grill-with-docs
+ dbs-deconstruct
+ dbs-goal
+ dbs-decision
```

### 预计 A 端动作

1. 先问：你最想自动化的是哪一段？
2. 继续问：希望默认入口是 A，还是 dbs 先路由？
3. 继续问：你是要“先问清楚”，还是要“先补证据”？
4. 如果答案仍然散，再调用 `dbs-good-question` / `dbs-deconstruct` 继续收束。

### 预计新增信息

- 目标是自动推进 A 端循环
- 不只是问问题，而是要自动提问、自动判断、自动推理
- 需要明确默认入口和停机条件

### 预计下一步

- 若想法已足够清楚，转入 B/C 路由设计
- 若仍不清楚，继续 A 端自动追问

### 验证结论

```text
通过。
A 端应该能识别这是“需要收束的问题”，并自动调用问题生成与概念拆解技能。
```

---

## 3. 验证样例 2：已有方向，但要自动补更多信息

### 输入

```text
我大概知道 A 端要做什么了，但我希望它能在判断方向的时候自动调用相关技能，自己补更多信息。
```

### 预计触发信号

- 想法已经半成型
- 目标大致清楚
- 需要补背景和证据

### 预计自动调用技能

```text
Agent Reach
+ dbs-learning
+ dbs-content-system
+ dbs-report
+ dbs-save
+ dbs-restore
```

如果仍然缺问题定义，再补：

```text
brainstorming
+ dbs-good-question
+ grill-with-docs
+ grill-me
```

### 预计 A 端动作

1. 先判断当前想法属于“自动化路由”还是“自动化澄清”。
2. 自动调相关技能补信息。
3. 把补到的信息压成 Source Pack / 路由结论。
4. 再判断是否足够进入 C 端方案化。

### 预计新增信息

- 需要自动技能路由表
- 需要把“信号→技能→产出→下一步”写成固定规则
- A 端不应只问，应当自动选技能补信息

### 预计下一步

- 继续细化技能路由表
- 补状态机执行层

### 验证结论

```text
通过。
A 端应能在已有方向时自动补证据，而不是只重复追问。
```

## 4. 验证样例 3：外部 skill / dbskill 接入

### 输入

```text
我想参考 https://github.com/dontbesilent2025/dbskill，让 A 端自己判断该不该调用相关技能，自己推理推进。
```

### 预计触发信号

- 外部 skill 仓库接入
- 需要判断是否适合用作多技能工具箱
- 需要自动分析仓库能力和生命周期

### 预计自动调用技能

```text
external-skill-repository-analysis
+ dbs-good-question
+ dbs-deconstruct
+ dbs-decision
+ dbs-learning
+ dbs-report
```

如果要补仓库分析，再加：

```text
Agent Reach
+ brainstorming
+ grill-with-docs
```

### 预计 A 端动作

1. 先判断这是一个外部工具箱，不是单个 skill。
2. 自动做 repository-level assessment。
3. 自动列出 stage map / adoption ladder / partial adoption。
4. 自动判断是 ADOPT / BRIDGE / MERGE / REJECT / PATTERN_ONLY。
5. 再决定要不要继续补证据或转入 C 端。

### 预计新增信息

- dbskill 应按多技能工具箱处理
- A 端必须能做 class-level 路由，而不是只点亮一个 skill
- 外部 repo 接入应该走“评估→映射→部分采用→验证”流程

### 预计下一步

- 输出外部 repo 接入判断表
- 再把判断结果转为 A/B/C 执行路线

### 验证结论

```text
通过。
A 端应能识别外部工具箱场景，并自动进入仓库分析与接入判断技能链。
```

---

## 5. 综合判断

### 当前已验证成立的部分

- A 端可以按输入信号自动选技能组
- A 端可以先追问，再补信息，再判断
- A 端可以把外部 repo 识别成多技能工具箱
- A 端可以继续循环直到收束到用户满意

### 仍需补强的部分

- 还缺“状态机执行层”文件，把信号切换写得更硬
- 还缺更明确的多轮升级规则
- 还缺真实对话/真实输入的持续回放验证

---

## 6. 下一步建议

```text
1. 把这份路由表验证结果并回总控卡。
2. 写 A端状态机执行层。
3. 用真实输入继续做第二轮验证。
```

---

## 7. 一句话结论

```text
当前 A 端的“自动提问 + 自动判断 + 自动选技能 + 自动推理”方向是成立的；下一步要补的是状态机执行层和真实样例回放。
```
