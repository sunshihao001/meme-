## 5. 桥接决策规则

### 5.1 什么时候只做 pattern_only

满足任一条件时，不创建本地 skill，只作为 pattern：

```text
1. 外部方法只是提供一个概念框架，不需要在每轮运行中完整执行。
2. 当前项目只需要吸收一个原则，例如“实现前必须先对齐设计”。
3. 完整外部 skill 会引入过多交互门或流程负担。
4. 当前任务是文档/契约修正，不需要完整设计会话。
```

### 5.2 什么时候创建 bridge skill

满足以下条件时，才考虑创建 Hermes bridge skill：

```text
1. Owner 明确要求该能力成为需求拷问端固定节点。
2. 当前 profile 没有等价能力，且 DBS 现有 skill 无法覆盖。
3. 多轮运行反复需要同一套外部 skill 协议。
4. 该能力可以写成轻量本地 SKILL.md，不需要复制整个外部项目。
5. 能明确触发条件、输入、输出、停止条件和验证字段。
```

### 5.3 什么时候安装外部 skill

只有满足以下条件时，才考虑安装外部 skill：

```text
1. Hermes / 当前运行环境支持该外部 skill 格式。
2. license / 来源 / 安装方式明确。
3. Owner 授权安装或引入第三方 skill。
4. 不会污染当前业务项目为方法轮仓库。
5. 能在当前 profile 中通过 skills_list / skill_view 验证。
```

## 7. 建议的桥接路线

### 路线 A：不创建新 skill，只保留 pattern_only

适合当前阶段。

优点：

```text
1. 轻量。
2. 不污染项目。
3. 不引入第三方依赖。
4. 不把需求拷问端变成方法论堆叠。
```

缺点：

```text
1. 不能报告为 loaded/executed。
2. 每轮需要手动说明 pattern_only。
```

### 路线 B：创建轻量 Hermes bridge skill

候选名称：

```text
demand-brainstorm-grill
```

角色：

```text
把 Superpowers brainstorming 的 design-first gate、Matt Pocock /grill-me 的 alignment grilling、项目的搜索前语义转译与旧知识召回整合成一个轻量桥接 skill。
```

不复制外部 skill 全文，只吸收可验证协议：

```text
1. 先读项目真源。
2. 一次只问一个关键澄清问题。
3. 给 2-3 个方案和 trade-off。
4. 进入实现前必须形成设计/收敛切片。
5. 记录 approval / not_approved / needs_owner。
6. 输出 skill_runtime_matrix。
```

### 路线 C：安装外部 skill

当前不建议直接做。

原因：

```text
1. 当前 Hermes profile 未确认可直接安装 obra/superpowers skill。
2. 业务仓库不应承担外部技能库安装实验。
3. Owner 当前要求是“按照建议做审计”，不是授权安装第三方 skill。
```
