## 1. 本轮正确表达

Owner 原始反馈可以转写为：


更专业的表达：

```text
当前需求拷问端缺少 Skill Orchestration Layer。
问题不是“少调用某个 skill”，而是缺少从能力注册、调用计划、运行证据、输出交接到 validator 的完整编排协议。
```

本轮收敛切片：

```text
先补“技能编排层 v0.1”的最小协议，不强行全量调用所有 skill，也不把方法论层继续膨胀成大系统。
```

## 2. 需求拷问端作为产品的定义

推荐产品名：

```text
Demand Cognition & Knowledge Integration Orchestrator
需求认知与知识接入编排器
```

一句话定义：

```text
它是在进入 spec / issue / Codex / 实现之前，负责理解用户真实意图、调度合适思考技能、召回内部旧知识、搜索外部来源、形成可验证收敛切片的 AI 工作流控制产品。
```

它不是：

```text
1. 单纯的问题改写器。
2. 单纯的最高价值问题排序器。
3. 单纯的 skill 清单。
4. 单纯的文档生成器。
```

它应该是：

```text
1. 需求认知入口。
2. skill 编排器。
3. 内部旧知识召回器。
4. 外部搜索补盲器。
5. 证据收敛与执行切片生成器。
6. owner decision / validator / issue / handoff 的前置控制层。
```

## 3. 当前缺陷诊断

### 3.1 listed skills ≠ orchestrated skills

当前项目已经能列出：

```text
dbs-good-question
dbs-goal
dbs-content-system
dbs-chatroom
dbs-slowisfast
dbs-decision
dbs-save
dbs-report
spec-first-ai-engineering
```

但“列出”不等于“编排”。

必须区分：

```text
listed_only：文档中列出，但没有运行证据。
installed：当前 profile 存在该 skill。
loaded：本轮通过 skill_view 实际加载。
executed：按该 skill 的协议真正执行到产出。
output_consumed：该 skill 的输出被下游步骤使用。
referenced_only：只作为方法或说明被引用。
conditional_not_executed：因 skill 自身规则或上下文不满足而未执行。
missing：预期存在但当前 profile 未发现。
forbidden：本轮明确禁止调用。
```

### 3.2 DBS 不是一个诊断技能，而是 router

`dbs` 的角色是：

```text
router：根据 Owner 问题路由到具体 dbs skill。
```

因此需求拷问端不能模糊写：

```text
调用 DBS。
```

必须写成：

```text
dbs.router → selected_dbs_specialists
```

例如：

```text
问题模糊 → dbs-good-question
目标空转 → dbs-goal
概念混乱 → dbs-deconstruct
需要多角色批评 → dbs-chatroom
过早收敛 / 伪效率 → dbs-slowisfast
涉及长期 owner 决策 → dbs-decision
知识结构化 / 旧知识接入 → dbs-content-system
保存诊断状态 → dbs-save
打包报告 → dbs-report
```

### 3.3 dbs-chatroom 有确认门，不能伪执行

`dbs-chatroom` 的协议要求：

```text
未指定专家时，先推荐 3-5 位专家，并等待 Owner 确认；确认后才启动多 Agent 聊天室。
```

因此其状态不能随便写成 executed。

更准确的状态包括：

```text
loaded：已读取 skill。
recommendation_needed：需要推荐专家。
waiting_owner_confirmation：已推荐但未确认。
conditional_not_executed：因未确认或本轮不需要多角色而未执行。
executed：Owner 确认后真的启动多专家对话。
```

### 3.4 sup / Superpowers brainstorming 当前必须标记为 missing 或 pattern_only

当前 Hermes profile 技能池中没有确认存在：

```text
superpowers
sup
brainstorm
```

因此不能报告为 loaded / executed。

它目前最多是：

```text
referenced_as_pattern_only：作为 intent-brainstorm-grill 的方法来源被吸收。
missing_runtime_skill：若 Owner 期望独立 skill，需要补 skill bridge 或安装/创建对应技能。
```

## 5. Skill Invocation Plan v0.1

每轮需求拷问必须先生成：

```text
skill_invocation_plan:
  - skill: <name>
    role: <router / clarifier / critic / gate / handoff / domain data>
    trigger_reason: <why this skill is needed>
    expected_output: <what artifact/state it should produce>
    downstream_consumer: <which next step consumes it>
    execution_mode: required / conditional / gated / forbidden / pattern_only / missing
```

规则：

```text
1. required 必须 loaded，并尽量 executed。
2. conditional 必须说明触发条件是否满足。
3. gated 必须说明需要 Owner 确认还是权限条件。
4. pattern_only 不得被写成 loaded/executed。
5. missing 必须进入 missing_skill_bridge，不得假装调用。
6. forbidden 必须列入 forbidden_capabilities。
```

## 6. Skill Handoff Chain v0.1

需求拷问端的标准交接链：

```text
owner_request
→ dbs.router / route_reason
→ dbs-good-question.problem_statement
→ dbs-deconstruct.concept_atoms（如概念不清）
→ intent-brainstorm-grill.options（pattern 或真实 skill）
→ dbs-chatroom.expert_critique（如 Owner 确认）
→ dbs-slowisfast.slow_audit（如存在过早收敛风险）
→ search_intent_translation.query_variants
→ internal_knowledge_recall.legacy_sources
→ external_absorption.findings
→ spec-first-ai-engineering.acceptance_criteria / handoff
→ validator / issue / owner_decision / project_file_patch
```

本链条不是每轮全量执行，而是每轮必须说明：

```text
哪些节点执行了；
哪些节点跳过了；
跳过原因是什么；
哪个输出被哪个下游消费。
```

## 7. Skill Runtime Matrix v0.1

每轮报告必须包含：

```text
skill_runtime_matrix:
  - skill: <name>
    installed: yes/no/unknown
    loaded: yes/no
    executed: yes/no/partial
    output_consumed: yes/no/not_applicable
    referenced_only: yes/no
    conditional_not_executed: reason_or_empty
    missing: reason_or_empty
    forbidden: reason_or_empty
```

禁止模糊表述：

```text
“按 DBS 执行了”
“调用了方法轮”
“使用了 sup 头脑风暴”
“dbs-chatroom 已使用”
```

除非能给出：

```text
skill_view 证据 / 子 Agent 执行证据 / 输出片段 / 下游消费位置 / 文件路径。
```

## 8. 初步修正边界

本版只做最小修正：

```text
1. 固化产品定义。
2. 固化能力注册表。
3. 固化 skill_invocation_plan。
4. 固化 skill_handoff_chain。
5. 固化 skill_runtime_matrix。
6. 更新 loop report validator，让未来报告不能再只写 loaded / referenced_only，而必须写 invocation plan 与 runtime matrix。
```

暂不做：

```text
1. 不强行安装或创建 sup / Superpowers skill。
2. 不自动启动 dbs-chatroom 多专家会议。
3. 不全量调用所有 DBS skills。
4. 不引入 RAG / 向量库。
5. 不把业务项目改造成方法轮产品仓库。
```
